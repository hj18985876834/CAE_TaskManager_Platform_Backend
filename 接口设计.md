下面给出一份**详细接口设计**。目标是：

- 满足前面的**需求分析 / 系统设计 / 数据库设计**
- 符合**微服务架构、分布式、容器化**思想
- **不设计得过重**，保证能实际开发
- 重点覆盖：**用户、求解器、任务、调度、节点代理、日志、结果、状态流转**

# 船舶CAE多求解器仿真任务管理与调度平台

## 接口设计

------

# 1. 接口设计目标

本系统接口设计需要满足以下目标：

1. 支持前后端分离开发；
2. 支持微服务拆分后的服务间调用；
3. 支持分布式节点代理接入与状态回传；
4. 支持多求解器、多任务类型、多模板的统一管理；
5. 支持任务状态流转、日志查看、结果查询等核心业务；
6. 接口设计尽量简洁，避免过度复杂，降低实现难度；
7. 统一通过网关暴露接口，便于鉴权、路由和部署。

------

# 2. 微服务与接口边界

系统采用如下服务划分：

1. `gateway-service`
2. `user-service`
3. `solver-service`
4. `task-service`
5. `scheduler-service`
6. `node-agent`

接口层面可分为四大类：

1. **前端业务接口**
   供 Web 前端调用，经网关统一转发。
2. **服务内部管理接口**
   供微服务之间调用，通常不直接暴露给前端。
3. **节点代理接口**
   中心服务与 node-agent 之间交互。
4. **文件与日志接口**
   用于上传输入文件、读取日志、下载结果文件。

------

# 3. 接口设计总体规范

------

## 3.1 URL 设计规范

统一采用 REST 风格，建议网关统一前缀：

```text
/api
```

再按服务划分路径，例如：

```text
/api/auth/...
/api/users/...
/api/solvers/...
/api/tasks/...
/api/scheduler/...
/api/nodes/...
```

------

## 3.2 请求方式规范

- `GET`：查询
- `POST`：新增 / 执行动作
- `PUT`：更新
- `DELETE`：删除或逻辑删除

对于“动作型接口”可使用：

- `POST /tasks/{id}/cancel`
- `POST /nodes/register`
- `POST /nodes/heartbeat`

这样更贴近业务语义。

------

## 3.3 统一返回结构

建议全系统统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明：

- `code`：0 表示成功，非 0 表示失败
- `message`：提示信息
- `data`：业务数据

分页接口建议：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "total": 100,
    "pageNum": 1,
    "pageSize": 10,
    "records": []
  }
}
```

------

## 3.4 鉴权规范

### 前端用户接口

采用 JWT：

```http
Authorization: Bearer {token}
```

### 节点代理接口

节点代理接口统一采用 nodeToken 鉴权，请求头为 X-Node-Token。

```http
X-Node-Token: xxxxxx
```

管理员接口和普通用户接口在网关层统一校验角色。

------

## 3.5 时间与状态字段规范

统一采用：

- 时间格式：`yyyy-MM-dd HH:mm:ss`
- 状态使用字符串枚举，不建议前期使用数字码

例如：

- `CREATED`
- `QUEUED`
- `RUNNING`
- `SUCCESS`
- `FAILED`

------

# 4. 网关设计

网关主要负责：

1. 用户登录后 token 校验；
2. 管理员权限校验；
3. 路由转发到各微服务；
4. 统一异常处理；
5. 统一跨域处理。

网关一般不承担复杂业务逻辑，因此无需设计过多业务接口，只保留认证入口与路由规则。

# 7. task-service 接口设计

这是最核心的一组接口。

------

## 7.1 任务创建接口

### 7.1.1 创建任务

**POST** `/api/tasks`

#### 请求参数

```json
{
  "taskName": "船体稳态流体分析任务1",
  "solverId": 1,
  "profileId": 11,
  "taskType": "CFD_STEADY",
  "params": {
    "caseName": "ship_case_01",
    "description": "基础工况"
  }
}
```

#### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,
    "taskNo": "TASK202603250001",
    "status": "CREATED"
  }
}
```

### 说明

此接口仅创建任务主记录，不一定完成全部文件上传。

------

### 7.1.2 上传任务输入文件

**POST** `/api/tasks/{taskId}/files`

#### 请求方式

```
multipart/form-data
```

#### 参数

- `file`
- `fileKey`
- `fileRole`

### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "fileId": 3001,
    "originName": "case.zip",
    "storagePath": "/data/tasks/1001/input/case.zip"
  }
}
```

### 说明

支持多次上传，适合不同模板。

------

### 7.1.3 校验任务

**POST** `/api/tasks/{taskId}/validate`

### 作用

触发：

- 表单校验
- 文件规则校验
- 模板校验

#### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,
    "valid": true,
    "status": "VALIDATED"
  }
}
```

#### 失败返回

```json
{
  "code": 4001,
  "message": "缺少必需文件 model.inp",
  "data": {
    "taskId": 1002,
    "valid": false
  }
}
```

------

### 7.1.4 提交任务入队

**POST** `/api/tasks/{taskId}/submit`

#### 返回

```json
{
  "code": 0,
  "message": "任务已进入队列",
  "data": {
    "taskId": 1001,
    "status": "QUEUED"
  }
}
```

### 说明

为什么拆成“创建 / 上传 / 校验 / 提交”四步：

- 更贴近实际流程
- 降低单接口复杂度
- 更适合大文件上传场景
- 前端交互更清晰

# 10. 典型业务流程对应接口设计

------

## 10.1 用户创建并提交任务流程

1. 登录
   `POST /api/auth/login`
2. 获取求解器列表
   `GET /api/solvers`
3. 根据求解器获取任务模板
   `GET /api/solvers/{solverId}/task-options`
4. 获取上传规范
   `GET /api/profiles/{profileId}/upload-spec`
5. 创建任务
   `POST /api/tasks`
6. 上传输入文件
   `POST /api/tasks/{taskId}/files`
7. 校验任务
   `POST /api/tasks/{taskId}/validate`
8. 提交任务入队
   `POST /api/tasks/{taskId}/submit`

------

## 10.2 调度执行流程

1. scheduler-service 查询待调度任务
   `GET /internal/tasks/queued`
2. 查询满足条件的节点
   `GET /internal/nodes/available?solverId=1`
3. task-service 标记任务为 `SCHEDULED`
   `POST /internal/tasks/{taskId}/mark-scheduled`
4. scheduler-service 调用 node-agent 下发任务
   `POST /internal/dispatch-task`
5. task-service 标记任务为 `DISPATCHED`
   `POST /internal/tasks/{taskId}/mark-dispatched`
6. 记录调度日志
   `POST /internal/schedules`

------

## 10.3 节点执行回传流程

1. node-agent 标记开始运行
   `POST /internal/tasks/{taskId}/status-report`
2. 周期上报日志
   `POST /internal/tasks/{taskId}/log-report`
3. 上报结果摘要
   `POST /internal/tasks/{taskId}/result-summary-report`
4. 上报结果文件
   `POST /internal/tasks/{taskId}/result-file-report`
5. 最终标记完成
   `POST /internal/tasks/{taskId}/mark-finished`

若失败：

- `POST /internal/tasks/{taskId}/mark-failed`

# 11. 接口与微服务思想的匹配说明

------

## 11.1 服务职责明确

- 用户接口由 user-service 负责
- 求解器与模板接口由 solver-service 负责
- 任务全生命周期接口由 task-service 负责
- 节点与调度接口由 scheduler-service 负责

## 11.2 数据和接口边界一致

数据库是按服务拆分的，接口也是按服务边界设计的，避免“接口归 task，数据却归 scheduler”这种混乱。

## 11.3 通过内部接口协作，而不是共享数据库

例如：

- scheduler-service 不直接操作任务表
- 它通过 task-service 内部接口更新任务状态
- task-service 不直接操作节点表
- 它通过 scheduler-service 查询节点信息

这符合微服务设计原则。

------

# 12. 接口与分布式思想的匹配说明

------

## 12.1 节点独立注册

每个 node-agent 都有独立注册和心跳接口。

## 12.2 调度与执行解耦

- 调度服务负责选节点
- 节点代理负责执行
- 两者通过派发接口和状态回传接口交互

## 12.3 节点异构能力可表达

节点注册时上报支持的求解器，调度时按 solverId 选择可用节点。

## 12.4 容错机制清晰

- 心跳丢失可判断离线
- 状态回传失败可重试
- 调度失败有专门记录接口

------

# 13. 接口与容器化思想的匹配说明

------

## 13.1 网关统一暴露服务

容器化部署后，只需暴露网关统一入口。

## 13.2 node-agent 可独立容器化

每个节点代理可单独部署为容器实例，对外暴露：

- `/internal/dispatch-task`
- `/internal/cancel-task`

## 13.3 服务地址可通过容器网络发现

在 Docker Compose 中可通过服务名调用，如：

- `http://task-service:8080`
- `http://scheduler-service:8080`

## 13.4 接口设计不依赖本地单机环境

全部采用 HTTP/JSON，天然适合容器化和分布式场景。
