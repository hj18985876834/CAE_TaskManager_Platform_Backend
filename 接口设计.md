下面给出一份**详细接口设计**。目标是：

- 满足前面的**需求分析 / 系统设计 / 数据库设计**
- 符合**微服务架构、分布式、容器化**思想
- **不设计得过重**，保证能实际开发
- 重点覆盖：**用户、求解器、任务、调度、节点代理、日志、结果、状态流转**

# 船舶CAE多求解器仿真任务管理与调度平台

## 接口设计

------

# 1. 接口设计目标

本系统接口设计需要满足以下目标：

1. 支持前后端分离开发；
2. 支持微服务拆分后的服务间调用；
3. 支持分布式节点代理接入与状态回传；
4. 支持多求解器、多任务类型、多模板的统一管理；
5. 支持任务状态流转、日志查看、结果查询等核心业务；
6. 接口设计尽量简洁，避免过度复杂，降低实现难度；
7. 统一通过网关暴露接口，便于鉴权、路由和部署。

------

# 2. 微服务与接口边界

系统采用如下服务划分：

1. `gateway-service`
2. `user-service`
3. `solver-service`
4. `task-service`
5. `scheduler-service`
6. `node-agent`

接口层面可分为四大类：

1. **前端业务接口**
   供 Web 前端调用，经网关统一转发。
2. **服务内部管理接口**
   供微服务之间调用，通常不直接暴露给前端。
3. **节点代理接口**
   中心服务与 node-agent 之间交互。
4. **文件与日志接口**
   用于上传输入文件、读取日志、下载结果文件。

------

# 3. 接口设计总体规范

------

## 3.1 URL 设计规范

统一采用 REST 风格，建议网关统一前缀：

```text
/api
```

再按服务划分路径，例如：

```text
/api/auth/...
/api/users/...
/api/solvers/...
/api/tasks/...
/api/scheduler/...
/api/nodes/...
```

------

## 3.2 请求方式规范

- `GET`：查询
- `POST`：新增 / 执行动作
- `PUT`：更新
- `DELETE`：删除或逻辑删除

对于“动作型接口”可使用：

- `POST /tasks/{id}/cancel`
- `POST /nodes/register`
- `POST /nodes/heartbeat`

这样更贴近业务语义。

------

## 3.3 统一返回结构

建议全系统统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明：

- `code`：0 表示成功，非 0 表示失败
- `message`：提示信息
- `data`：业务数据

分页接口建议：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "total": 100,
    "pageNum": 1,
    "pageSize": 10,
    "records": []
  }
}
```

------

## 3.4 鉴权规范

### 前端用户接口

采用 JWT：

```http
Authorization: Bearer {token}
```

### 节点代理接口

节点代理接口统一采用 nodeToken 鉴权，请求头为 X-Node-Token。

```http
X-Node-Token: xxxxxx
```

管理员接口和普通用户接口在网关层统一校验角色。

------

## 3.5 时间与状态字段规范

统一采用：

- 时间格式：`yyyy-MM-dd HH:mm:ss`
- 状态使用字符串枚举，不建议前期使用数字码
- 前端业务接口与服务内部接口的 JSON 字段统一使用 `camelCase`
- 数据库字段统一使用 `snake_case`，由后端实体/DTO 映射转换

例如：

- `CREATED`
- `VALIDATED`
- `QUEUED`
- `SCHEDULED`
- `DISPATCHED`
- `RUNNING`
- `SUCCESS`
- `FAILED`
- `CANCELED`
- `TIMEOUT`

字段命名映射示例：

- 数据库 `task_id` 对应接口 `taskId`
- 数据库 `node_id` 对应接口 `nodeId`
- 数据库 `submit_time` 对应接口 `submitTime`
- 数据库 `running_count` 对应接口 `runningCount`
- 数据库 `reserved_count` 对应接口 `reservedCount`
- 数据库 `strategy_name` 对应接口 `strategyName`

前后端联调时应以接口字段名为准，数据库字段名只在持久化设计和 SQL 中使用。

------

## 3.6 任务文件校验错误码规范

为便于前后端联调，任务文件校验采用“双层错误码”：

- 顶层 `code`：接口级错误码（如 `4001` 表示任务文件校验失败）
- 明细 `issues[].errorCode`：规则级错误码（用于逐条提示）

建议错误码清单如下：

| errorCode | 含义 | 典型触发场景 |
| --------- | ---- | ------------ |
| `ARCHIVE_MISSING` | 缺少输入压缩包 | 未上传 `input_archive` |
| `ARCHIVE_BROKEN` | 压缩包损坏或不可解压 | ZIP 格式错误、解压失败 |
| `ARCHIVE_UNSAFE_PATH` | 压缩包路径非法 | 包内包含 `../` 等目录穿越路径 |
| `INVALID_ARCHIVE_SUFFIX` | 压缩包后缀不合法 | 上传了非 `.zip` 文件 |
| `ARCHIVE_TOO_LARGE` | 压缩包超过大小限制 | 超过模板或系统最大上传限制 |
| `MISSING_REQUIRED_PATH` | 缺少必需目录或文件 | 例如缺少 `system/**` 或 `model.inp` |
| `INVALID_FILE_TYPE` | 文件类型不符合规则 | 要求目录却为普通文件，或反之 |
| `INVALID_FILENAME_PATTERN` | 文件名不匹配规则 | 如不满足 `model.inp` 或通配规则 |
| `INVALID_SUFFIX` | 文件扩展名不合法 | 例如需要 `.inp` 却上传 `.txt` |
| `PATH_COUNT_OUT_OF_RANGE` | 数量不在规则范围内 | 某类文件数量超出 `min/max` 约束 |
| `TEMPLATE_RULE_CONFLICT` | 模板规则冲突或不可执行 | 模板定义本身存在冲突 |
| `PARAM_SCHEMA_INVALID` | 参数与模板约束不匹配 | `params` 不满足 `paramsSchema` |

任务校验失败返回示例：

```json
{
  "code": 4001,
  "message": "任务文件校验失败",
  "data": {
    "taskId": 1002,
    "valid": false,
    "issues": [
      {
        "ruleKey": "dir_system",
        "path": "system",
        "errorCode": "MISSING_REQUIRED_PATH",
        "message": "缺少必需目录 system"
      },
      {
        "ruleKey": "input_archive",
        "path": "case.rar",
        "errorCode": "INVALID_ARCHIVE_SUFFIX",
        "message": "仅支持 zip 压缩包"
      }
    ]
  }
}
```

约定：

- `POST /api/tasks/{taskId}/validate` 必须返回上述 `issues` 结构；
- `issues` 允许多条，前端应逐条展示，不应只展示顶层 `message`。

# 4. 网关设计

网关主要负责：

1. 用户登录后 token 校验；
2. 管理员权限校验；
3. 路由转发到各微服务；
4. 统一异常处理；
5. 统一跨域处理。

网关一般不承担复杂业务逻辑，因此无需设计过多业务接口，只保留认证入口与路由规则。

# 7. task-service 接口设计

这是最核心的一组接口。

------

## 7.1 任务创建接口

### 7.1.1 创建任务

**POST** `/api/tasks`

#### 请求参数

```json
{
  "taskName": "船体稳态流体分析任务1",
  "solverId": 1,
  "profileId": 11,
  "taskType": "CFD_STEADY",
  "priority": 0,
  "params": {
    "caseName": "ship_case_01",
    "description": "基础工况"
  }
}
```

#### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,
    "taskNo": "TASK202603250001",
    "status": "CREATED"
  }
}
```

### 说明

此接口只在新建流程开始时调用一次。前端拿到 `taskId` 后，后续上传、校验和提交都必须复用该 `taskId`，不得因为校验失败再次调用创建接口。

------

### 7.1.2 编辑任务

**PUT** `/api/tasks/{taskId}`

#### 请求参数

```json
{
  "taskName": "船体稳态流体分析任务1-修订版",
  "priority": 0,
  "params": {
    "caseName": "ship_case_01_v2",
    "description": "修订后的基础工况"
  }
}
```

#### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,
    "status": "CREATED"
  }
}
```

### 说明

- 仅 `CREATED`、`VALIDATED` 状态允许调用；
- 编辑任务不会创建新任务记录；
- 如果任务此前已经处于 `VALIDATED`，则参数修改后状态回退为 `CREATED`，需要重新校验；
- 建议该接口同时用于更新任务名称、优先级和 `params`。

------

### 7.1.3 上传任务输入文件

**POST** `/api/tasks/{taskId}/files`

#### 请求方式

```
multipart/form-data
```

#### 参数

- `file`
- `fileKey`（固定为 `input_archive`）
- `fileRole`（固定为 `ARCHIVE`）

### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "fileId": 3001,
    "originName": "case.zip",
    "storagePath": "/data/tasks/1001/input/case.zip",
    "status": "CREATED"
  }
}
```

### 说明

统一采用单个 ZIP 包上传。若当前任务已经上传过输入包，则本次上传应覆盖旧文件并继续使用同一个 `taskId`，同时将任务状态重置为 `CREATED`。

------

### 7.1.4 校验任务

**POST** `/api/tasks/{taskId}/validate`

### 作用

触发：

- 表单校验
- 文件规则校验
- 模板校验

#### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,
    "valid": true,
    "status": "VALIDATED"
  }
}
```

#### 失败返回

```json
{
  "code": 4001,
  "message": "任务文件校验失败",
  "data": {
    "taskId": 1001,
    "valid": false,
    "status": "CREATED",
    "issues": [
      {
        "ruleKey": "main_inp",
        "path": "model.inp",
        "errorCode": "MISSING_REQUIRED_PATH",
        "message": "缺少必需文件 model.inp"
      }
    ]
  }
}
```

### 说明

校验接口只对已存在的任务执行校验，不得创建新的任务记录。用户修正文件或参数后再次校验，仍然针对原 `taskId` 执行。

------

### 7.1.5 提交任务入队

**POST** `/api/tasks/{taskId}/submit`

#### 返回

```json
{
  "code": 0,
  "message": "任务已进入队列",
  "data": {
    "taskId": 1001,
    "status": "QUEUED"
  }
}
```

### 说明

为什么拆成“创建 / 上传 / 校验 / 提交”四步：

- 更贴近实际流程
- 降低单接口复杂度
- 更适合大文件上传场景
- 前端交互更清晰
- 与当前已搭建好的前后端框架保持一致

补充说明：

- 前端只上传 ZIP，不上传解压后的目录；
- 后端在校验阶段完成解压、规则匹配与错误明细生成；
- 只有状态为 `VALIDATED` 的任务允许提交入队；
- 长时间未提交的 `CREATED` 任务应通过后台清理任务定期清理，避免残留数据堆积。

### 说明

- 该接口应具有“原子抢占调度权”的语义；
- 调用前，scheduler-service 应已经完成目标节点选择并成功预占该节点容量；

```json
}
```

```json
{

  "taskId": 1001,
  "nodeId": 21,
  "status": "SCHEDULED"
}
```

若任务已被其他调度实例抢占、已取消或状态已变化，也可以返回：

```json
{
  "claimed": false,
  "taskId": 1001,
  "nodeId": 21,
  "status": "QUEUED"
}
```

### 作用

将任务状态从 `QUEUED` 改为 `SCHEDULED`

### 7.6.3 修改任务为已下发

**POST** `/internal/tasks/{taskId}/mark-dispatched`

#### 请求参数

```json
{
  "nodeId": 21
}
```

#### 返回参数

```json
{
  "taskId": 1001,
  "nodeId": 21,
  "status": "DISPATCHED"
}
```

#### 说明

- 该接口只负责把任务从 `SCHEDULED` 标记为 `DISPATCHED`；
- `nodeId` 必须与 `mark-scheduled` 阶段已经绑定到任务上的节点一致；
- 该接口不负责把任务标记为 `RUNNING`，节点真正开始执行后应调用 `status-report`；
- 若调度器因网络重试重复调用，服务端应按同一 `taskId + nodeId` 做幂等处理，避免重复写入错误状态。

### 7.6.4 节点上报任务状态

**POST** `/internal/tasks/{taskId}/status-report`

#### 请求参数

```json
{
  "nodeId": 21,
  "fromStatus": "DISPATCHED",
  "toStatus": "RUNNING",
  "changeReason": "节点已启动执行",
  "operatorType": "NODE"
}
```

### 说明

node-agent 通过该接口回传运行状态。

- 首版固定只允许节点通过该接口上报 `DISPATCHED -> RUNNING`；
- 终态上报必须使用 `mark-finished` 或 `mark-failed`，不要通过该接口传入 `SUCCESS / FAILED / TIMEOUT / CANCELED`；
- `nodeId` 必须与任务当前绑定节点一致，避免其他节点误报状态；
- task-service 成功完成 `DISPATCHED -> RUNNING` 后，应调用 scheduler-service 的 `/internal/nodes/{nodeId}/release-reservation` 释放本任务对应的调度预占；
- 如果任务不在 `DISPATCHED` 状态，接口应返回状态冲突信息，不应隐式补跳或跨级修正状态。

### 7.6.8 标记任务完成

**POST** `/internal/tasks/{taskId}/mark-finished`

#### 请求参数

```json
{
  "nodeId": 21,
  "finalStatus": "SUCCESS"
}
```

#### 说明

- 该接口用于节点完成执行后的终态回传；
- 首版 `finalStatus` 只允许为 `SUCCESS` 或 `TIMEOUT`；
- `CANCELED` 仅在第二阶段实现“运行中取消”后才允许通过该接口回传；
- 业务上应要求任务当前已进入 `RUNNING`，不应从 `SCHEDULED`、`DISPATCHED` 直接跳到终态；
- 不建议在该接口内部隐式补做 `DISPATCHED -> RUNNING` 之类的多级状态跳转。

### 7.6.9 标记任务下发失败

**POST** `/internal/tasks/{taskId}/dispatch-failed`

#### 请求参数

```json
{
  "nodeId": 21,
  "failType": "DISPATCH_ERROR",
  "reason": "node-agent dispatch request failed",
  "recoverable": true
}
```

#### 说明

- 供 scheduler-service 在“已标记 `SCHEDULED` 或已进入 `DISPATCHED` 但尚未真正 `RUNNING`、派发过程失败”时回写任务状态；
- 若已选出目标节点，`nodeId` 应传入且必须与任务当前绑定节点一致；
- 当 `recoverable = true` 时，任务应回退到 `QUEUED`，并清理本轮节点绑定与运行态字段；
- 当 `recoverable = false` 时，任务可更新为 `FAILED`，并记录失败类型与原因；
- 若节点已先一步把任务推进到 `RUNNING` 或终态，则该接口应按“同一 `taskId + nodeId` 的迟到失败回写”幂等忽略，不再要求 scheduler-service 额外先查一次任务状态；
- 该接口只负责任务状态回写，不直接释放节点预占；若本轮已持有节点预占，应由 scheduler-service 额外调用 `/internal/nodes/{nodeId}/release-reservation` 完成释放；
- 用于避免把所有派发失败都直接归类为不可恢复失败。

------

### 7.6.10 标记任务失败

**POST** `/internal/tasks/{taskId}/mark-failed`

#### 请求参数

```json
{
  "nodeId": 21,
  "failType": "RUNTIME_ERROR",
  "failMessage": "solver exit code != 0"
}
```

#### 说明

- 该接口用于节点在运行阶段发生不可恢复错误后的失败回传；
- 业务上应要求任务当前已进入 `RUNNING`，避免把“派发失败”“节点拒收”等非运行态失败混入此接口；
- `SCHEDULED` 或 `DISPATCHED` 阶段的失败，应优先通过 `dispatch-failed` 或状态回退机制处理；
- 这样可以保证终态来源清晰，便于后续排查和审计。

------

### 7.6.10 节点离线补偿回写

**POST** `/internal/tasks/node-offline/fail`

#### 请求参数

```json
{
  "nodeId": 21,
  "reason": "node heartbeat timeout, scheduler marked node offline: NODE-001"
}
```

#### 返回参数

```json
{
  "code": 0,
  "message": "success",
  "data": 3
}
```

#### 说明

- 该接口由 scheduler-service 的节点心跳检测或离线补偿流程调用，不对前端或 node-agent 暴露；
- `task-service` 收到后，应按最新状态规则分流处理该节点上的受影响任务：
  - `SCHEDULED / DISPATCHED -> QUEUED`
  - `RUNNING -> FAILED`，并记录 `failType = NODE_OFFLINE`
- 若任务已进入其他终态或已不再绑定该节点，应按当前状态安全忽略，不得重复改写；
- 返回值表示本次扫描到并处理过的受影响任务数量，便于 scheduler-service 记录补偿审计日志。

### 8.3.3 预占节点容量

**POST** `/internal/nodes/{nodeId}/reserve`

#### 请求参数

```json
{
  "taskId": 1001
}
```

#### 返回参数

```json
{
  "taskId": 1001,
  "nodeId": 21,
  "reservationStatus": "RESERVED",
  "reservedCount": 2
}
```

### 说明

- 该接口用于 scheduler-service 在正式派发前增加节点 `reservedCount`；
- 只有当 `runningCount + reservedCount < maxConcurrency` 时才允许预占成功；
- `reservedCount` 只表达调度预占，不表示节点已真正开始运行；
- 预占成功必须以 `node_reservation(taskId, nodeId)` 写为 `RESERVED` 为准；
- 首次成功预占时，scheduler-service 应在同一本地事务内写入或更新 `node_reservation` 并执行 `reservedCount + 1`；
- 如果同一 `taskId + nodeId` 已为 `RESERVED`，应按幂等成功返回，不得重复增加 `reservedCount`。
- 重复预占返回当前 `reservationStatus` 与 `reservedCount`，调用方不得据此再次执行本地加计数。

------

### 8.3.4 释放节点预占容量

**POST** `/internal/nodes/{nodeId}/release-reservation`

#### 请求参数

```json
{
  "taskId": 1001
}
```

#### 返回参数

```json
{
  "taskId": 1001,
  "nodeId": 21,
  "reservationStatus": "RELEASED",
  "reservedCount": 1
}
```

### 说明

- 当节点拒收、派发失败或任务已进入 `RUNNING` 且由心跳接管真实运行数时，应释放对应 `reservedCount`；
- 正常执行链路中，node-agent 先向 task-service 上报 `RUNNING`，task-service 状态更新成功后再调用本接口释放预占；
- 调度下发失败链路中，由 scheduler-service 在回写 `dispatch-failed` 前后释放预占；
- 释放动作应按 `taskId + nodeId` 做幂等处理，幂等依据为 `node_reservation`；
- 只有当 `node_reservation(taskId, nodeId)` 从 `RESERVED` 成功更新为 `RELEASED` 时，才允许执行 `reservedCount - 1`；
- 如果该记录已为 `RELEASED`，接口应按幂等成功返回，不得重复减少 `reservedCount`；
- 如果不存在对应 `node_reservation` 记录，接口应返回明确冲突或失败，不得静默修改 `reservedCount`；
- 重复释放返回当前 `reservationStatus` 与 `reservedCount`，调用方不得据此再次执行本地减计数；
- 不再建议通过 `running-count +1/-1` 这种方式同时承担“真实运行数”和“调度预占数”两种职责。

# 10. 典型业务流程对应接口设计

------

## 10.1 用户创建并提交任务流程

1. 登录
   `POST /api/auth/login`
2. 获取求解器列表
   `GET /api/solvers`
3. 根据求解器获取任务模板
   `GET /api/solvers/{solverId}/task-options`
4. 获取上传规范
   `GET /api/profiles/{profileId}/upload-spec`
5. 创建任务
   `POST /api/tasks`
6. 编辑任务参数或名称
   `PUT /api/tasks/{taskId}`
7. 上传输入文件
   `POST /api/tasks/{taskId}/files`
8. 校验任务
   `POST /api/tasks/{taskId}/validate`
9. 提交任务入队
   `POST /api/tasks/{taskId}/submit`

------

## 10.2 调度执行流程

1. scheduler-service 查询待调度任务
   `GET /internal/tasks/queued`
2. 查询满足条件的节点
   `GET /internal/nodes/available?solverId=1`
3. scheduler-service 根据优先级、提交时间和节点负载选出目标节点
4. scheduler-service 预占目标节点容量
   `POST /internal/nodes/{nodeId}/reserve`
5. task-service 原子标记任务为 `SCHEDULED`
   `POST /internal/tasks/{taskId}/mark-scheduled`
6. scheduler-service 调用 node-agent 下发任务
   `POST /internal/dispatch-task`
7. task-service 标记任务为 `DISPATCHED`
   `POST /internal/tasks/{taskId}/mark-dispatched`
8. 记录调度日志
   `POST /internal/schedules`

若第 5 步失败：

- scheduler-service 释放节点预占
  `POST /internal/nodes/{nodeId}/release-reservation`
- scheduler-service 继续处理下一条任务或重新选择任务

若第 6 步失败：

- scheduler-service 释放节点预占
  `POST /internal/nodes/{nodeId}/release-reservation`
- task-service 根据失败是否可恢复，回退为 `QUEUED` 或标记为 `FAILED`
  `POST /internal/tasks/{taskId}/dispatch-failed`
- scheduler-service 记录调度失败日志
  `POST /internal/schedules`

### 调度排序说明

- 先按 `priority` 从高到低排序
- 优先级相同再按 `submitTime` 从早到晚排序
- 再结合节点最小负载策略选择最终执行节点

------

## 10.3 节点执行回传流程

1. node-agent 标记开始运行
   `POST /internal/tasks/{taskId}/status-report`
2. task-service 在状态成功转为 `RUNNING` 后释放节点预占
   `POST /internal/nodes/{nodeId}/release-reservation`
3. 周期上报日志
   `POST /internal/tasks/{taskId}/log-report`
4. 上报结果摘要
   `POST /internal/tasks/{taskId}/result-summary-report`
5. 上报结果文件
   `POST /internal/tasks/{taskId}/result-file-report`
6. 最终标记完成
   `POST /internal/tasks/{taskId}/mark-finished`

补充约束：

- 节点在上报终态前应已成功上报 `RUNNING`；
- 若任务仍停留在 `SCHEDULED` 或 `DISPATCHED`，应优先补齐运行态或按调度失败语义回退，不应长期依赖跨级自动补跳。

若失败：

- `POST /internal/tasks/{taskId}/mark-failed`

若属于调度下发失败而尚未进入节点执行：

- `POST /internal/tasks/{taskId}/dispatch-failed`

# 11. 接口与微服务思想的匹配说明

------

## 11.1 服务职责明确

- 用户接口由 user-service 负责
- 求解器与模板接口由 solver-service 负责
- 任务全生命周期接口由 task-service 负责
- 节点与调度接口由 scheduler-service 负责

## 11.2 数据和接口边界一致

数据库是按服务拆分的，接口也是按服务边界设计的，避免“接口归 task，数据却归 scheduler”这种混乱。

## 11.3 通过内部接口协作，而不是共享数据库

例如：

- scheduler-service 不直接操作任务表
- 它通过 task-service 内部接口更新任务状态
- task-service 不直接操作节点表
- 它通过 scheduler-service 查询节点信息

这符合微服务设计原则。

------

# 12. 接口与分布式思想的匹配说明

------

## 12.1 节点独立注册

每个 node-agent 都有独立注册和心跳接口。

## 12.2 调度与执行解耦

- 调度服务负责选节点
- 节点代理负责执行
- 两者通过派发接口和状态回传接口交互

## 12.3 节点异构能力可表达

节点注册时上报本机实际支持的求解器及版本，管理员可额外控制节点级和节点-求解器级是否允许参与调度；调度时按 solverId 从可用节点中选择。

## 12.4 容错机制清晰

- 心跳丢失可判断离线
- 状态回传失败可重试
- 调度失败有专门记录接口

------

# 13. 接口与容器化思想的匹配说明

------

## 13.1 网关统一暴露服务

容器化部署后，只需暴露网关统一入口。

## 13.2 node-agent 可独立容器化

每个节点代理可单独部署为容器实例，对外暴露：

- `/internal/dispatch-task`
- `/internal/cancel-task`

## 13.3 服务地址可通过容器网络发现

在 Docker Compose 中可通过服务名调用，如：

- `http://task-service:8080`
- `http://scheduler-service:8080`

## 13.4 接口设计不依赖本地单机环境

全部采用 HTTP/JSON，天然适合容器化和分布式场景。
