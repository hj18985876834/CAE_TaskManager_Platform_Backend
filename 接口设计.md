# 船舶CAE多求解器仿真任务管理与调度平台

## 接口设计

------

# 1. 接口设计目标

本系统接口设计需要满足以下目标：

1. 支持前后端分离开发；
2. 支持微服务拆分后的服务间调用；
3. 支持分布式节点代理接入与状态回传；
4. 支持多求解器、多任务类型、多模板的统一管理；
5. 支持任务状态流转、日志查看、结果查询等核心业务；
6. 接口设计尽量简洁，避免过度复杂，降低实现难度；
7. 统一通过网关暴露接口，便于鉴权、路由和部署。

------

# 2. 微服务与接口边界

系统采用如下服务划分：

1. `gateway-service`
2. `user-service`
3. `solver-service`
4. `task-service`
5. `scheduler-service`
6. `node-agent`

接口层面可分为四大类：

1. **前端业务接口**
   供 Web 前端调用，经网关统一转发。
2. **服务内部管理接口**
   供微服务之间调用，通常不直接暴露给前端。
3. **节点代理接口**
   中心服务与 node-agent 之间交互。
4. **文件与日志接口**
   用于上传输入文件、读取日志、下载结果文件。

# 3. 接口设计总体规范

------

## 3.1 URL 设计规范

统一采用 REST 风格，网关统一前缀为：

```text
/api
```

再按服务划分路径，例如：

```text
/api/auth/...
/api/users/...
/api/solvers/...
/api/tasks/...
/api/scheduler/...
/api/nodes/...
```

------

## 3.2 请求方式规范

- `GET`：查询
- `POST`：新增 / 执行动作
- `PUT`：更新
- `DELETE`：删除或逻辑删除

对于“动作型接口”可使用：

- `POST /tasks/{id}/cancel`
- `POST /nodes/register`
- `POST /nodes/heartbeat`

这样更贴近业务语义。

------

## 3.3 统一返回结构

全系统统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明：

- `code`：0 表示成功，非 0 表示失败
- `message`：提示信息
- `data`：业务数据

分页接口统一返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "total": 100,
    "pageNum": 1,
    "pageSize": 10,
    "records": []
  }
}
```

------

## 3.4 鉴权规范

### 前端用户接口

采用 JWT：

```http
Authorization: Bearer {token}
```

### 节点代理接口

节点代理接口统一采用 nodeToken 鉴权，请求头为 X-Node-Token。

```http
X-Node-Token: xxxxxx
```

管理员接口和普通用户接口在网关层统一校验角色。

------

## 3.5 时间与状态字段规范

统一采用：

- 时间格式：`yyyy-MM-dd HH:mm:ss`
- 状态统一使用字符串枚举，接口层禁止使用数字状态码
- 前端业务接口与服务内部接口的 JSON 字段统一使用 `camelCase`
- 数据库字段统一使用 `snake_case`，由后端实体/DTO 映射转换

例如：

- `CREATED`
- `VALIDATED`
- `QUEUED`
- `SCHEDULED`
- `DISPATCHED`
- `RUNNING`
- `SUCCESS`
- `FAILED`
- `CANCELED`
- `TIMEOUT`

核心字段命名映射如下：

| 数据库字段 | 接口字段 | 说明 |
| ---------- | -------- | ---- |
| `task_id` | `taskId` | 任务ID |
| `task_no` | `taskNo` | 任务编号 |
| `task_type` | `taskType` | 任务类型 |
| `profile_code` | `profileCode` | 模板编码 |
| `profile_name` | `profileName` | 模板名称 |
| `command_template` | `commandTemplate` | 命令模板 |
| `parser_name` | `parserName` | 结果解析器名称 |
| `node_id` | `nodeId` | 节点ID |
| `submit_time` | `submitTime` | 提交时间 |
| `start_time` | `startTime` | 开始时间 |
| `end_time` | `endTime` | 结束时间 |
| `fail_type` | `failType` | 失败类型 |
| `fail_message` | `failMessage` | 失败原因 |
| `running_count` | `runningCount` | 节点真实运行任务数 |
| `reserved_count` | `reservedCount` | 节点调度预占任务数 |
| `max_concurrency` | `maxConcurrency` | 节点最大并发数 |
| `strategy_name` | `strategyName` | 调度策略标识 |
| `unpack_dir` | `unpackDir` | 校验后真实执行目录 |
| 无落库字段 | `queueReason` | 查询时动态派生的排队原因 |

前后端联调时应以接口字段名为准，数据库字段名只在持久化设计和 SQL 中使用。

本文件中若某接口章节使用“返回字段”列举返回内容，则视为该接口首版正式返回合同；未列出的业务字段不作为前后端联调依赖项，新增字段应走文档同步。

## 3.6 任务文件校验错误码规范

为便于前后端联调，任务文件校验采用“双层错误码”：

- 顶层 `code`：接口级错误码（如 `4001` 表示任务文件校验失败）
- 明细 `issues[].errorCode`：规则级错误码（用于逐条提示）

首版错误码清单如下：

| errorCode | 含义 | 典型触发场景 |
| --------- | ---- | ------------ |
| `ARCHIVE_MISSING` | 缺少输入压缩包 | 未上传 `input_archive` |
| `ARCHIVE_BROKEN` | 压缩包损坏或不可解压 | ZIP 格式错误、解压失败 |
| `ARCHIVE_UNSAFE_PATH` | 压缩包路径非法 | 包内包含 `../` 等目录穿越路径 |
| `INVALID_ARCHIVE_SUFFIX` | 压缩包后缀不合法 | 上传了非 `.zip` 文件 |
| `ARCHIVE_TOO_LARGE` | 压缩包超过大小限制 | 超过模板或系统最大上传限制 |
| `MISSING_REQUIRED_PATH` | 缺少必需目录或文件 | 例如缺少 `system/**` 或 `model.inp` |
| `INVALID_FILE_TYPE` | 文件类型不符合规则 | 要求目录却为普通文件，或反之 |
| `INVALID_FILENAME_PATTERN` | 文件名不匹配规则 | 如不满足 `model.inp` 或通配规则 |
| `INVALID_SUFFIX` | 文件扩展名不合法 | 例如需要 `.inp` 却上传 `.txt` |
| `PATH_COUNT_OUT_OF_RANGE` | 数量不在规则范围内 | 某类文件数量超出 `min/max` 约束 |
| `TEMPLATE_RULE_CONFLICT` | 模板规则冲突或不可执行 | 模板定义本身存在冲突 |
| `PARAM_SCHEMA_INVALID` | 参数与模板约束不匹配 | `params` 不满足 `paramsSchema` |

任务校验失败返回示例：

```json
{
  "code": 4001,
  "message": "任务文件校验失败",
  "data": {
    "taskId": 1002,
    "valid": false,
    "issues": [
      {
        "ruleKey": "dir_system",
        "path": "system",
        "errorCode": "MISSING_REQUIRED_PATH",
        "message": "缺少必需目录 system"
      },
      {
        "ruleKey": "input_archive",
        "path": "case.rar",
        "errorCode": "INVALID_ARCHIVE_SUFFIX",
        "message": "仅支持 zip 压缩包"
      }
    ]
  }
}
```

约定：

- `POST /api/tasks/{taskId}/validate` 必须返回上述 `issues` 结构；
- `issues` 允许多条，前端应逐条展示，不应只展示顶层 `message`。

## 3.7 核心接口契约总表

本节用于把首版最关键的业务接口收敛为统一契约模板。前后端联调、服务间联调、测试用例设计应优先以本节为准；若与后文单接口章节存在冲突，以本节和状态流转规则表为准。

### 3.7.1 前端任务主链路

| 接口 | 调用方 | 请求关键字段 | 成功返回关键字段 | 失败/冲突语义 | 状态约束 | 幂等要求 |
| ---- | ------ | ------------ | ---------------- | ------------- | -------- | -------- |
| `POST /api/tasks` | 前端用户 | `taskName`、`solverId`、`profileId`、`taskType`、`priority`、`params` | `taskId`、`taskNo`、`status=CREATED` | 参数非法返回失败；不复用旧任务 | 新建流程起点 | 不幂等；每次成功调用产生新 `taskId` |
| `PUT /api/tasks/{taskId}` | 前端用户 | `taskName`、`priority`、`params` | `taskId`、`status` | 非 `CREATED/VALIDATED` 返回冲突 | 仅 `CREATED/VALIDATED` 可编辑 | 同一内容重复提交可视为幂等更新 |
| `POST /api/tasks/{taskId}/files` | 前端用户 | `multipart/form-data`，`file`、`fileKey=input_archive`、`fileRole=ARCHIVE` | `fileId`、`originName`、`storagePath`、`status=CREATED` | 非法文件或任务状态不允许时返回失败/冲突 | 首版上传 ZIP；重新上传后状态回到 `CREATED` | 同一任务重复上传覆盖旧归档，不新增新任务 |
| `POST /api/tasks/{taskId}/validate` | 前端用户 | 无额外 JSON；按当前任务与已上传文件校验 | `taskId`、`valid`、`status`，失败时必须返回 `issues[]` | 校验失败返回 `code=4001` 且 `status=CREATED` | 以当前任务数据为准 | 同一轮重复请求若已 `VALIDATED` 可按幂等成功返回 |
| `POST /api/tasks/{taskId}/submit` | 前端用户 | 无额外 JSON | `taskId`、`status=QUEUED`、`submitTime` | 非 `VALIDATED`、求解器/模板已禁用、模板归属或任务类型不匹配时返回冲突/失败 | 仅 `VALIDATED` 且当前求解器/模板仍有效可提交入队 | 重复提交不得重复入队 |
| `GET /api/tasks` | 前端用户 | `pageNum`、`pageSize`、筛选条件 | 分页 `records[]`，至少含 `taskId`、`status`、`canCancel`、`canRetry`、`queueReason` | 参数非法返回失败 | 查询接口 | 幂等查询 |
| `GET /api/tasks/{taskId}` | 前端用户/管理员 | 路径参数 `taskId` | 详情对象，至少含 `taskId`、`status`、`failType`、`queueReason`、`params` | 无权限或不存在返回失败 | 查询接口 | 幂等查询 |
| `POST /api/admin/tasks/{taskId}/priority` | 管理员 | `priority` | `taskId`、调整后状态 | 非允许状态返回冲突 | 仅 `CREATED / VALIDATED / QUEUED` 允许调整 | 同一优先级重复提交可按幂等成功返回 |
| `POST /api/admin/tasks/{taskId}/retry` | 管理员 | `reason` | `taskId`、`status=QUEUED` | 非 `FAILED/TIMEOUT` 返回冲突 | 仅 `FAILED/TIMEOUT` 可重试 | 重复重试不得重复清理或重复入队 |

### 3.7.2 task-service 内部状态接口

| 接口 | 调用方 | 请求关键字段 | 成功返回关键字段 | 失败/冲突语义 | 状态约束 | 幂等要求 |
| ---- | ------ | ------------ | ---------------- | ------------- | -------- | -------- |
| `GET /internal/tasks/queued` | `scheduler-service` | `limit` | 任务列表，至少含 `taskId`、`solverId`、`profileId`、`taskType`、`priority`、`submitTime` | 无可调度任务返回空列表 | 只返回 `QUEUED` 任务 | 幂等查询 |
| `POST /internal/tasks/{taskId}/mark-scheduled` | `scheduler-service` | `nodeId` | `claimed`、`taskId`、`nodeId`、`status` | 抢占失败返回 `claimed=false` 或冲突 | 仅 `QUEUED -> SCHEDULED` | 同一抢占请求不得重复绑定；同节点仍为 `SCHEDULED` 可继续派发；同节点已 `DISPATCHED` 应停止且不释放在途预占；`RUNNING/终态` 或其他失败场景不得重复派发，并按需释放本次预占 |
| `POST /internal/tasks/{taskId}/mark-dispatched` | `scheduler-service` | `nodeId` | `taskId`、`nodeId`、`status` | 非法状态返回冲突；迟到确认按同节点幂等处理 | 仅 `SCHEDULED -> DISPATCHED` | 同一 `taskId + nodeId` 重试不重复写状态 |
| `POST /internal/tasks/{taskId}/status-report` | `node-agent` | `nodeId`、`fromStatus`、`toStatus`、`changeReason`、`operatorType` | `taskId`、`status=RUNNING` | 非 `DISPATCHED` 或节点不匹配返回冲突 | 首版仅允许 `DISPATCHED -> RUNNING` | 重复 `RUNNING` 回传不得重复释放预占 |
| `POST /internal/tasks/{taskId}/dispatch-failed` | `scheduler-service` | `nodeId`、`failType`、`reason`、`recoverable`；`failType` 必须属于统一失败类型枚举 | `taskId`、回写后的 `status` | 已终态按幂等忽略；已进入 `RUNNING` 时返回冲突并补 same-status 节点审计；非法 `failType` 返回参数错误 | 仅处理 `SCHEDULED/DISPATCHED` 且未真正运行的任务 | task-service 只回写状态；预占由 scheduler-service 释放 |
| `POST /internal/tasks/{taskId}/mark-finished` | `node-agent` | `nodeId`、`finalStatus` | `taskId`、终态 `status` | 非 `RUNNING` 返回冲突 | 首版 `finalStatus` 仅 `SUCCESS/TIMEOUT` | 终态重复回传按幂等忽略 |
| `POST /internal/tasks/{taskId}/mark-failed` | `node-agent` | `nodeId`、`failType`、`failMessage`；`failType` 必须属于统一失败类型枚举且不得为 `TIMEOUT` | `taskId`、`status=FAILED` | 非 `RUNNING` 返回冲突；非法 `failType` 返回参数错误 | 仅 `RUNNING -> FAILED` | 终态重复回传按幂等忽略 |
| `POST /internal/tasks/node-offline/fail` | `scheduler-service` | `nodeId`、`reason` | `data=实际发生状态变更的任务数量` | 已终态、已解绑节点、重复补偿的任务应安全忽略 | `SCHEDULED/DISPATCHED -> QUEUED`，`RUNNING -> FAILED` | 回队任务不得重复释放预占；返回值不统计被忽略任务 |
| `POST /internal/tasks/{taskId}/log-report` | `node-agent` | `nodeId`、`seqNo`、`logContent` | 成功确认 | 非绑定节点返回冲突 | 不改变任务状态 | 按 `taskId + seqNo` 幂等 |
| `POST /internal/tasks/{taskId}/result-summary-report` | `node-agent` | `nodeId`、`successFlag`、`durationSeconds`、`summaryText`、`metrics` | 成功确认 | 非绑定节点返回冲突；`FAILED/CANCELED` 等非法阶段返回冲突并补 same-status 审计 | 仅 `RUNNING / SUCCESS / TIMEOUT` 接收 | 按 `taskId` 幂等覆盖 |
| `POST /internal/tasks/{taskId}/result-file-report` | `node-agent` | `nodeId`、`fileType`、`fileName`、`storagePath`、`fileSize` | 成功确认 | 非绑定节点返回冲突；`FAILED/CANCELED` 等非法阶段返回冲突并补 same-status 审计 | 仅 `RUNNING / SUCCESS / TIMEOUT` 接收 | 按 `taskId + fileType + fileName` 幂等覆盖 |

### 3.7.3 scheduler-service 内部协作接口

| 接口 | 调用方 | 请求关键字段 | 成功返回关键字段 | 失败/冲突语义 | 状态约束 | 幂等要求 |
| ---- | ------ | ------------ | ---------------- | ------------- | -------- | -------- |
| `GET /internal/nodes/available` | `scheduler-service` | `solverId` | 节点列表，至少含 `nodeId`、`runningCount`、`reservedCount`、`maxConcurrency` | 无可用节点返回空列表 | 仅返回满足可调度条件的节点 | 幂等查询 |
| `POST /internal/nodes/{nodeId}/reserve` | `scheduler-service` | `taskId` | `taskId`、`nodeId`、`reservationStatus`、`reservedCount` | 容量不足返回失败；无效节点返回失败 | 仅 `runningCount + reservedCount < maxConcurrency` 时可成功 | 以 `node_reservation(taskId,nodeId)` 为准，重复预占不重复加计数 |
| `POST /internal/nodes/{nodeId}/release-reservation` | `scheduler-service` 或 `task-service` | `taskId` | `taskId`、`nodeId`、`reservationStatus`、`reservedCount` | 无对应预占记录返回冲突；已释放按幂等成功 | 用于 `RUNNING`、派发失败、离线补偿后的预占释放 | 仅当 `node_reservation` 从 `RESERVED` 变为 `RELEASED` 时减少计数 |
| `POST /internal/nodes/{nodeId}/audit-reservation` | 运维/内部审计 | 无 | `nodeId`、`reservedCount`、`reservedDetailCount`、`consistent`、`issues[]` | 下游任务基础信息查询失败时显式失败 | 只读审计任务状态与预占明细是否一致 | 不修改预占明细、汇总计数或任务状态 |
| `GET /internal/nodes/{nodeId}/token/verify` | `task-service` | `nodeToken` | `true / false` | 节点不存在或 token 无效返回 `false` 或失败 | 用于 node-agent 回传 task-service 前的节点 token 校验 | 幂等查询 |
| `POST /internal/schedules` | `scheduler-service` | `taskId`、`nodeId`、`strategyName`、`scheduleStatus`、`scheduleMessage` | 成功确认 | 参数非法返回失败 | `strategyName` 固定为稳定标识，如 `FCFS_MIN_LOAD` | 重复写同一轮调度记录时不得生成冲突数据 |
| `POST /internal/tasks/{taskId}/dispatch-failed` | `node-agent` | `nodeId`、`failType`、`reason`、`recoverable`，请求头 `X-Node-Token` | `taskId`、回写后的 `status` | scheduler-service 校验节点 token 后统一编排；非法节点、非法状态或迟到失败按冲突/幂等处理 | 仅用于“节点已接收但未进入 `RUNNING` 的执行前失败”回调 scheduler-service；scheduler-service 再驱动 task-service 状态回写与预占释放 | 同一 `taskId + nodeId` 重复回调不得重复释放预占 |
| `GET /api/schedules` | 前端管理员 | 分页与筛选参数 | 调度记录分页结果 | 参数非法返回失败 | 查询接口 | 幂等查询 |
| `GET /api/tasks/{taskId}/schedules` | 前端用户/管理员 | 路径参数 `taskId` | 单任务调度记录列表 | 无权限或不存在返回失败 | 查询接口 | 幂等查询 |

### 3.7.4 首版错误处理总规则

| 场景 | 处理规则 |
| ---- | -------- |
| 参数非法、缺字段、格式错误 | 返回接口级失败，不进入业务状态流转 |
| 状态不满足前置条件 | 返回明确冲突，不做隐式补跳 |
| 节点不匹配、任务已解绑 | 返回冲突或按安全忽略处理，不能改写错误任务 |
| 同一请求因网络重试重复到达 | 按接口定义的幂等规则处理，不得重复增加队列、重复释放预占、重复生成终态 |
| 返回字段未在本文件正式列出 | 前端与调用方不得作为联调依赖 |

## 3.8 接口总规约表

本节用于给开发和论文整理提供统一总览。若需要快速确认“某接口由谁调用、受什么状态约束、是否要求幂等”，可优先查阅本表；具体请求字段和返回字段仍以 `3.7 核心接口契约总表` 与各章节详细定义为准。

| 接口 | 方法 | 调用方 | 主要用途 | 状态约束 | 幂等要求 |
| ---- | ---- | ------ | -------- | -------- | -------- |
| `/api/tasks` | `POST` | 前端用户 | 创建任务 | 新建流程起点 | 非幂等，成功即生成新任务 |
| `/api/tasks/{taskId}` | `PUT` | 前端用户 | 编辑任务 | 仅 `CREATED/VALIDATED` | 同内容重复提交可幂等 |

# 4. 网关设计

网关主要负责：

1. 用户登录后 token 校验；

网关一般不承担复杂业务逻辑，因此无需设计过多业务接口，只保留认证入口与路由规则。

### 6.2.5.1 模板配置约束

`commandTemplate` 应采用轻量模板写法：

- 只允许描述主执行命令，例如 `${solverExecPath} -case ${taskDir}` 或 `${solverExecPath} ${taskDir}/model`；
- 首版只允许使用稳定运行上下文占位符和任务参数；稳定运行上下文占位符包括 `${taskId}`、`${taskNo}`、`${solverId}`、`${solverCode}`、`${solverExecMode}`、`${solverExecPath}`、`${profileId}`、`${taskType}`、`${taskDir}`、`${inputDir}`、`${outputDir}`、`${logDir}`；
- 任务参数占位符必须来自 `paramsSchema/paramsSchemaJson` 的 JSON Schema `properties`，模板新增或编辑时后端必须前置拒绝未知占位符，例如 `${taskdir}` 不能被当作 `${taskDir}` 自动容错；
- `taskDir` 固定表示节点侧最终执行目录，由后端校验阶段产生的 `unpackDir` 映射得到；首版不支持 `${workDir}` 作为对外模板占位符；
- 禁止把目录创建、结果复制、退出码包装、多段 shell 流程直接写入 `commandTemplate`；
- `paramsSchema/paramsSchemaJson` 如非空，必须是 JSON object；首版后端保存模板时应校验 `properties`、`required`、`items`、`additionalProperties` 和字符串 `pattern` 等已支持字段的基本合法性，避免非法参数 schema 延迟到任务创建或校验阶段才暴露；
- 求解器特有的命令构建、输入准备和结果归集逻辑，应优先由节点代理适配器或后端执行策略承担。

文件规则中的 `ruleJson` 应优先表达简单约束：

- 首版允许使用 `allowSuffix`、`minCount`、`maxCount`、`maxSizeMb`；
- 在确有必要时，也允许使用少量作用域选择器，如 `solverCodes`、`taskTypes`，以及受控的 `deriveParam/deriveParams` 做参数派生；`deriveParam` 与 `deriveParams` 同一条规则内互斥，派生项首版仅支持 `name/source/value/pattern/group/required/trim/stripQuotes/sanitizeRegex/preprocess`，`preprocess` 子项仅支持 `pattern/replacement`；`deriveParam.pattern`、`sanitizeRegex` 和 `preprocess.pattern` 必须在保存规则时完成正则合法性校验；
- 首版正式支持的作用域选择器为 `solverId/solverIds`、`profileId/profileIds`、`taskType/taskTypes`、`solverCode/solverCodes`、`solverExecMode/solverExecModes`、`solverExecPath/solverExecPaths`；未知选择器必须在保存文件规则时拒绝，存量未知选择器在任务校验时按模板规则冲突处理；
- 复杂正则派生、命令推导等逻辑不应作为普通模板录入的常规要求；
- `fileType` 首版只允许 `ZIP`、`FILE`、`DIR`、`DIRECTORY`，新增/编辑文件规则时必须拒绝其他值；任务校验遇到历史非法 `fileType` 时按 `TEMPLATE_RULE_CONFLICT` 处理；
- `allowSuffix` 支持填写 `zip` 或 `.zip`，后端保存校验、上传校验、解压后文件校验和 `upload-spec` 返回时均按去除前导点的小写后缀语义处理；
- 如确需支持复杂求解器特性，应同步更新设计文档和适配器实现，避免只在数据库中形成隐式逻辑。
- `requiredFlag=1` 表示该规则必须至少存在匹配项，不能与 `maxCount/max=0` 同时配置；若配置 `requiredFlag=0` 但 `minCount/min>0`，系统仍按数量约束要求至少匹配对应数量，模板录入时应避免这种容易误解的组合。
- `maxSizeMb` 必须是正整数，`minCount/min/maxCount/max/group` 必须是非负整数；历史 `ruleJson` 若违反上述合同，任务校验阶段统一按 `TEMPLATE_RULE_CONFLICT` 处理。

```json
        "requiredFlag": 1,
        "description": "解压后必须存在 system 目录"
      }
    ]
  }
}
```

`upload-spec` 是前端创建任务与上传文件的主要配置来源，必须返回 `uploadMode`、`archiveRule`、`fileRules`、`paramsSchema/paramsSchemaJson` 等字段；前端应使用 `paramsSchema/paramsSchemaJson` 渲染任务参数表单，后端在任务创建、任务更新和任务校验阶段再次执行参数校验。

# 7. task-service 接口设计

这是最核心的一组接口。

## 7.1 任务创建接口

### 7.1.1 创建任务

**POST** `/api/tasks`

#### 请求参数

```json
{
  "taskName": "船体稳态流体分析任务1",
  "solverId": 1,
  "profileId": 1,
  "taskType": "CFD_STEADY",
  "priority": 0,
  "params": {
    "caseName": "ship_case_01",
    "description": "基础工况"
  }
}
```

#### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,
    "taskNo": "TASK202603250001",
    "status": "CREATED"
  }
}
```

### 说明

此接口只在新建流程开始时调用一次。前端拿到 `taskId` 后，后续上传、校验和提交都必须复用该 `taskId`，不得因为校验失败再次调用创建接口。

------

### 7.1.2 编辑任务

**PUT** `/api/tasks/{taskId}`

#### 请求参数

```json
{
  "taskName": "船体稳态流体分析任务1-修订版",
  "priority": 0,
  "params": {
    "caseName": "ship_case_01_v2",
    "description": "修订后的基础工况"
  }
}
```

#### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,
    "status": "CREATED"
  }
}
```

### 说明

- 仅 `CREATED`、`VALIDATED` 状态允许调用；
- 编辑任务不会创建新任务记录；
- 如果任务此前已经处于 `VALIDATED`，则参数修改后状态回退为 `CREATED`，需要重新校验；
- 该接口用于更新任务名称、优先级和 `params`。

------

### 7.1.3 上传任务输入文件

**POST** `/api/tasks/{taskId}/files`

#### 请求方式

```
multipart/form-data
```

#### 参数

- `file`
- `fileKey`（固定为 `input_archive`）
- `fileRole`（固定为 `ARCHIVE`）

### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "fileId": 3001,
    "originName": "case.zip",
    "storagePath": "/data/tasks/1001/input/case.zip",
    "status": "CREATED"
  }
}
```

### 说明

统一采用单个 ZIP 包上传。若当前任务已经上传过输入包，则本次上传应覆盖旧文件并继续使用同一个 `taskId`，同时将任务状态重置为 `CREATED`。

------

### 7.1.4 校验任务

**POST** `/api/tasks/{taskId}/validate`

### 作用

触发：

- 表单校验
- 文件规则校验
- 模板校验

#### 返回

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": 1001,
    "valid": true,
    "status": "VALIDATED"
  }
}
```

#### 失败返回

```json
{
  "code": 4001,
  "message": "任务文件校验失败",
  "data": {
    "taskId": 1001,
    "valid": false,
    "status": "CREATED",
    "issues": [
      {
        "ruleKey": "main_inp",
        "path": "model.inp",
        "errorCode": "MISSING_REQUIRED_PATH",
        "message": "缺少必需文件 model.inp"
      }
    ]
  }
}
```

### 说明

校验接口只对已存在的任务执行校验，不得创建新的任务记录。用户修正文件或参数后再次校验，仍然针对原 `taskId` 执行。

------

### 7.1.5 提交任务入队

**POST** `/api/tasks/{taskId}/submit`

#### 返回

```json
{
  "code": 0,
  "message": "任务已进入队列",
  "data": {
    "taskId": 1001,
    "status": "QUEUED",
    "submitTime": "2026-03-25 10:10:40"
  }
}
```

### 说明

为什么拆成“创建 / 上传 / 校验 / 提交”四步：

- 更贴近实际流程
- 降低单接口复杂度
- 更适合大文件上传场景
- 前端交互更清晰
- 与当前已搭建好的前后端框架保持一致

补充说明：

- 前端只上传 ZIP，不上传解压后的目录；
- 后端在校验阶段完成解压、规则匹配与错误明细生成；
- 只有状态为 `VALIDATED` 的任务允许提交入队；
- 长时间未提交的 `CREATED` 任务应通过后台清理任务定期清理，避免残留数据堆积。

- `pageNum`
- `pageSize`

- `status`

- `taskType`
- `priority`

- `taskType`

- `solverId`

- `profileName`

- `canCancel`
- `canRetry`
- `queueReason`：当任务处于 `QUEUED` 时返回当前排队原因说明，否则为 `null`

#### 说明

`queueReason` 作为查询时动态派生的用户解释字段返回，不要求单独落库，也不作为调度审计字段。真正的调度过程、失败原因与状态变化仍应以 `task_status_history` 和调度记录为准。首版固定按以下优先顺序生成简短说明：

- 若当前任务在真实排队顺序中前方仍有更高优先级或更早提交的任务，则返回“前方仍有更高优先级或更早提交的任务，等待调度”
- 若当前任务已排到前面，但不存在满足条件的在线且启用节点，则返回“暂无满足条件的可用节点”
- 若当前任务已排到前面，且存在候选节点但都已满载，则返回“候选节点当前满载，等待资源释放”
- 其余情况返回“排队中，等待调度器处理”

### 7.2.2 管理员查询全量任务

**GET** `/api/admin/tasks`

#### 查询参数

- 继承 `/api/tasks` 的全部查询参数
- `userId`
- `nodeId`
- `failType`

#### 返回字段

- 与 `/api/tasks` 的分页记录字段保持一致
- 每条记录额外返回 `userId`
- 每条记录额外返回 `username`
- 每条记录额外返回 `failType`

#### 说明

- 仅管理员可用；
- 本接口与普通用户任务列表共用相同的任务状态口径、`canCancel`、`canRetry` 和 `queueReason` 派生规则；
- `nodeId` 允许为空，表示任务尚未选定节点，或调度失败发生在选点之前；
- `failType` 在任务未失败时返回 `null`；
- 分页结构、排序规则与 `/api/tasks` 保持一致，首版固定按 `createdAt` 或 `submitTime` 倒序返回，避免前后端各自定义默认排序。

## 7.6 服务内部接口

### 7.6.1 调度前查询待执行任务

**GET** `/internal/tasks/queued`

#### 查询参数

- `limit`

#### 返回

返回若干 `QUEUED` 状态任务的基础信息：

- `taskId`
- `solverId`
- `profileId`
- `taskType`
- `priority`
- `submitTime`

### 说明

- 调度服务应优先按 `priority` 降序取任务

### 7.6.2 修改任务为已调度

**POST** `/internal/tasks/{taskId}/mark-scheduled`

### 说明

- 该接口应具有“原子抢占调度权”的语义；
- 调用前，scheduler-service 应已经完成目标节点选择并成功预占该节点容量；
- 仅当任务当前状态仍为 `QUEUED` 时，才允许绑定 `nodeId` 并转为 `SCHEDULED`；
- 若任务已被取消、已被其他调度实例抢占或已不处于 `QUEUED`，则应返回明确的状态冲突信息或 `claimed = false`，避免重复派发。

#### 请求参数

```json
{
  "nodeId": 21
}
```

#### 返回参数

```json
{
  "claimed": true,
  "taskId": 1001,
  "nodeId": 21,
  "status": "SCHEDULED"
}
```

若任务已被其他调度实例抢占、已取消或状态已变化，也可以返回：

```json
{
  "claimed": false,
  "taskId": 1001,
  "nodeId": 21,
  "status": "QUEUED"
}
```

### 作用

将任务状态从 `QUEUED` 改为 `SCHEDULED`

- 若 `claimed=false` 后判断为“同一 `taskId + nodeId` 仍为 `SCHEDULED`”，scheduler-service 可继续本轮下发，用于覆盖抢占响应丢失后的重试；
- 若 `claimed=false` 且同一 `taskId + nodeId` 已为 `DISPATCHED`，scheduler-service 应停止本轮且不得释放仍由在途派发持有的预占，不额外补写失败调度记录；
- 若 `claimed=false` 且任务已进入 `RUNNING`、终态，或已被其他节点/其他调度实例抢占，scheduler-service 在按规则释放本轮预占后，应尽量补写一条 `schedule_record(FAILED)` 短说明，标记“本轮调度抢占已被拒绝”，避免管理员只能看到状态历史却无法理解本轮调度为何未继续。
- 若 `mark-scheduled` 多次网络重试后仍无法取得结构化 ACK，scheduler-service 应先通过任务基础信息查询探测当前 `status/nodeId`，再复用上述 `claimed=false` 规则处理；同节点已为 `SCHEDULED` 时继续下发，同节点已为 `DISPATCHED` 时停止且不释放预占。

------

### 7.6.3 修改任务为已下发

**POST** `/internal/tasks/{taskId}/mark-dispatched`

#### 请求参数

```json
{
  "nodeId": 21
}
```

#### 返回参数

```json
{
  "taskId": 1001,
  "nodeId": 21,
  "status": "DISPATCHED"
}
```

#### 说明

- 该接口只负责把任务从 `SCHEDULED` 标记为 `DISPATCHED`；
- `nodeId` 必须与 `mark-scheduled` 阶段已经绑定到任务上的节点一致；
- 该接口不负责把任务标记为 `RUNNING`，节点真正开始执行后应调用 `status-report`；
- 正常确认派发时返回 `status = DISPATCHED`；若同一节点的迟到确认发生在任务已推进到 `RUNNING` 等后续阶段，可按同节点幂等返回当前状态；
- 若调度器因网络重试重复调用，服务端应按同一 `taskId + nodeId` 做幂等处理，避免重复写入错误状态。

### 7.6.4 节点上报任务状态

**POST** `/internal/tasks/{taskId}/status-report`

#### 请求参数

```json
{
  "nodeId": 21,
  "fromStatus": "DISPATCHED",
  "toStatus": "RUNNING",
  "changeReason": "节点已启动执行",
  "operatorType": "NODE"
}
```

### 说明

node-agent 通过该接口回传运行状态。

- 首版固定只允许节点通过该接口上报 `DISPATCHED -> RUNNING`；
- 终态上报必须使用 `mark-finished` 或 `mark-failed`，不要通过该接口传入 `SUCCESS / FAILED / TIMEOUT / CANCELED`；
- `nodeId` 必须与任务当前绑定节点一致，避免其他节点误报状态；
- task-service 成功完成 `DISPATCHED -> RUNNING` 后，应调用 scheduler-service 的 `/internal/nodes/{nodeId}/release-reservation` 释放本任务对应的调度预占；
- 如果任务不在 `DISPATCHED` 状态，接口应返回状态冲突信息，不应隐式补跳或跨级修正状态。

### 7.6.5 节点上报日志

**POST** `/internal/tasks/{taskId}/log-report`

#### 请求参数

```json
{
  "nodeId": 21,
  "seqNo": 1,
  "logContent": "Start solver...\n"
}
```

#### 说明

- `nodeId` 必须与任务当前绑定节点一致；
- `seqNo` 在同一任务内应保持递增，服务端按 `taskId + seqNo` 做幂等写入，避免网络重试导致重复日志块；
- `logContent` 字段必须存在，但允许为空字符串，用于保留求解器原始输出中的空行；不得因空行拒绝日志分片。
- node-agent 按行读取求解器 stdout/stderr 时应在上报内容中保留行分隔符，例如每行追加 `\n`；task-service 下载完整日志时按 `seqNo` 直接拼接分片，不额外猜测换行。
- 若任务已进入终态，后续迟到日志可忽略或只作为补充日志保存，但不得改变任务状态。

### 7.6.6 节点上报结果摘要

**POST** `/internal/tasks/{taskId}/result-summary-report`

#### 请求参数

```json
{
  "nodeId": 21,
  "successFlag": 1,
  "durationSeconds": 380,
  "summaryText": "求解完成",
  "metrics": {
    "maxStress": 245.6,
    "maxDisplacement": 1.23
  }
}
```

#### 说明

- `nodeId` 必须与任务当前绑定节点一致；
- 同一任务的结果摘要重复上报时按 `taskId` 幂等覆盖或忽略，不应生成多条互相冲突的摘要记录；
- 结果摘要不负责改变任务终态，终态仍以 `mark-finished` 或 `mark-failed` 为准。
- 结果摘要只允许在 `RUNNING / SUCCESS / TIMEOUT` 阶段接收；已 `FAILED / CANCELED` 的迟到节点回传不得继续写入成功结果。
- 若中心已先进入 `FAILED / CANCELED` 等不允许结果写入的终态，迟到 `result-summary-report` 应返回状态冲突，并补一条 same-status 的节点审计说明，用于解释“迟到结果摘要回调已被拒绝”。
- 为避免与 `mark-finished / mark-failed / node-offline` 等终态更新并发穿透，task-service 应在结果摘要阶段校验与摘要落库之间持有同一任务行锁，保证“允许阶段判断”和“结果写入”原子一致。

### 7.6.7 节点上报结果文件

**POST** `/internal/tasks/{taskId}/result-file-report`

#### 请求参数

```json
{
  "nodeId": 21,
  "fileType": "RESULT",
  "fileName": "result.dat",
  "storagePath": "/data/tasks/1001/output/result.dat",
  "fileSize": 20480
}
```

#### 说明

- `nodeId` 必须与任务当前绑定节点一致；
- 同一任务下相同 `fileType + fileName` 的结果文件重复上报时应幂等处理，避免重复展示；
- `fileName` 仍必须是普通文件名，不得包含路径分隔符或控制字符，长度不得超过 255；若 `outputDir` 下存在嵌套目录，为避免同名文件覆盖，节点应将相对目录信息编码进展示名，例如 `a/result.dat` 可上报为 `a__result.dat`，深层路径编码后超过长度限制时应使用稳定截断加短哈希的方式保持唯一性；
- `fileName` 是前端展示和幂等去重使用的展示名，不要求等于 `storagePath` 的物理路径末段；
- `fileType` 必须限定为 `RESULT / LOG / REPORT / IMAGE`，非法类型应拒绝落库；
- `storagePath` 只作为后端内部定位使用，必须位于当前任务 `output/` 目录下且文件真实存在；前端下载仍通过结果文件下载接口完成。
- 结果文件只允许在 `RUNNING / SUCCESS / TIMEOUT` 阶段接收；已 `FAILED / CANCELED` 的迟到节点回传不得继续写入成功结果文件。
- 若中心已先进入 `FAILED / CANCELED` 等不允许结果写入的终态，迟到 `result-file-report` 应返回状态冲突，并补一条 same-status 的节点审计说明，用于解释“迟到结果文件回调已被拒绝”。
- 为避免与终态更新并发穿透，task-service 应在结果文件阶段校验与元数据落库之间持有同一任务行锁，保证不会在任务已并发转入 `FAILED / CANCELED` 后仍写入成功结果元数据。

### 7.6.8 标记任务完成

**POST** `/internal/tasks/{taskId}/mark-finished`

#### 请求参数

```json
{
  "nodeId": 21,
  "finalStatus": "SUCCESS"
}
```

#### 说明

- 该接口用于节点完成执行后的终态回传；
- 首版 `finalStatus` 只允许为 `SUCCESS` 或 `TIMEOUT`；
- `CANCELED` 仅在第二阶段实现“运行中取消”后才允许通过该接口回传；
- 业务上应要求任务当前已进入 `RUNNING`，不应从 `SCHEDULED`、`DISPATCHED` 直接跳到终态；
- 禁止在该接口内部隐式补做 `DISPATCHED -> RUNNING` 之类的多级状态跳转。
- 状态更新成功后，应尽力再次释放该任务在 scheduler-service 中可能残留的预占；该动作只用于修复 `DISPATCHED -> RUNNING` 时历史释放失败遗留的 `RESERVED` 明细，若释放失败只能记录审计，不得回滚终态；
- 若任务已先进入任一终态，中心可幂等返回当前终态；node-agent 客户端不得把这种终态 ACK 误判为回调失败。
- 对于这类迟到 `mark-finished`，task-service 应补一条不改变状态的节点审计说明（same-status `task_status_history`），用于标记“迟到终态确认已被幂等接受”，便于排查 ACK 丢失或重复回调。

------

### 7.6.9 标记任务下发失败

**POST** `/internal/tasks/{taskId}/dispatch-failed`

#### 请求参数

```json
{
  "nodeId": 21,
  "failType": "DISPATCH_ERROR",
  "reason": "node-agent dispatch request failed",
  "recoverable": true
}
```

#### 说明

- 供 scheduler-service 在“已标记 `SCHEDULED` 或已进入 `DISPATCHED` 但尚未真正 `RUNNING`、派发过程失败”时回写任务状态；
- 若已选出目标节点，`nodeId` 应传入且必须与任务当前绑定节点一致；
- 当 `recoverable = true` 时，任务应回退到 `QUEUED`，并清理本轮节点绑定与运行态字段；
- 当 `recoverable = false` 时，任务可更新为 `FAILED`，并记录失败类型与原因；
- node-agent 在调用本接口前，应先确认失败发生在真正上报 `RUNNING` 之前；执行器选择失败、命令模板预检失败、模板变量非法等都属于该范围，不应先写 `RUNNING` 再改走 `mark-failed`；
- scheduler-service 从 node-agent 接收“拒收本次派发”的信号时，应统一收口为正式短说明 `node-agent rejected dispatch request`；其中既包括显式 `accepted=false`，也包括 node-agent 直接返回 `409/CONFLICT`（如 `node is busy`）这类业务拒收，避免把节点内部文案直接写入调度审计；
- 若节点已先一步把任务推进到终态，则该接口可按“同一 `taskId + nodeId` 的迟到失败回写”幂等忽略，不再要求 scheduler-service 额外先查一次任务状态；
- 若中心已先进入 `RUNNING`，则该接口必须返回状态冲突，并补一条 same-status 的节点审计说明，标记“迟到 dispatch-failed 已被拒绝”，用于解释 `RUNNING` ACK 丢失后的执行前失败兜底请求；
- scheduler-service 收到上述状态冲突后，应尽量补写一条 `schedule_record(FAILED)` 短说明 `dispatch-failed rejected, task already running`，用于把该次执行前失败兜底被拒绝的事实保留到调度时间线；
- 该接口只负责 task-service 内部状态回写，不读取 `X-Node-Token`，也不直接释放本轮预占；派发失败的节点鉴权、预占释放和调度失败记录由 scheduler-service 统一编排；
- `reason` 最大长度为 255；node-agent / scheduler-service 在把异常消息转换为该字段前必须先做默认值兜底和长度截断，避免派发失败兜底回调因字段校验失败而无法收口；
- scheduler-service 在收到该接口成功 ACK 后调用 `/internal/nodes/{nodeId}/release-reservation` 释放本轮预占；若释放失败，应尽量通过 `/internal/schedules` 写入一条 `scheduleStatus=FAILED` 的调度审计记录，保留 `taskId/nodeId` 与失败说明；
- 用于避免把所有派发失败都直接归类为不可恢复失败。

------

### 7.6.10 标记任务失败

**POST** `/internal/tasks/{taskId}/mark-failed`

#### 请求参数

```json
{
  "nodeId": 21,
  "failType": "RUNTIME_ERROR",
  "failMessage": "solver exit code != 0"
}
```

#### 说明

- 该接口用于节点在运行阶段发生不可恢复错误后的失败回传；
- 业务上应要求任务当前已进入 `RUNNING`，避免把“派发失败”“节点拒收”等非运行态失败混入此接口；
- `SCHEDULED` 或 `DISPATCHED` 阶段的失败，应优先通过 `dispatch-failed` 或状态回退机制处理；
- `failMessage` 最大长度为 255；node-agent 在上报运行失败、执行前歧义兜底失败、求解器成功后结果/终态回调失败时，必须先把异常消息归一化到该长度内，避免终态收口再次因字段长度失败而让任务长期停留 `RUNNING`；
- 当求解器已成功结束，但结果摘要、结果文件或 `mark-finished` 回调持续失败时，node-agent 应使用 `failType=CALLBACK_ERROR` 调用本接口，而不是继续复用 `RUNTIME_ERROR`；
- `CALLBACK_ERROR` 成功落库后，task-service 应尽力清理该任务已写入的结果摘要与结果文件元数据，避免 `FAILED` 任务继续展示成功结果；若清理失败，只允许记录告警，不得回滚已完成的 `FAILED` 终态；
- 状态更新成功后，应尽力再次释放该任务在 scheduler-service 中可能残留的预占；该动作只用于修复 `DISPATCHED -> RUNNING` 时历史释放失败遗留的 `RESERVED` 明细，若释放失败只能记录审计，不得回滚终态；
- 这样可以保证终态来源清晰，便于后续排查和审计；
- 若任务已先进入任一终态，中心可幂等返回当前终态；node-agent 客户端不得把这种终态 ACK 误判为回调失败。
- 对于这类迟到 `mark-failed`，task-service 应补一条不改变状态的节点审计说明（same-status `task_status_history`），用于区分“真实失败终态提交”和“迟到失败回调被幂等忽略”。

------

### 7.6.11 节点离线补偿回写

**POST** `/internal/tasks/node-offline/fail`

#### 请求参数

```json
{
  "nodeId": 21,
  "reason": "node heartbeat timeout, scheduler marked node offline: NODE-001"
}
```

#### 返回参数

```json
{
  "code": 0,
  "message": "success",
  "data": 3
}
```

#### 说明

- 该接口由 scheduler-service 的节点心跳检测或离线补偿流程调用，不对前端或 node-agent 暴露；
- `task-service` 收到后，应按最新状态规则分流处理该节点上的受影响任务：
  - `SCHEDULED / DISPATCHED -> QUEUED`
  - `RUNNING -> FAILED`，并记录 `failType = NODE_OFFLINE`
- 对于被回退到 `QUEUED` 的 `SCHEDULED / DISPATCHED` 任务，`NODE_OFFLINE` 只写入状态历史或调度记录，不写入任务主记录 `failType`；
- 节点离线补偿对每条实际完成状态变更的任务，应尽量补写一条 `schedule_record` 失败审计，保留 `taskId/nodeId/scheduleStatus=FAILED/scheduleMessage`，便于管理员在调度时间线上直接看到该轮调度最终因节点离线回队或失败；
- 对于被回退到 `QUEUED` 的 `SCHEDULED / DISPATCHED` 任务，task-service 在状态更新成功后必须调用 scheduler-service 的 `/internal/nodes/{nodeId}/release-reservation` 释放对应预占；
- 释放预占仍以 `node_reservation(taskId, nodeId)` 为幂等依据，重复补偿不得导致 `reservedCount` 被重复减少；
- 若释放预占失败，不得回滚已完成的离线状态补偿，应记录告警并尽量写入 `schedule_record` 失败审计，后续通过 `audit-reservation` 或 `reconcile-reservation` 排查处理；
- 对于被收口为 `FAILED(NODE_OFFLINE)` 的 `RUNNING` 任务，若仍残留历史预占明细，状态更新成功后也应尽力释放；释放失败只记录审计，不得回滚终态；
- 对于被收口为 `FAILED(NODE_OFFLINE)` 的 `RUNNING` 任务，若节点在离线前已成功写入部分结果摘要或结果文件元数据，task-service 也应尽力清理这些成功结果元数据，避免失败任务继续展示成功结果；清理失败只记录告警，不得回滚终态；
- 若任务已进入其他终态或已不再绑定该节点，应按当前状态安全忽略，不得重复改写；
- 返回值表示本次实际发生状态变更的任务数量，不包含因已终态、已解绑节点或重复补偿而被安全忽略的任务，便于 scheduler-service 记录补偿审计日志。

- 仅管理员可用；
- 该接口只控制“节点是否支持某求解器参与调度”，不改变节点运行态 `status`；
- 若节点不存在、求解器不存在或两者未建立能力关系，应返回明确失败；
- 重复设置为相同 `enabled` 值时按幂等成功返回；
- 关闭后不影响该节点上已在运行中的任务，但后续调度不再将该节点作为对应求解器候选节点。

### 8.3.2 记录调度结果

**POST** `/internal/schedules`

#### 请求参数

```json
{
  "taskId": 1001,
  "nodeId": 21,
  "strategyName": "FCFS_MIN_LOAD",
  "scheduleStatus": "SUCCESS",
  "scheduleMessage": "分配成功"
}
```

### 说明

- `strategyName` 统一使用稳定策略标识 `FCFS_MIN_LOAD`；
- `nodeId` 在成功选出目标节点时必填；
- 若调度失败发生在选出节点之前，例如没有可用节点，`nodeId` 可以为 `null`；
- 若已经选出节点但后续预占、确认或派发失败，`nodeId` 应保留实际目标节点，便于审计；
- `scheduleMessage` 允许为空；服务端写入调度成功/失败记录时应补默认短说明并截断到 255 字符以内，避免调用方异常长消息导致审计记录落库失败。

------

### 8.3.3 预占节点容量

**POST** `/internal/nodes/{nodeId}/reserve`

#### 请求参数

```json
{
  "taskId": 1001
}
```

#### 返回参数

```json
{
  "taskId": 1001,
  "nodeId": 21,
  "reservationStatus": "RESERVED",
  "reservedCount": 2
}
```

### 说明

- 该接口用于 scheduler-service 在正式派发前增加节点 `reservedCount`；
- 只有当 `runningCount + reservedCount < maxConcurrency` 时才允许预占成功；
- `reservedCount` 只表达调度预占，不表示节点已真正开始运行；
- 预占成功必须以 `node_reservation(taskId, nodeId)` 写为 `RESERVED` 为准；
- 首次成功预占时，scheduler-service 应在同一本地事务内写入或更新 `node_reservation` 并执行 `reservedCount + 1`；
- 如果同一 `taskId + nodeId` 已为 `RESERVED`，应按幂等成功返回，不得重复增加 `reservedCount`。
- 重复预占返回当前 `reservationStatus` 与 `reservedCount`，调用方不得据此再次执行本地加计数。

------

### 8.3.4 释放节点预占容量

**POST** `/internal/nodes/{nodeId}/release-reservation`

#### 请求参数

```json
{
  "taskId": 1001
}
```

#### 返回参数

```json
{
  "taskId": 1001,
  "nodeId": 21,
  "reservationStatus": "RELEASED",
  "reservedCount": 1
}
```

### 说明

- 当节点拒收、派发失败或任务已进入 `RUNNING` 且由心跳接管真实运行数时，应释放对应 `reservedCount`；
- 正常执行链路中，node-agent 先向 task-service 上报 `RUNNING`，task-service 状态更新成功后再调用本接口释放预占；
- 调度下发失败链路中，由 scheduler-service 在回写 `dispatch-failed` 前后释放预占；
- 若 node-agent 已明确 accepted，但 scheduler-service 随后的 `mark-dispatched` 确认因网络或 task-service 短暂异常失败，scheduler-service 不得再主动调用 `dispatch-failed` 回队，只记录调度确认失败；该记录的 `scheduleMessage` 应明确区分“节点已接收但中心确认失败”和“节点拒收/派发失败”；若中心实际已进入 `DISPATCHED`，节点后续可正常 `status-report`，若仍停留 `SCHEDULED`，由 node-agent 执行前 `RUNNING` 回传失败后的 `dispatch-failed` 闭环处理；
- scheduler-service 调用 node-agent `/internal/dispatch-task` 时，若遇到网络异常、空响应或网关类失败，应先进行有限重试，再按派发失败处理；重试依赖 node-agent 对同一 `taskId` 的幂等 accepted 语义，不得导致重复启动执行线程。
- 若重试全部耗尽，scheduler-service 应先调用 node-agent 只读 runtime 探测接口确认该 `taskId` 是否已在节点 active；若已 active，应视为节点已接收并继续 `mark-dispatched` 确认，不得直接 `dispatch-failed` 回队。
- 若 node-agent 已接收任务但在进入 `RUNNING` 前准备失败，应先回调 scheduler-service 的 `POST /internal/tasks/{taskId}/dispatch-failed`，再由 scheduler-service 统一驱动 task-service 回写任务状态与释放预占；
- 若 node-agent 的 `status-report(RUNNING)` 重试耗尽后改走 `dispatch-failed(recoverable=true)`，但中心因任务已经进入 `RUNNING` 而拒绝回队，node-agent 应改用 `mark-failed(RUNTIME_ERROR)` 收口该运行态任务，避免本地未启动求解器而中心长期停留 `RUNNING`；
- 释放动作应按 `taskId + nodeId` 做幂等处理，幂等依据为 `node_reservation`；
- 只有当 `node_reservation(taskId, nodeId)` 从 `RESERVED` 成功更新为 `RELEASED` 时，才允许执行 `reservedCount - 1`；
- 如果该记录已为 `RELEASED`，接口应按幂等成功返回，不得重复减少 `reservedCount`；
- 如果不存在对应 `node_reservation` 记录，接口应返回明确冲突或失败，不得静默修改 `reservedCount`；
- 重复释放返回当前 `reservationStatus` 与 `reservedCount`，调用方不得据此再次执行本地减计数；
- 禁止通过 `running-count +1/-1` 这种方式同时承担“真实运行数”和“调度预占数”两种职责。

------

### 8.3.5 校验节点鉴权 token

**GET** `/internal/nodes/{nodeId}/token/verify`

#### 查询参数

- `nodeToken`

#### 返回参数

```json
true
```

### 说明

- 该接口仅供中心服务内部调用，首版调用方为 task-service；

```json
{
  "nodeCode": "node-01",
```

### 9.2.1 下发任务到节点

**POST** `http://node-agent-host:port/internal/dispatch-task`

#### 请求参数

```json
{
```

- scheduler-service 对网络异常、空响应或网关类失败可以有限重试本接口；node-agent 必须依靠同一 `taskId` 幂等保护保证重试安全。

### 9.2.2 查询节点本地任务运行态

```http
X-Node-Token: <node-token>
```

响应：

```json
{
  "code": 0,
  "message": "success",
```

说明：

- 该接口为只读探测接口，只返回节点本地是否仍持有该任务，不修改任务状态；

- `active=true` 表示节点已接收该任务，scheduler-service 应继续中心侧 `mark-dispatched` 确认，而不是直接按派发失败回队；
- `runningReported=true` 表示节点已经完成或正在完成 `RUNNING` 回传，调度侧不得把该任务重新下发或回队。

# 10. 典型业务流程对应接口设计

------

## 10.1 用户创建并提交任务流程

1. 登录
   `POST /api/auth/login`
2. 获取求解器列表
   `GET /api/solvers`
3. 根据求解器获取任务模板
   `GET /api/solvers/{solverId}/task-options`
4. 获取上传规范
   `GET /api/profiles/{profileId}/upload-spec`
5. 创建任务
   `POST /api/tasks`
6. 编辑任务参数或名称
   `PUT /api/tasks/{taskId}`
7. 上传输入文件
   `POST /api/tasks/{taskId}/files`
8. 校验任务
   `POST /api/tasks/{taskId}/validate`
9. 提交任务入队
   `POST /api/tasks/{taskId}/submit`

------

## 10.2 调度执行流程

1. scheduler-service 查询待调度任务
   `GET /internal/tasks/queued`
2. 查询满足条件的节点
   `GET /internal/nodes/available?solverId=1`
3. scheduler-service 先校验全局求解器启用、模板启用、模板归属求解器与任务类型匹配，再根据优先级、提交时间和节点负载选出目标节点
4. scheduler-service 预占目标节点容量
   `POST /internal/nodes/{nodeId}/reserve`
5. task-service 原子标记任务为 `SCHEDULED`
   `POST /internal/tasks/{taskId}/mark-scheduled`
6. scheduler-service 调用 node-agent 下发任务
   `POST /internal/dispatch-task`
7. task-service 标记任务为 `DISPATCHED`
   `POST /internal/tasks/{taskId}/mark-dispatched`
8. 记录调度日志
   `POST /internal/schedules`

若第 5 步失败：

- scheduler-service 释放节点预占
  `POST /internal/nodes/{nodeId}/release-reservation`
- scheduler-service 继续处理下一条任务或重新选择任务

若第 6 步失败：

- task-service 根据失败是否可恢复，回退为 `QUEUED` 或标记为 `FAILED`
  `POST /internal/tasks/{taskId}/dispatch-failed`
- scheduler-service 释放节点预占
  `POST /internal/nodes/{nodeId}/release-reservation`
- scheduler-service 记录调度失败日志
  `POST /internal/schedules`

### 调度排序说明

- 先按 `priority` 从高到低排序
- 优先级相同再按 `submitTime` 从早到晚排序
- 再结合节点最小负载策略选择最终执行节点

------

## 10.3 节点执行回传流程

1. node-agent 标记开始运行
   `POST /internal/tasks/{taskId}/status-report`
2. task-service 在状态成功转为 `RUNNING` 后释放节点预占
   `POST /internal/nodes/{nodeId}/release-reservation`
3. 周期上报日志
   `POST /internal/tasks/{taskId}/log-report`
4. 上报结果摘要
   `POST /internal/tasks/{taskId}/result-summary-report`
5. 上报结果文件
   `POST /internal/tasks/{taskId}/result-file-report`
6. 最终标记完成
   `POST /internal/tasks/{taskId}/mark-finished`

补充约束：

- 节点在上报终态前应已成功上报 `RUNNING`；
- 若任务仍停留在 `SCHEDULED` 或 `DISPATCHED`，应优先补齐运行态或按调度失败语义回退，不应长期依赖跨级自动补跳。

若失败：

- `POST /internal/tasks/{taskId}/mark-failed`

若属于调度下发失败而尚未进入节点执行：

- `POST {scheduler-service}/internal/tasks/{taskId}/dispatch-failed`

# 11. 接口与微服务思想的匹配说明

------

## 11.1 服务职责明确

- 用户接口由 user-service 负责
- 求解器与模板接口由 solver-service 负责
- 任务全生命周期接口由 task-service 负责
- 节点与调度接口由 scheduler-service 负责

## 11.2 数据和接口边界一致

数据库是按服务拆分的，接口也是按服务边界设计的，避免“接口归 task，数据却归 scheduler”这种混乱。

## 11.3 通过内部接口协作，而不是共享数据库

例如：

- scheduler-service 不直接操作任务表
- 它通过 task-service 内部接口更新任务状态
- task-service 不直接操作节点表
- 它通过 scheduler-service 查询节点信息

这符合微服务设计原则。

------

# 12. 接口与分布式思想的匹配说明

------

## 12.1 节点独立注册

每个 node-agent 都有独立注册和心跳接口。

## 12.2 调度与执行解耦

- 调度服务负责选节点
- 节点代理负责执行
- 两者通过派发接口和状态回传接口交互

## 12.3 节点异构能力可表达

节点注册时上报本机实际支持的求解器及版本，管理员可额外控制节点级和节点-求解器级是否允许参与调度；调度时按 solverId 从可用节点中选择。

## 12.4 容错机制清晰

- 心跳丢失可判断离线
- 状态回传失败可重试

### 内部接口：重算节点预占汇总

**POST** `/internal/nodes/{nodeId}/reconcile-reservation`

用于容量异常补偿或运维审计。scheduler-service 以 `node_reservation` 中 `status=RESERVED` 的明细数量作为真值，修正 `compute_node.reserved_count`。该接口不修改 `node_reservation` 明细、不修改任务状态，仅返回 `nodeId / beforeReservedCount / actualReservedCount / afterReservedCount / changed`。

### 内部接口：审计节点预占明细

**POST** `/internal/nodes/{nodeId}/audit-reservation`

用于排查任务状态与预占明细的交叉一致性。scheduler-service 只读取当前节点 `node_reservation.status=RESERVED` 明细，并通过 task-service 查询任务基础状态；仅当任务仍处于 `SCHEDULED / DISPATCHED` 且任务主表 `nodeId` 与预占节点一致时，视为合法当前预占。

返回字段：

- `nodeId`
- `reservedCount`：节点汇总预占数；
- `reservedDetailCount`：当前 `RESERVED` 明细数；
- `consistent`：汇总计数一致且无交叉审计问题时为 `true`；
- `issues[]`：问题明细，字段包括 `reservationId / taskId / reservationNodeId / reservationStatus / taskStatus / taskNodeId / issueType / message / updatedAt`。

`issueType` 首版包括：

- `TASK_NOT_FOUND`：预占明细指向的任务不存在；
- `TASK_STATUS_NOT_RESERVABLE`：任务已不处于 `SCHEDULED / DISPATCHED`；
- `TASK_NODE_MISMATCH`：任务当前绑定节点与预占节点不一致。

该接口是只读审计能力，不修改 `node_reservation`、`compute_node.reserved_count` 或任务状态；实际修复应通过正式释放、离线补偿、管理员重试或 `reconcile-reservation` 等明确动作完成。
