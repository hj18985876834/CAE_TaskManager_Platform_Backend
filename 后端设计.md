## 5. 状态流转控制（非常关键）

必须集中管理：

```text
CREATED → VALIDATED → QUEUED → SCHEDULED → DISPATCHED → RUNNING → SUCCESS
                       ├──────→ QUEUED           （调度失败但可恢复时回队）
                       └──────→ FAILED           （调度失败且不可恢复）
                                          
DISPATCHED → QUEUED                            （节点已接收但允许重新入队）
DISPATCHED → FAILED                        （节点拒收或下发后立即失败）
RUNNING → FAILED / TIMEOUT
FAILED / TIMEOUT → QUEUED                  （管理员手动重试）
```

补充规则：

- 重新上传文件或修改参数后：`VALIDATED → CREATED`
- 校验失败时保持 `CREATED`，只返回问题明细，不新建任务
- 用户在 `CREATED`、`VALIDATED` 阶段取消创建时，应执行“放弃未提交任务（discard）”，不进入 `CANCELED`
- 对长时间停留在 `CREATED`、`VALIDATED` 且未提交的任务，后台应提供“未提交任务定时清理”，做逻辑删除和工作目录回收
- 若调度服务已将任务标记为 `SCHEDULED`，但下发到 node-agent 失败，应区分“可恢复失败”和“不可恢复失败”：
  - 可恢复失败优先回写为 `QUEUED`
  - 不可恢复失败再回写为 `FAILED`
- 对于节点已接收但尚未真正进入 `RUNNING` 的场景，允许 `DISPATCHED → QUEUED` 或 `DISPATCHED → FAILED`，用于表达执行前准备失败和拒收等真实场景
- 首版执行结果类终态固定从 `RUNNING` 进入；`SCHEDULED / DISPATCHED -> FAILED` 仅用于调度或派发阶段的不可恢复失败，不应通过接口内部补跳状态把它伪装成运行结果
- 首版不实现 `RUNNING -> CANCELED`；运行中终止与 node-agent `/internal/cancel-task` 一并归为第二阶段扩展
- 首版支持管理员对 `FAILED`、`TIMEOUT` 任务执行手动重试，使其重新回到 `QUEUED`
- 手动重试复用原任务记录，但应清理 `nodeId / failType / failMessage / startTime / endTime` 等上一轮运行态字段，并刷新 `submitTime` 为本次重新入队时间
- 自动重试策略仍作为后续增强项，不在首版强制实现

禁止乱跳状态！

### （3）调度流程

```
1. 查询 QUEUED 任务
2. 查询可用节点
3. 按 priority 排序后选择任务
4. 按能力、启用状态和 `runningCount + reservedCount` 选择目标节点
5. 对目标节点执行容量预占
6. 原子确认任务仍处于 QUEUED
7. 抢占成功后写入 nodeId 并标记为 SCHEDULED
8. 调用 node-agent 下发任务
9. 根据派发结果将任务更新为 DISPATCHED / QUEUED / FAILED
10. 写 schedule_record
```

补充约定：

- 节点预占与任务确认之间允许出现并发竞争，因此若任务确认失败，必须通过 `releaseReservation(...)` 释放刚才的节点预占；
- 节点预占必须以 `node_reservation(taskId, nodeId)` 为幂等依据，`reservedCount` 只是 `RESERVED` 状态预占的汇总值；
- `node_reservation` 与 `compute_node.reservedCount` 必须由 scheduler-service 在同一本地事务中维护；
- scheduler-service 提供内部 `reconcile-reservation` 能力，按 `node_reservation.status=RESERVED` 重算单节点 `reservedCount`，用于释放失败、历史异常或人工运维后的容量汇总修复；该能力只修复汇总字段，不修改预占明细和任务状态；
- scheduler-service 提供内部 `audit-reservation` 只读审计能力，用于检查当前 `RESERVED` 明细是否仍对应 `SCHEDULED / DISPATCHED` 且绑定同一节点的任务；该能力只返回问题列表，不自动修改预占明细、汇总计数或任务状态；
- 若调度在“选中节点之前”失败，`schedule_record.nodeId` 允许为空；
- 若已经选中节点但在派发或确认阶段失败，`schedule_record.nodeId` 应保留实际目标节点，便于审计问题发生在哪个节点；
- 调度失败记录的 `scheduleMessage` 必须使用可直接审计的短说明，并区分 `no online node`、`no enabled node solver capability`、`no dispatchable node with capacity`、`candidate nodes are full or reservation conflicted` 等原因；写入前应按 `schedule_record.schedule_message` 列宽截断，避免内部异常长消息导致调度记录落库失败。

------

### （4）调度策略（简单版）

```
优先级优先 + FCFS + 最小负载
```

即：

- 先按优先级高低排序
- 同优先级下先到先服务
- 选 `runningCount + reservedCount` 最小的节点
- 其中 `runningCount` 表达节点通过心跳上报的真实运行数，`reservedCount` 表达调度预占数
- 调度可用条件统一表达为：`runningCount + reservedCount < maxConcurrency`
- 代码中的策略类名可为 `FcfsLeastLoadStrategy`
- 但接口参数、调度记录表 `strategyName / strategy_name` 以及对外文档统一使用稳定标识 `FCFS_MIN_LOAD`

## 3. Node心跳机制

```
node-agent → /heartbeat
scheduler-service 更新：
- lastHeartbeatTime
- cpu/memory
- runningCount
```

补充说明：

- 心跳只负责上报节点真实运行态与负载信息；
- 调度预占不应再复用 `runningCount`，而应由 `reservedCount` 独立表达；
- `reservedCount` 不由心跳覆盖，也不由 task-service 直接写库；预占明细和幂等状态统一落在 scheduler-service 的 `node_reservation`；
- 节点进入 `RUNNING` 后，由心跳逐步反映真实运行数；`reservedCount` 应由 task-service 在成功处理 `DISPATCHED -> RUNNING` 后调用 scheduler-service 显式释放。

## 4. 节点离线判断

```text
当前时间 - lastHeartbeatTime > 30s → OFFLINE
```

这里的 `OFFLINE` 只表示节点运行态离线；管理员是否允许该节点参与调度，应使用独立的 `enabled` 字段控制，而不应混入运行态状态机。

补充约定：

- scheduler-service 发现节点离线后，应调用 task-service 的 `POST /internal/tasks/node-offline/fail` 统一做任务补偿；
- 节点离线补偿在首版固定收口为：
  - `SCHEDULED / DISPATCHED -> QUEUED`
  - `RUNNING -> FAILED`，并写入 `failType = NODE_OFFLINE`
- 对回退到 `QUEUED` 的任务，`NODE_OFFLINE` 只进入状态历史或调度记录，不写回任务主记录 `failType`；
- 节点离线补偿对每条实际完成状态变更的任务，都应尽量补写一条 `schedule_record` 失败审计，保留 `taskId / nodeId / scheduleStatus=FAILED / scheduleMessage=node offline ...`，便于管理员从调度时间线追溯本轮调度为何收口；
- 对回退到 `QUEUED` 的任务，task-service 在状态更新成功后应调用 scheduler-service 的 `releaseReservation(...)` 释放对应预占；
- 离线补偿时任务状态回队优先，预占释放失败不得回滚已经完成的状态收口，应记录告警并交由后续补偿或管理员审计处理；
- 预占释放失败应尽量写入 `schedule_record` 失败记录，保留 `taskId / nodeId / scheduleStatus=FAILED / scheduleMessage`，便于管理员从任务调度历史定位容量异常；
- `POST /internal/tasks/node-offline/fail` 的返回值应表示本次实际发生状态变更的任务数量，便于 scheduler-service 记录离线补偿审计；
- 离线检查定时任务按节点逐个处理补偿，单个节点补偿失败不得中断本轮其他节点处理；失败节点保留 `OFFLINE` 状态，并由下一轮定时检查继续重试补偿。

## 3. 执行流程

```
1. 校验节点本地并发是否允许接受
2. 准备任务目录并复用已校验 `unpackDir`
3. 基于 `unpackDir` 准备运行上下文，首版不得把重新解压 ZIP 作为正常执行路径
4. 回传任务已接收并进入运行前准备阶段
5. 回传 `RUNNING`
6. 构建命令并执行（ProcessBuilder）
7. 实时读取日志
8. 解析结果
9. 回传结果摘要、结果文件与最终状态
```

补充说明：

- 节点代理不应再把“解压 ZIP”当作固定步骤，校验阶段已经确认的 `unpackDir` 应作为首选执行目录；
- 当调度请求携带 `unpackDir` 时，node-agent 必须在创建运行目录前先校验映射后的目录真实存在且为目录，不能通过创建空 `workdir` 让目录检查误通过；
- task-service 向 scheduler-service 暴露 queued 任务时，必须确认输入归档已经完成校验并具备 `unpackDir`，否则不得进入正常派发链路；
- 节点回传 `RUNNING` 后，不直接修改节点容量，而是由 task-service 通过 scheduler-service 内部接口释放该任务对应的预占；
- 若派发后节点发现暂时不可执行，应优先通过调度失败或回队机制处理，而不是直接跳过运行态进入终态；
- node-agent 在执行前准备阶段失败且尚未回传 `RUNNING` 时，可调用 scheduler-service 的 `dispatch-failed` 入口，由 scheduler-service 统一调用 task-service 回写派发失败并释放对应预占；若直接调用 task-service 内部回写接口，task-service 只负责状态校验与状态落库，不直接释放预占；
- node-agent 必须在 `reportRunning(...)` 之前完成 `selectExecutor(...)` 与命令模板预检；若执行器不存在、命令模板无法解析、模板变量包含非法 shell 控制等，均属于“尚未真正运行”的执行前失败，应走 `dispatch-failed` 链路，而不能先把任务写成 `RUNNING` 再记为 `RUNTIME_ERROR`；
- scheduler-service 处理 `mark-scheduled` 返回 `claimed=false` 时必须区分当前状态：同节点仍为 `SCHEDULED` 才能继续下发；同节点已 `DISPATCHED` 说明已有在途派发，应停止本轮且不释放该在途预占；已 `RUNNING` 或终态不得重复通知 node-agent，可释放预占作为兜底。
- 当 `mark-scheduled` 的网络重试全部失败且没有拿到结构化 ACK 时，scheduler-service 不应立即释放预占；应先查询任务基础状态探测当前 `status/nodeId`，再按 `claimed=false` 的同一规则决定继续下发、停止本轮或释放预占，避免中心已成功落库 `SCHEDULED` 但 ACK 丢失时把任务和预占拆散。
- 对于上述 `claimed=false` 且未继续下发的场景，若原因是任务已进入 `RUNNING`、终态，或已被其他节点/其他调度实例抢占，scheduler-service 在完成本轮预占释放后应尽量补写一条 `schedule_record(FAILED)` 短说明，标记“本轮调度抢占已被拒绝”；但同一 `taskId + nodeId` 已为 `DISPATCHED` 的在途幂等场景只停止本轮，不额外补写失败调度记录，避免把迟到重试误记为真实失败。
- 节点已 accepted 但 `mark-dispatched` 确认失败时，scheduler-service 只能记录确认失败，不得主动回写 `dispatch-failed` 使任务回队，避免中心状态与节点实际执行分裂；该调度失败记录应明确写成“节点已接收但中心 `mark-dispatched` 确认失败，等待节点 `RUNNING` 或执行前失败回调补偿”，不能和节点拒收、派发失败混用同一笼统说明；
- node-agent 的 `status-report(RUNNING)` 在求解器启动前重试耗尽后，可先通过 scheduler-service 的 `dispatch-failed(recoverable=true)` 回队；若回队因中心任务已进入 `RUNNING` 等状态冲突被拒绝，应视为 `RUNNING` 回传可能已落库但 ACK 丢失，改用 `mark-failed(RUNTIME_ERROR)` 收口，避免中心长期停留 `RUNNING` 而节点本地并未执行求解器；
- 对上述“迟到 `dispatch-failed` 被 `RUNNING` 拒绝”的场景，task-service 虽不改变任务状态，但应补一条 same-status 的节点状态历史说明，标记“迟到 dispatch-failed 已被拒绝”，便于排查 `RUNNING` ACK 丢失与执行前失败兜底之间的竞态；
- scheduler-service 在收到这类 `dispatch-failed` 冲突后，也应尽量补写一条 `schedule_record(FAILED)` 短说明 `dispatch-failed rejected, task already running`，用于把“节点执行前失败兜底已被中心拒绝”明确落到调度时间线；该审计失败不得改变 node-agent 后续改走 `mark-failed(RUNTIME_ERROR)` 的闭环；
- scheduler-service 对 node-agent `dispatch-task` 的业务拒收应统一归并为 `NODE_AGENT_REJECTED` 语义：无论 node-agent 通过 `accepted=false` 还是直接返回 `409/CONFLICT`（如 `node is busy`）表达拒收，调度侧都应收口为正式短说明 `node-agent rejected dispatch request`，并继续按可恢复派发失败处理，避免把节点内部实现文案直接写入 `dispatch-failed` 或 `schedule_record`；
- scheduler-service 调用 node-agent `/internal/dispatch-task` 时，对网络异常、空响应或网关类失败应先执行有限重试；每次重发前都应先通过 node-agent 只读 runtime 探测和 task-service 任务基础状态复核确认是否仍可重试。若节点已 active，或中心已推进为同节点 `DISPATCHED/RUNNING`，应按“节点已 accepted”继续执行 `mark-dispatched` 确认，不得再次重发；若中心已不再是同节点 `SCHEDULED`，应停止重发并进入既有派发失败收口，避免执行前失败已回队/失败后又被旧派发请求重新接收。
- 如果上述重试全部耗尽，scheduler-service 仍应在进入 `dispatch-failed` 前调用 node-agent 只读 runtime 探测接口确认该任务是否已被节点接收；若节点返回 active，则按“节点已 accepted”继续执行 `mark-dispatched` 确认，不得主动回队。
- 若求解器已成功结束，但结果摘要、结果文件或 `mark-finished` 回调持续失败，node-agent 应使用 `mark-failed(CALLBACK_ERROR)` 收口，而不是继续记为普通 `RUNTIME_ERROR`；task-service 在该终态成功落库后应尽力清理已写入的结果摘要与结果文件元数据，避免 `FAILED` 任务继续展示成功结果；若清理失败，只允许记录告警，不得回滚已完成的终态；
- 若节点在结果回传阶段被离线补偿收口为 `FAILED(NODE_OFFLINE)`，而此前已经写入了部分结果摘要或结果文件元数据，task-service 也应尽力清理这些成功结果元数据；清理失败只记录告警，不得回滚终态；
- 任一 `RUNNING -> SUCCESS / TIMEOUT / FAILED` 以及节点离线补偿触发的 `RUNNING -> FAILED(NODE_OFFLINE)` 在状态收口成功后，都应尽力再次释放该任务的残留预占；该动作只用于修复 `DISPATCHED -> RUNNING` 时历史释放失败遗留的 `node_reservation=RESERVED`，释放失败只能记录审计，不得回滚终态。
- 若中心已先进入任一终态，后续迟到的 `mark-finished / mark-failed` 仍按幂等返回当前终态，但 task-service 应补一条 same-status 的节点状态历史说明，区分“真实终态提交”与“ACK 丢失后的迟到确认/重复回调”。
- 结果摘要与结果文件只允许在 `RUNNING / SUCCESS / TIMEOUT` 阶段写入；若中心已先进入 `FAILED / CANCELED` 等不允许结果写入的终态，迟到 `result-summary-report / result-file-report` 必须返回冲突，并补一条 same-status 的节点状态历史说明，标记“迟到结果回调已被拒绝”。
- node-agent 的 `/internal/dispatch-task` 必须对同一 `taskId` 重复下发保持幂等：若该任务已在本节点运行，应返回 accepted 成功但不得启动重复执行线程；节点容量不足仍按失败返回。该幂等语义用于支撑 scheduler-service 在传输失败或 ACK 丢失时安全重试下发。
- node-agent 应提供内部只读 runtime 探测能力，返回指定 `taskId` 是否仍在本节点 active，以及是否已经上报过 `RUNNING`；该接口只用于调度侧处理 dispatch ACK 歧义，不改变节点任务状态。

@Component
public class JwtFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");

        if (token != null) {
            Long userId = JwtUtil.parseToken(token);
            exchange.getRequest().mutate()
                    .header("X-User-Id", String.valueOf(userId))
                    .build();
        }

        return chain.filter(exchange);
    }
}

# 一、总体设计原则

这套后端建议统一采用：

> **按微服务拆分 + 服务内采用分层/模块化结构**

每个微服务内部统一分成 5 层：

1. **interfaces 层**：对外暴露接口
   放 Controller、Request/Response VO
2. **application 层**：应用编排层
   放 UseCase、AppService、Facade，负责流程编排
3. **domain 层**：领域层
   放 Entity、Enum、DomainService、Repository 接口、状态流转规则
4. **infrastructure 层**：基础设施层
   放 Mapper、RepositoryImpl、FeignClient、文件存储、第三方调用
5. **shared/config 层**：共享配置层
   放配置类、常量、本服务内工具类

这样做的好处是：

- Controller 不会塞业务
- Service 不会无限膨胀
- Mapper 不会直接污染业务逻辑
- 后面好写单元测试
- 论文里也更好解释“分层解耦”

# 三、common-lib 设计

`common-lib` 只放**真正跨服务通用**的内容，不能变成“大杂烩”。

## 1. 推荐结构

```text
common-lib/
└── src/main/java/com/yourorg/common/
    ├── response/
    ├── exception/
    ├── enums/
    ├── constants/
    ├── dto/
    ├── utils/
    ├── model/
    └── config/
```

## 2. 各目录职责

### `response/`

放统一返回结构：

- `Result<T>`
- `PageResult<T>`

### `exception/`

放公共异常：

- `BizException`
- `UnauthorizedException`
- `ForbiddenException`
- `GlobalExceptionCode`

### `enums/`

只放所有服务都可能共用的枚举：

- `TaskStatusEnum`
- `FailTypeEnum`
- `NodeStatusEnum`
- `RoleCodeEnum`

### `constants/`

放公共常量：

- Header 常量
- Token 前缀常量
- 默认分页常量

### `dto/`

只放跨服务传输对象：

- `TaskDTO`
- `SolverProfileDTO`
- `FileRuleDTO`

### `utils/`

放纯工具类：

- `JwtUtil`
- `DateTimeUtil`
- `JsonUtil`
- `IdUtil`

### `model/`

放特别通用的基础模型：

- `BasePageQuery`
- `BaseUserContext`

### `config/`

尽量少放，最多放一些公共 Jackson / 时间格式配置。

## 3. 不建议放进 common-lib 的内容

不要放：

- 各服务自己的 entity
- 各服务自己的 request/response vo
- 各服务 mapper
- 业务过强的 service

原则是：**common-lib 越小越好**。

## 3. 这个服务的关键规范

建议明确：

- **模板是核心聚合根**
- 文件规则围绕模板管理
- 求解器定义与模板是一对多
- 模板新增或编辑时必须校验 `commandTemplate` 的轻量命令约束、禁止 `${workDir}`，并校验未知占位符；任务参数占位符只能来自 `paramsSchema/paramsSchemaJson` 的 JSON Schema `properties`。
- 对外模板合同与 node-agent 的命令解析合同必须保持一致，避免保存成功但执行阶段才发现 `${taskdir}`、`${case}` 等未声明占位符。
- `ProfileTemplateContractValidator` 同时负责 `paramsSchema/paramsSchemaJson` 的轻量 JSON Schema 合同校验：非空时必须是 JSON object，首版校验 `properties/required/items/additionalProperties/pattern` 等 task-service 已支持字段，避免模板配置错误后移到普通用户任务流程。
- 文件规则 `ruleJson` 的受控派生字段应在 solver-service 保存阶段完成基本合法性校验，包括 `deriveParam.pattern`、`sanitizeRegex`、`preprocess.pattern` 等正则字段，避免运行到任务校验阶段才暴露模板配置错误；`deriveParam` 与 `deriveParams` 同一条规则内互斥，派生项首版仅支持 `name/source/value/pattern/group/required/trim/stripQuotes/sanitizeRegex/preprocess`，`preprocess` 子项仅支持 `pattern/replacement`。
- 文件规则 `allowSuffix` 在 solver-service 上传规范输出、task-service 上传校验和解压后文件校验中应统一归一化为去除前导点的小写后缀，兼容管理员填写 `zip` 或 `.zip`。
- 文件规则作用域选择器必须采用白名单合同，首版仅支持 `solverId/solverIds`、`profileId/profileIds`、`taskType/taskTypes`、`solverCode/solverCodes`、`solverExecMode/solverExecModes`、`solverExecPath/solverExecPaths`；solver-service 保存规则时拒绝未知选择器，task-service 校验存量规则时不能静默忽略未知选择器。
- 文件规则路径合同必须与校验执行目录一致：`pathPattern` 只能表示相对已校验 `unpackDir/caseDir` 的路径 glob，统一使用 `/`，支持 `*`、`?`、`**`，不得使用绝对路径或 `..`；`fileNamePattern` 只能表示单个文件名 glob，不得夹带路径。solver-service 保存规则时前置拒绝非法路径，task-service 对存量非法路径按 `TEMPLATE_RULE_CONFLICT` 兜底处理。
- 文件规则 `fileType` 必须采用枚举合同，首版仅支持 `ZIP`、`FILE`、`DIR`、`DIRECTORY`；solver-service 保存规则时拒绝非法值，task-service 校验存量规则时按 `TEMPLATE_RULE_CONFLICT` 兜底处理。
- 文件规则数量合同必须避免自相矛盾：`requiredFlag=1` 时 `maxCount/max` 不能为 0；task-service 对存量冲突规则按 `TEMPLATE_RULE_CONFLICT` 处理，避免把模板配置错误表现为用户输入文件错误。
- 文件规则元数据合同应保存期与任务校验期一致：`maxSizeMb` 必须为正整数，`minCount/min/maxCount/max/group` 必须为非负整数；task-service 对历史非法 `ruleJson` 不得静默降级或跳过，应按 `TEMPLATE_RULE_CONFLICT` 兜底处理。

这样你在代码里就会更清楚：

- 修改模板时要不要级联规则
- 删除模板时如何校验依赖
- 获取上传规范时从哪里组装数据

# 七、task-service 项目结构设计

这是全系统最核心、也最容易写乱的服务，必须重点设计。

## 1. 推荐结构

```text
task-service/
└── src/main/java/com/yourorg/task/
    ├── TaskApplication.java
    ├── interfaces/
    │   ├── controller/
    │   ├── internal/
    │   ├── request/
    │   └── response/
    ├── application/
    │   ├── service/
    │   ├── facade/
    │   ├── manager/
    │   ├── assembler/
    │   └── executor/
    ├── domain/
    │   ├── model/
    │   ├── repository/
    │   ├── service/
    │   ├── enums/
    │   └── rule/
    ├── infrastructure/
    │   ├── persistence/
    │   │   ├── entity/
    │   │   ├── mapper/
    │   │   └── repository/
    │   ├── client/
    │   ├── storage/
    │   └── support/
    ├── config/
    └── support/
```

## 2. 为什么要比别的服务更细

因为它同时承担：

- 任务创建
- 文件上传
- 文件校验
- 状态流转
- 日志处理
- 结果处理
- 内部回调
- 多服务协同

所以一定要把“流程编排”和“领域规则”拆开。

## 3. 重点目录说明

### interfaces/controller

对外接口：

- `TaskController`
- `TaskQueryController`
- `TaskLogController`
- `TaskResultController`

### interfaces/internal

对内接口：

- `InternalTaskDispatchController`
- `InternalTaskReportController`

建议把内部接口和外部接口分目录，避免混淆。

### interfaces/request

- `CreateTaskRequest`
- `UpdateTaskRequest`
- `ValidateTaskRequest`
- `SubmitTaskRequest`
- `DiscardTaskRequest`
- `CancelTaskRequest`
- `AdjustTaskPriorityRequest`
- `RetryTaskRequest`
- `TaskListQueryRequest`

### interfaces/response

- `TaskDetailResponse`
- `TaskListItemResponse`
- `TaskStatusHistoryResponse`
- `TaskLogResponse`
- `TaskResultSummaryResponse`

其中 `TaskListItemResponse`、`TaskDetailResponse` 建议补充派生字段：

- `canCancel`
- `canRetry`

### application/manager

这里是 task-service 的核心，建议至少拆成：

- `TaskLifecycleManager`
- `TaskValidationManager`
- `TaskDispatchManager`
- `TaskResultManager`

分别负责：

- 生命周期流转
- 校验编排
- 调度相关编排
- 结果与日志收尾

### application/service

放更偏应用层的服务：

- `TaskCommandAppService`
- `TaskQueryAppService`
- `TaskLogAppService`
- `TaskResultAppService`

补充要求：
- 查询层除基础展示字段外，应统一派生交互字段与解释字段，如 `canCancel`、`canRetry`、`queueReason`
- `queueReason` 必须按查询时动态计算，不单独写入任务主表
- `queueReason` 属于面向用户的解释字段，不作为调度审计字段；审计仍以状态历史和调度记录为准

### application/executor

这里不是 node-agent 的执行器，而是 task-service 内部的“命令执行编排器”，例如：

- `TaskCreateExecutor`
- `TaskValidateExecutor`
- `TaskSubmitExecutor`

如果你想让流程更清晰，可以用这种方式；如果不想太复杂，也可以只保留 manager。

### domain/model

建议按业务实体拆：

- `Task`
- `TaskFile`
- `TaskStatusHistory`
- `TaskResultSummary`
- `TaskResultFile`
- `TaskLogChunk`

### domain/rule

这里很重要，建议专门放业务规则：

- `TaskStatusRule`
- `TaskValidationRule`
- `TaskCancelRule`

### domain/service

- `TaskDomainService`
- `TaskStatusDomainService`
- `TaskValidationDomainService`

### infrastructure/client

- `SolverClient`
- `SchedulerClient`

### infrastructure/storage

- `TaskFileStorageService`
- `LocalTaskFileStorageService`

### infrastructure/support

- `TaskNoGenerator`
- `TaskPathResolver`
- `LogChunkAppender`

## 4. 强烈建议的聚合划分

task-service 里建议按三个聚合理解：

### 聚合 1：Task 主聚合

包含：

- Task
- TaskStatusHistory

### 聚合 2：Task 文件聚合

包含：

- TaskFile

### 聚合 3：Task 结果聚合

包含：

- TaskResultSummary
- TaskResultFile
- TaskLogChunk

这样你写代码时会很清楚：

- 创建任务更新哪些对象
- 校验文件更新哪些对象
- 运行结束更新哪些对象

## 5. task-service 最推荐的包命名例子

```text
task-service
├── interfaces/controller/TaskController.java
├── interfaces/internal/InternalTaskReportController.java
├── application/manager/TaskLifecycleManager.java
├── application/service/TaskCommandAppService.java
├── domain/model/Task.java
├── domain/rule/TaskStatusRule.java
├── infrastructure/client/SolverClient.java
├── infrastructure/storage/LocalTaskFileStorageService.java
└── infrastructure/persistence/repository/TaskRepositoryImpl.java
```

这个结构很清晰，后面维护起来也不容易乱。

### 关键请求对象字段约定

为避免实现时和接口设计脱节，以下请求对象字段应与接口文档保持一致：

* `StatusReportRequest`：`nodeId / fromStatus / toStatus / changeReason / operatorType`
* `FinishTaskRequest`：`nodeId / finalStatus`
* `FailTaskRequest`：`nodeId / failType / failMessage`
* `DispatchFailedRequest`：`nodeId / failType / reason / recoverable`
* `NodeOfflineTasksRequest`：`nodeId / reason`
* `NodeReservationRequest`：`taskId`

其中 `StatusReportRequest` 第一版只用于 `DISPATCHED -> RUNNING`，终态仍通过 `FinishTaskRequest` 或 `FailTaskRequest` 处理；`NodeOfflineTasksRequest` 仅供 scheduler-service 在节点离线补偿场景调用，返回值应表示本次实际发生状态变更的任务数量。

### `infrastructure/support/CommandTemplateResolver.java`

职责：命令模板解析

建议方法：

* `resolve(String commandTemplate, Map<String, Object> params)`

设计约束：

* 该组件只负责轻量占位符替换，不应演变为复杂脚本解释器；
* 支持的常规占位符包括 `solverExecPath`、`taskDir`、`inputDir`、`outputDir`、`logDir` 和任务参数；
* 任务参数占位符必须由模板的 `paramsSchema/paramsSchemaJson.properties` 明确定义；solver-service 保存模板时应前置拒绝未知占位符，node-agent 执行阶段仍保留未解析占位符兜底校验；
* solver-service 保存模板时还应校验 `paramsSchema/paramsSchemaJson` 本身的基本合法性，至少覆盖 task-service 当前实际支持的 JSON Schema 子集；
* `taskDir` 必须来自后端校验阶段确认的 `unpackDir`；首版不支持 `${workDir}` 作为独立占位符，避免与 `workdir/unpackDir/caseDir` 术语混淆；
* 如果渲染后仍存在未解析占位符，应直接报错，不应把未替换模板下发给执行器；
* 如果某个求解器需要复杂命令构建或多阶段准备，应由对应适配器处理，而不是继续增加模板配置复杂度。

* node-agent 递归收集 `outputDir` 下结果文件时，如果文件位于嵌套目录，应将相对目录信息编码进不含路径分隔符和控制字符的 `fileName`，避免同名嵌套文件按唯一约束互相覆盖；编码后长度不得超过 `task_result_file.file_name` 的 255 字符限制，超长时应采用稳定截断加短哈希的方式保持唯一性；
* task-service 只要求 `storagePath` 位于任务 `output/` 下且文件真实存在，不要求上报 `fileName` 必须等于路径末段；结果下载时也应按同一合同校验展示名安全性和物理路径边界，不能重新引入 `fileName == basename(storagePath)` 的旧假设。
* task-service 对结果摘要、结果文件的“允许阶段校验”和“元数据落库”必须在同一事务内基于 `sim_task` 行锁完成，避免与 `mark-finished / mark-failed / node-offline` 并发时把成功结果写入已转为 `FAILED / CANCELED` 的任务；

# 七、scheduler-service 初始化代码骨架清单

* `reserveCapacity(...)` 必须在同一本地事务中完成容量判断、`node_reservation=RESERVED` 写入或更新、`reservedCount + 1`
* `releaseReservation(...)` 必须在同一本地事务中将 `node_reservation` 从 `RESERVED` 更新为 `RELEASED`，且只有更新成功时才能执行 `reservedCount - 1`
* 如果释放时发现 `node_reservation` 已为 `RELEASED`，应幂等成功返回；如果不存在对应记录，应返回明确冲突，禁止静默修改 `reservedCount`
* `markOfflineNodes()` 收到 `markNodeOfflineTasksFailed(...)` 返回值后，应按“实际发生状态变更的任务数量”记录补偿审计，而不是按扫描到的任务总数记账
