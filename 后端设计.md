### （3）调度流程

```
1. 查询 QUEUED 任务
2. 查询可用节点
3. 按 priority 排序后选择任务
4. 选择节点（最简单：最小负载）
5. 调 node-agent
6. 更新任务状态
7. 写 schedule_record
```

------

### （4）调度策略（简单版）

```
优先级优先 + FCFS + 最小负载
```

即：

- 先按优先级高低排序
- 同优先级下先到先服务
- 选 runningCount 最小的节点

## 3. JWT过滤器（核心）

```java
@Component
public class JwtFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");

        if (token != null) {
            Long userId = JwtUtil.parseToken(token);
            exchange.getRequest().mutate()
                    .header("X-User-Id", String.valueOf(userId))
                    .build();
        }

        return chain.filter(exchange);
    }
}
```

# 一、总体设计原则

这套后端建议统一采用：

> **按微服务拆分 + 服务内采用分层/模块化结构**

每个微服务内部统一分成 5 层：

1. **interfaces 层**：对外暴露接口
   放 Controller、Request/Response VO
2. **application 层**：应用编排层
   放 UseCase、AppService、Facade，负责流程编排
3. **domain 层**：领域层
   放 Entity、Enum、DomainService、Repository 接口、状态流转规则
4. **infrastructure 层**：基础设施层
   放 Mapper、RepositoryImpl、FeignClient、文件存储、第三方调用
5. **shared/config 层**：共享配置层
   放配置类、常量、本服务内工具类

这样做的好处是：

- Controller 不会塞业务
- Service 不会无限膨胀
- Mapper 不会直接污染业务逻辑
- 后面好写单元测试
- 论文里也更好解释“分层解耦”

# 三、common-lib 设计

`common-lib` 只放**真正跨服务通用**的内容，不能变成“大杂烩”。

## 1. 推荐结构

```text
common-lib/
└── src/main/java/com/yourorg/common/
    ├── response/
    ├── exception/
    ├── enums/
    ├── constants/
    ├── dto/
    ├── utils/
    ├── model/
    └── config/
```

## 2. 各目录职责

### `response/`

放统一返回结构：

- `Result<T>`
- `PageResult<T>`

### `exception/`

放公共异常：

- `BizException`
- `UnauthorizedException`
- `ForbiddenException`
- `GlobalExceptionCode`

### `enums/`

只放所有服务都可能共用的枚举：

- `TaskStatusEnum`
- `FailTypeEnum`
- `NodeStatusEnum`
- `RoleCodeEnum`

### `constants/`

放公共常量：

- Header 常量
- Token 前缀常量
- 默认分页常量

### `dto/`

只放跨服务传输对象：

- `TaskDTO`
- `NodeDTO`
- `SolverProfileDTO`
- `FileRuleDTO`

### `utils/`

放纯工具类：

- `JwtUtil`
- `DateTimeUtil`
- `JsonUtil`
- `IdUtil`

### `model/`

放特别通用的基础模型：

- `BasePageQuery`
- `BaseUserContext`

### `config/`

尽量少放，最多放一些公共 Jackson / 时间格式配置。

## 3. 不建议放进 common-lib 的内容

不要放：

- 各服务自己的 entity
- 各服务自己的 request/response vo
- 各服务 mapper
- 业务过强的 service

原则是：**common-lib 越小越好**。

这里是 task-service 的核心，建议至少拆成：

分别负责：

- 生命周期流转

放更偏应用层的服务：

- `TaskCommandAppService`
- `TaskQueryAppService`
- `TaskLogAppService`
- `TaskResultAppService`

- `TaskCreateExecutor`
- `TaskValidateExecutor`

如果你想让流程更清晰，可以用这种方式；如果不想太复杂，也可以只保留 manager。

### domain/model

建议按业务实体拆：

- `Task`
- `TaskFile`
- `TaskStatusHistory`
- `TaskResultSummary`

- `TaskDomainService`

### infrastructure/storage

- `TaskFileStorageService`
- `LocalTaskFileStorageService`

### infrastructure/support

- `TaskNoGenerator`
- `TaskPathResolver`
- `LogChunkAppender`

## 4. 强烈建议的聚合划分

task-service 里建议按三个聚合理解：

### 聚合 1：Task 主聚合

包含：

- Task
- TaskStatusHistory

### 聚合 2：Task 文件聚合

包含：

- TaskFile

### 聚合 3：Task 结果聚合

包含：

- TaskResultSummary

这样你写代码时会很清楚：

- 创建任务更新哪些对象

# 二、统一包名前缀建议

建议统一使用类似下面的包名：

```text
com.example.cae
```

然后各服务分别是：

- `com.example.cae.common`
- `com.example.cae.gateway`
- `com.example.cae.user`
- `com.example.cae.solver`
- `com.example.cae.task`
- `com.example.cae.scheduler`
- `com.example.cae.nodeagent`

不要混用：

- `com.example`
- `com.xxx.project`
- `org.demo`
- `cn.test`

统一包名前缀会让整个工程非常整齐。

**创建任务 → 校验 → 提交 → 调度 → mock 执行 → 回传成功**

这正是你文档里强调的“核心必须完成闭环”。
