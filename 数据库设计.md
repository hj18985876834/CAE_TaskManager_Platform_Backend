下面我直接给出一套**可开发、不过度复杂、符合前面需求分析与系统设计**的方案。

# 一、先明确微服务数量

结合你前面已经确定的需求，我建议系统采用 **5 个核心微服务 + 1 个前端 + 1 个节点代理程序** 的结构。

## 1. 核心微服务

### （1）gateway-service

职责：

- 统一入口
- 路由转发
- JWT 鉴权
- 权限拦截

### （2）user-service

职责：

- 用户登录
- 用户管理
- 角色管理

### （3）task-service

职责：

- 仿真任务创建
- 文件信息管理
- 任务状态管理
- 任务日志与结果管理

### （4）solver-service

职责：

- 求解器定义管理
- 求解器支持的任务类型管理
- 输入文件模板管理
- 执行命令模板管理

### （5）scheduler-service

职责：

- 节点管理
- 节点心跳
- 调度策略执行
- 调度记录管理

------

## 2. 节点代理（不是中心微服务，但必须有）

### node-agent

职责：

- 部署在计算节点
- 向 scheduler-service 注册和发心跳
- 接收调度任务
- 调用真实求解器或模拟执行器
- 回传状态、日志、结果

它更像“边缘执行程序”，不是中心业务微服务，但在分布式架构里非常关键。

------

# 二、数据库设计原则

这部分最重要。你要求“符合微服务思想 / 分布式思想”，那数据库一定不能设计成一个大一统、所有服务都乱写的库。

我建议采用：

> **每个微服务独立数据库（或独立 schema），服务间通过接口通信，不直接跨库联表**

这是最符合微服务思想，也最适合你论文写法的。

------

## 1. 推荐数据库拆分

### user_db

给 `user-service`

### task_db

给 `task-service`

### solver_db

给 `solver-service`

### scheduler_db

给 `scheduler-service`

这样做的好处：

- 服务职责清晰
- 数据边界清晰
- 后续扩展方便
- 符合“去中心化数据管理”思想

------

## 2. 跨服务关联方式

不要做跨库外键。
统一采用：

- `user_id`
- `solver_id`
- `profile_id`
- `node_id`
- `task_id`

这些业务主键在服务间传递。

例如：

- task-service 里只存 `user_id`，不存完整用户表
- task-service 里只存 `solver_id`、`profile_id`，不直接维护求解器定义
- scheduler-service 里只存 `task_id`，不直接维护任务全部详情

这才符合微服务思想。

------

数据库名：`solver_db`

------

## 表 3：solver_definition

用途：求解器定义

| 字段名      | 类型         | 说明                                    |
| ----------- | ------------ | --------------------------------------- |
| id          | bigint       | 主键                                    |
| solver_code | varchar(50)  | 唯一编码，如 OPENFOAM / CALCULIX / MOCK |
| solver_name | varchar(100) | 显示名称                                |
| version     | varchar(50)  | 版本                                    |
| exec_mode   | varchar(30)  | 执行方式：LOCAL / CONTAINER             |
| exec_path   | varchar(255) | 可执行路径或镜像名                      |

### 说明

`GET /api/solvers` 的分页查询基于本表完成，不需要为分页展示新增额外字段。

## 表 4：solver_task_profile

用途：某个求解器支持某个任务类型时的模板定义

| 字段名           | 类型         | 说明                                    |
| ---------------- | ------------ | --------------------------------------- |
| id               | bigint       | 主键                                    |
| solver_id        | bigint       | 求解器ID                                |
| profile_code     | varchar(50)  | 模板编码                                |
| task_type        | varchar(50)  | 任务类型，如 STRUCT_STATIC / CFD_STEADY |
| profile_name     | varchar(100) | 模板名称                                |
| upload_mode      | varchar(30)  | 上传模式：ZIP_ONLY                      |
| command_template | varchar(255) | 轻量命令模板，主要描述主执行命令        |
| params_schema_json | text         | 参数模板定义JSON                        |
| parser_name      | varchar(100) | 结果解析器名称                          |
| timeout_seconds  | int          | 超时时间                                |
| enabled          | tinyint      | 是否启用                                |
| description      | varchar(255) | 说明                                    |
| created_at       | datetime     | 创建时间                                |
| updated_at       | datetime     | 更新时间                                |

### 说明

这是最关键的表。
它体现：

- 这个求解器支持什么任务类型
- 用什么命令执行
- 用什么解析器处理结果
- `GET /api/profiles` 的分页查询基于本表完成，可按 `solver_id/task_type/profile_code/enabled` 等字段筛选，不需要新增接口专用字段

补充约束：

- `command_template` 应主要保存主执行命令和少量占位符，例如 `${solverExecPath}`、`${taskDir}`、`${inputDir}`、`${outputDir}`、`${logDir}`；
- `${taskDir}` 统一表示节点侧最终执行目录，由任务校验阶段记录的 `task_file.unpack_dir` 映射得到；首版不使用 `${workDir}` 作为数据库模板占位符；
- 不建议在该字段中长期保存复杂 shell 流程，例如目录创建、结果复制、退出码包装、多段命令串联；
- 平台通用目录准备、结果归集、日志收集等流程应由节点代理或求解器适配器处理，避免模板配置过重。

## 表 5：solver_profile_file_rule

用途：模板所需文件规则

| 字段名            | 类型         | 说明                             |
| ----------------- | ------------ | -------------------------------- |
| id                | bigint       | 主键                             |
| profile_id        | bigint       | 模板ID                           |
| file_key          | varchar(50)  | 文件标识，如 main_inp / dir_system |
| path_pattern      | varchar(255) | 解压后相对路径规则，如 system/** |
| file_name_pattern | varchar(100) | 文件名或匹配规则                 |
| file_type         | varchar(30)  | ZIP / FILE / DIR / DIRECTORY    |
| required_flag     | tinyint      | 是否必需                         |
| sort_order        | int          | 排序                             |
| rule_json         | text         | 扩展约束JSON（数量/后缀/大小等简单约束） |
| description       | varchar(255) | 说明                             |

### 说明

这张表专门解决“不同求解器、不同任务类型需要哪些文件”的问题。

`rule_json` 使用边界：

- 推荐用于 `allowSuffix`、`minCount`、`maxCount`、`maxSizeMb` 等简单规则；
- 在确有必要时，也允许少量作用域选择器，如 `solverCodes`、`taskTypes`，以及受控的 `deriveParam/deriveParams` 用于参数派生；同一条规则内 `deriveParam` 与 `deriveParams` 互斥；
- 首版支持的作用域选择器限定为 `solverId/solverIds`、`profileId/profileIds`、`taskType/taskTypes`、`solverCode/solverCodes`、`solverExecMode/solverExecModes`、`solverExecPath/solverExecPaths`，未知选择器不得保存为有效规则；
- 不建议在首版中把大量复杂正则预处理、命令推导、结果收集流程写入 `rule_json`；如果使用 `deriveParam.pattern`、`sanitizeRegex`、`preprocess.pattern`，保存规则时必须校验正则合法性；派生项首版仅支持 `name/source/value/pattern/group/required/trim/stripQuotes/sanitizeRegex/preprocess`，`preprocess` 子项仅支持 `pattern/replacement`；
- 如果某类规则明显属于某个求解器的特有逻辑，应优先放入求解器适配器或后端校验组件。
- `path_pattern` 表示相对校验后 `unpack_dir/caseDir` 的路径 glob，统一使用 `/` 分隔，支持 `*`、`?`、`**`，不得保存绝对路径或包含 `..` 的路径；`file_name_pattern` 只表示文件名 glob，不得包含路径分隔符。
- `allowSuffix` 表示文件扩展名，允许保存为 `zip` 或 `.zip`，业务校验时统一归一化为去除前导点的小写后缀。
- `file_type` 首版限定为 `ZIP`、`FILE`、`DIR`、`DIRECTORY`，非法值不能保存为有效模板规则。
- `required_flag=1` 与 `maxCount/max=0` 属于自相矛盾的模板规则，不应保存；历史数据如存在该组合，应在任务校验时按模板规则冲突处理。
- `maxSizeMb` 必须为正整数，`minCount/min/maxCount/max/group` 必须为非负整数；历史非法 `rule_json` 应在任务校验阶段按模板规则冲突处理。

例如：

#### CalculiX 结构静力模板

- `model.inp`，必需，FILE

#### OpenFOAM 稳态流体模板

- 上传模式：ZIP_ONLY
- 解压后必须存在：
- `system/**`，DIR
- `constant/**`，DIR
- `0/**`，DIR

### 为什么单独拆这张表

因为它会变化，而且后续新增求解器时最容易扩展。

### 这个表必须满足的点

它要能支撑：

- 任务创建
- 任务参数编辑
- 任务优先级设置
- 状态展示
- 节点归属
- 失败原因
- 执行时间统计
- 按用户筛选
- 按求解器筛选
- 按任务类型筛选
- 未提交任务逻辑删除

### 优先级说明

- `priority` 数值越大，任务优先级越高
- 普通用户创建任务时默认值为 `0`
- 管理员可在任务创建后调整优先级
- 调度时优先级作为首要排序条件，之后再结合提交时间和节点负载进行分配

### 统一状态枚举

```
CREATED
VALIDATED
QUEUED
SCHEDULED
DISPATCHED
RUNNING
SUCCESS
FAILED
CANCELED
TIMEOUT
```

首版统一补充约束：

- `SCHEDULED → QUEUED`：用于表达“调度已选中节点，但派发失败且允许重新入队”；
- `SCHEDULED → FAILED`：用于表达“调度已选中节点，但下发失败”；
- `DISPATCHED → QUEUED`：用于表达“节点已接收任务，但尚未真正进入运行且允许重新入队”；
- `DISPATCHED → FAILED`：用于表达“节点接收后未进入运行即失败/拒收”；
- `FAILED → QUEUED`、`TIMEOUT → QUEUED`：用于管理员手动重试；
- 手动重试复用同一条 `sim_task` 记录，不额外新增“重试任务表”。

### 统一失败类型枚举

```
VALIDATION_ERROR
DISPATCH_ERROR
NODE_OFFLINE
EXECUTOR_START_ERROR
RUNTIME_ERROR
CALLBACK_ERROR
TIMEOUT
MANUAL_CANCEL
```

其中 `DISPATCH_ERROR` 用于 scheduler-service 已将任务置为 `SCHEDULED` 或 `DISPATCHED`，但派发过程失败的场景；是否最终回退到 `QUEUED` 还是进入 `FAILED`，由失败是否可恢复决定；`NODE_OFFLINE` 仅用于节点心跳超时后最终收口为 `FAILED` 的任务。若 `RUNNING` 任务在节点离线补偿前已经写入了部分结果摘要或结果文件元数据，task-service 在落库 `FAILED(NODE_OFFLINE)` 后也应尽力清理这些成功结果元数据；清理失败只允许记录告警，不得回滚已完成的终态。`CALLBACK_ERROR` 用于求解器已成功结束，但结果摘要、结果文件或 `mark-finished` 回调持续失败，node-agent 最终只能通过 `mark-failed` 收口的场景。若 `CALLBACK_ERROR` 成功落库，为避免 `FAILED` 任务继续展示成功结果，task-service 应尽力清理该任务已写入的结果摘要与结果文件元数据；若清理失败，只允许记录告警，不得回滚已完成的终态。若 `SCHEDULED`、`DISPATCHED` 任务因节点离线补偿回退到 `QUEUED`，则该原因只写入状态历史或调度记录，不写回 `sim_task.fail_type`。

## 表 7：task_status_history

用途：任务状态流转记录表

| 字段名        | 类型         | 说明                                   |
| ------------- | ------------ | -------------------------------------- |
| id            | bigint       | 主键                                   |
| task_id       | bigint       | 任务ID                                 |
| from_status   | varchar(30)  | 原状态                                 |
| to_status     | varchar(30)  | 新状态                                 |
| change_reason | varchar(255) | 变更原因                               |
| operator_type | varchar(30)  | 操作来源：SYSTEM / USER / NODE / ADMIN |
| operator_id   | bigint       | 操作人ID，可为空                       |
| created_at    | datetime     | 变更时间                               |

### 说明

这张表一定要有。
因为老师一问“任务全过程如何追踪”，你就靠这张表回答。

### 它解决的问题

- 任务状态全链路可追溯
- 支持详情页显示状态时间线
- 支持异常排查
- 支持幂等与合法流转审计

补充说明：

- “当前排队原因说明（queueReason）”更适合作为查询层动态派生字段，而不是落库存储字段；
- 因为排队原因会随着节点在线状态、节点负载、前序任务完成情况而变化，直接写入 `sim_task` 容易很快过期；
- 首版无需为 `queueReason` 新增数据库字段，可在任务查询时结合任务状态、调度排序规则和当前节点快照动态计算；
- `queueReason` 仅作为面向用户的解释字段使用，不作为调度审计字段；调度审计仍应依赖 `task_status_history` 和 `schedule_record`。

## 表 8：task_file

用途：任务输入文件表

| 字段名       | 类型         | 说明                               |
| ------------ | ------------ | ---------------------------------- |
| id           | bigint       | 主键                               |
| task_id      | bigint       | 任务ID                             |
| file_role    | varchar(30)  | 文件角色：INPUT / CONFIG / ARCHIVE |
| file_key     | varchar(50)  | 对应模板中的文件标识               |
| origin_name  | varchar(255) | 原文件名                           |
| storage_path | varchar(500) | 存储路径                           |
| unpack_dir   | varchar(500) | 校验后真实执行目录，可等于 `workdir` 或其子目录 |
| relative_path| varchar(500) | 解压后相对路径（可空）             |
| archive_flag | tinyint      | 是否归档包：1是，0否               |
| file_size    | bigint       | 文件大小                           |
| file_suffix  | varchar(20)  | 后缀名                             |
| checksum     | varchar(100) | 校验值，可选                       |
| created_at   | datetime     | 创建时间                           |

### 说明

数据库不存大文件，只存元数据和路径。首版统一上传 ZIP，`task_file` 记录压缩包及其解压目录元信息，供后端校验与执行复用。

为避免重复校验堆积无效文件记录，设计上应保证同一任务当前有效的输入压缩包只有一份。用户重新上传文件时，应覆盖或替换该任务原有的 `ARCHIVE + input_archive` 记录，而不是不断追加；数据库层通过唯一约束 `uk_task_file_role_key(task_id, file_role, file_key)` 为该合同提供兜底。
同时，用户主动执行“放弃未提交任务（discard）”或后台执行“未提交任务定时清理”时，可直接利用 `deleted_flag` 对 `CREATED`、`VALIDATED` 的未提交任务做逻辑删除，无需额外新增草稿表。
任务参数修改不需要新增数据库表或字段，直接更新 `sim_task.params_json` 即可；若修改发生在 `VALIDATED` 状态下，业务上应同步将状态回退到 `CREATED`。
管理员手动重试失败任务时，不需要新增数据库字段；首版统一复用同一条 `sim_task`，将 `status` 回到 `QUEUED`，清空 `node_id / fail_type / fail_message / start_time / end_time` 等运行态字段，同时刷新 `submit_time` 为本次重新入队时间，并通过 `task_status_history` 记录一次新的状态流转。若未来需要“自动重试次数上限”能力，再考虑新增 `retry_count`、`max_retry_count` 一类字段。

## 表 9：task_result_summary

| task_id          | bigint       | 任务ID   |
| success_flag     | tinyint      | 是否成功 |
| duration_seconds | int          | 执行耗时 |
| summary_text     | varchar(500) | 结果摘要 |

## 表 13：node_reservation

用途：节点容量预占记录表

| 字段名      | 类型        | 说明 |
| ----------- | ----------- | ---- |
| id          | bigint      | 主键 |
| node_id     | bigint      | 节点ID |
| task_id     | bigint      | 任务ID |
| status      | varchar(20) | `RESERVED / RELEASED` |
| created_at  | datetime    | 创建时间 |
| updated_at  | datetime    | 更新时间 |
| released_at | datetime    | 释放时间，可空 |

### 说明

这张表用于把节点调度预占从“单纯加减 `reserved_count`”升级为“按 `task_id + node_id` 可追踪、可防重”的幂等机制。

补充约束：

- `node_id + task_id` 必须唯一；
- 当某任务首次在某节点上成功预占时，写入 `RESERVED`；若同一 `task_id + node_id` 已为 `RESERVED`，不得重复增加 `compute_node.reserved_count`；
- 当同一任务因管理员重试等原因再次被调度到同一节点，且原记录已为 `RELEASED` 时，可将该记录重新更新为 `RESERVED`，同时将 `released_at` 置空；
- 当该任务派发失败，或成功进入 `RUNNING` 后释放预占时，只允许将 `RESERVED` 更新为 `RELEASED`；
- 重复预占同一 `task_id + node_id` 不应重复增加 `compute_node.reserved_count`；
- 重复释放同一 `task_id + node_id` 不应重复减少 `compute_node.reserved_count`；
- `node_reservation` 表达当前预占幂等状态，不作为完整调度历史表；多轮调度过程仍以 `schedule_record` 和 `task_status_history` 审计；
- `node_reservation` 与 `compute_node.reserved_count` 必须由 scheduler-service 在同一本地事务中维护，禁止 task-service 或 node-agent 直接写入。
- 当历史异常导致 `compute_node.reserved_count` 与 `node_reservation.status=RESERVED` 明细数不一致时，可通过 scheduler-service 内部 `reconcile-reservation` 能力按明细重算汇总字段；数据库层不额外保存修复历史，审计可通过服务日志和运维记录完成。
- 当需要判断 `RESERVED` 明细是否仍对应合法任务阶段时，可通过 scheduler-service 内部 `audit-reservation` 只读审计能力交叉查询任务状态；数据库层不为该审计结果新增表，审计问题由接口实时派生。

## 表 14：node_solver_capability

用途：节点支持的求解器能力表

| 字段名         | 类型        | 说明         |
| -------------- | ----------- | ------------ |
| id             | bigint      | 主键         |
| node_id        | bigint      | 节点ID       |
| solver_id      | bigint      | 求解器ID     |
| solver_version | varchar(50) | 节点安装版本 |
| enabled        | tinyint     | 是否允许该节点以此求解器参与调度 |
| created_at     | datetime    | 创建时间     |

### 为什么必须有这张表

因为你的系统现在不是“所有节点都能跑所有任务”，而是：

- 有的节点支持 OpenFOAM
- 有的节点支持 CalculiX
- 有的节点两个都支持

这正是分布式思想里“异构节点”的体现。

补充约束：

- `compute_node.status` 只表达运行态在线/离线；
- `compute_node.enabled` 表达管理员是否允许该节点参与调度；
- `compute_node.running_count` 表达节点通过心跳上报的真实运行任务数；
- `compute_node.reserved_count` 表达调度服务在“已选中节点但任务尚未真正进入运行”期间持有的预占数；
- `node_reservation` 用于支撑 `task_id + node_id` 级预占/释放幂等，避免重复加减 `reserved_count`；
- 调度可用条件统一表达为 `running_count + reserved_count < max_concurrency`；
- `node_solver_capability` 由节点注册时按实际上报能力同步；
- 节点注册写入 `node_solver_capability` 前，应先确认上报的 `solver_id` 已存在于 `solver_definition`，未知求解器不得生成能力记录；
- 对于已存在的 `node_id + solver_id` 记录，节点重新注册时应保留原有 `enabled` 值，只更新事实上报信息；
- 对于节点最新一次注册中已不再上报的求解器能力，应删除对应能力记录，避免陈旧能力残留。

## 表 15：schedule_record

用途：调度记录表

| 字段名           | 类型         | 说明             |
| ---------------- | ------------ | ---------------- |
| id               | bigint       | 主键             |
| task_id          | bigint       | 任务ID           |
| node_id          | bigint       | 目标节点ID；未选出节点时可为空 |
| strategy_name    | varchar(50)  | 调度策略标识，例如 `FCFS_MIN_LOAD` |
| schedule_status  | varchar(30)  | SUCCESS / FAILED |
| schedule_message | varchar(255) | 调度说明         |
| created_at       | datetime     | 调度时间         |

### 说明

这张表专门解决：

- 任务是如何被分配的
- 分配给了谁
- 失败原因是什么
- 调度过程怎么审计

补充说明：

- 对外统一使用稳定策略标识，如 `FCFS_MIN_LOAD`；
- 代码中的策略类名可以不同，例如 `FcfsLeastLoadStrategy`，但不建议直接把类名写入数据库。
- 对于接口中的 `recoverable` 一类字段，建议仅作为调度失败时的流程控制参数使用，不单独新增数据库列；
- `node_id` 语义应统一为：未成功选出节点时允许为空；若已选出节点但后续派发失败，应保留对应 `node_id` 便于审计；
- `schedule_message` 保存面向管理员审计的短说明，应在服务端写入前按 255 字符列宽截断，并尽量区分无在线节点、无启用求解器能力、节点不可调度或容量不足、预占冲突、派发失败等原因；
- 是否回退到 `QUEUED` 或转为 `FAILED`，应最终体现在 `sim_task.status`、失败字段以及 `task_status_history` / `schedule_record` 中。

# 五、为什么这套设计符合微服务思想

你老师很可能会问这个，所以这里我直接帮你把答辩逻辑也写出来。

## 1. 数据归属明确

- 用户数据归 `user-service`
- 求解器与模板数据归 `solver-service`
- 任务与结果数据归 `task-service`
- 节点与调度数据归 `scheduler-service`

## 2. 不跨服务直接写库

服务之间通过接口或消息传递业务主键，不通过数据库直接耦合。

例如：

- task-service 创建任务后，scheduler-service 只拿 `task_id`
- scheduler-service 不需要保存任务全部详情
- task-service 只记录 `node_id`，不维护节点完整数据

## 3. 去中心化数据管理

每个服务拥有自己数据的写权限，这就是微服务最核心的数据库思想。

# 六、为什么这套设计符合分布式思想

## 1. 节点是独立资源单元

通过 `compute_node` 表管理多个节点，体现多节点协同。

## 2. 节点能力异构

通过 `node_solver_capability` 体现不同节点支持不同求解器；其中节点负责上报事实能力，平台负责控制是否允许该节点 / 该节点-求解器能力参与调度。

## 3. 调度与执行分离

- scheduler-service 负责选节点
- node-agent 负责执行
- task-service 负责记录任务结果

这是一个典型分布式执行闭环。

## 4. 心跳机制支持故障识别

通过 `last_heartbeat_time` 可以检测节点是否离线。

# 八、几个实现上的简化建议

为了容易开发，我建议你这样落地。

## 1. 不做复杂外键

数据库层面可以不强依赖外键约束，改由应用层保证一致性。
这样更贴近微服务风格，也方便后续拆分。

## 2. 参数和结果指标统一用 JSON

不要过度关系化。

- `params_json`
- `metrics_json`

这样不同求解器和任务类型都能兼容。

## 3. 日志采用“分片表”或“文件 + 偏移读取”二选一

想简单就文件。
想演示效果更好就分片表。

## 4. 文件本体不进数据库

只存路径、大小、校验值。

## 补充说明

这份 SQL 采用了文档中的核心拆分方式：

- `user-service`：`sys_user`、`sys_role`
- `solver-service`：`solver_definition`、`solver_task_profile`、`solver_profile_file_rule`
- `task-service`：`sim_task`、`task_status_history`、`task_file`、`task_result_summary`、`task_result_file`、`task_log_chunk`
- `scheduler-service`：`compute_node`、`node_reservation`、`node_solver_capability`、`schedule_record`

同时，文档还明确推荐：

- 服务间通过业务主键关联，不做跨库联表
- 不强依赖复杂外键
- `params_json`、`metrics_json` 这类扩展字段使用 JSON 文本存储
- 重点索引包括 `sim_task` 的 `user_id/status/solver_id/profile_id/submit_time`，以及 `task_result_file(task_id, file_type, file_name)`、`task_log_chunk(task_id, seq_no)`、`compute_node(status/enabled/running_count/reserved_count/last_heartbeat_time)`、`schedule_record(task_id/node_id)` 等。

## 实际落地建议

1. 初始化测试数据
   比如默认管理员、默认求解器、默认模板、默认节点。
2. 主键生成策略
   这份脚本里的 `id` 是 `BIGINT NOT NULL`，适合你在项目里统一用：

- 雪花算法
- MyBatis-Plus 分配 ID
- 或者改成 `AUTO_INCREMENT`

如果你想，我下一条可以继续直接给你一份 **“带初始化数据的完整版 SQL”**，把管理员、角色、求解器、模板、节点示例数据也一并补齐。
