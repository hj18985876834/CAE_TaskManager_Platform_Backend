- `node-agent` 还只是“本机进程执行”，并没有把 `CONTAINER` 执行模式真正做出来。`solver-service` 有 `CONTAINER` 枚举，但执行侧没有容器运行时集成。

# 4.2 详细开发计划（可执行版）

目标：把当前“可联调原型”提升为“可稳定演示、可容器部署、具备基础分布式治理能力”的版本。

## 阶段划分

- 阶段 A（P0，1~2 周）：联调闭环与口径收口
- 阶段 B（P1，2~3 周）：调度能力增强与失败恢复
- 阶段 C（P2，2~3 周）：容器化与部署标准化
- 阶段 D（P3，2 周+）：安全与治理增强

## 阶段 A（P0）任务清单

1. 端到端烟测固化

- 目标：按主链完成一次可复现验证。
- 涉及：gateway/user/solver/task/scheduler/node-agent 全链。
- 产物：`docs/e2e-run-report.md`（每步请求、响应、截图、结论）。
- 验收：至少 2 个任务（1 成功、1 失败）跑完整状态流转。

2. 接口与数据库口径收口

- 目标：文档、SQL、代码三方一致。
- 涉及：错误码、字段命名、空值语义、状态枚举。
- 产物：`docs/api-contract-delta.md`（差异清单 + 处理状态）。
- 验收：无“代码允许但库表不允许”的字段冲突。

3. 乱码与编码稳定性回归

- 目标：Windows/Linux 场景日志均可读。
- 涉及：node-agent 日志采集、task-service 日志查询。
- 产物：`docs/encoding-regression.md`。
- 验收：中文/英文日志均正常显示，无 `����`。

## 阶段 B（P1）任务清单

1. 调度失败重试与重调度

- 目标：失败可恢复，不出现任务“卡死”。
- 涉及：`scheduler-service`、`task-service`。
- 实施建议：
  - 任务增加调度尝试计数。
  - 区分临时失败与终态失败。
  - 达到上限后进入失败终态。
- 验收：模拟节点拒绝/离线后，任务可重试并最终有明确终态。

2. 节点离线任务恢复

- 目标：节点掉线后任务可重置为待调度或失败。
- 涉及：`NodeOfflineCheckJob`、任务状态迁移。
- 验收：节点离线后不会存在长期悬挂 `RUNNING` 任务。

3. 调度策略可配置化

- 目标：在策略框架上增加至少 1 个新策略并可切换。
- 建议新增：`PriorityLeastLoadStrategy`。
- 验收：策略名写入调度记录，切换策略后行为可观察。

## 阶段 C（P2）任务清单

1. 容器化交付

- 目标：一键拉起完整环境。
- 涉及：所有服务 + MySQL。
- 产物：
  - 根目录 `docker-compose.yml`
  - 各服务 `Dockerfile`
- 验收：`docker compose up -d` 后可完成主链烟测。

2. 服务地址配置化与环境分层

- 目标：本地/测试/容器环境通过配置切换，不改代码。
- 涉及：gateway 路由配置、remote client base-url。
- 产物：`application-local.yml`、`application-docker.yml`。
- 验收：三种环境切换只改 profile，不改 Java 代码。

3. 存储抽象升级（第一步）

- 目标：保留本地存储实现，同时引入对象存储接口抽象。
- 涉及：`task-service` 文件上传与下载、`node-agent` 输入准备。
- 验收：存储实现可插拔，现有功能不回退。

## 阶段 D（P3）任务清单

1. 真正 JWT 与认证升级

- 目标：签名、过期、角色声明、刷新机制。
- 涉及：`common-lib`、`gateway-service`、`user-service`。
- 验收：伪造 token 不可通过，过期 token 正确拦截。

2. 密码方案升级为 BCrypt

- 目标：替换 SHA-256，支持平滑迁移。
- 方案：旧密码兼容校验 + 登录后升级写回。
- 验收：历史账号可登录，新密码均为 BCrypt。

3. 可观测性基础建设

- 目标：traceId + 核心指标。
- 涉及：网关、调度器、任务服务、节点代理。
- 验收：可追踪单任务全链路日志和关键耗时。

## 里程碑与验收建议

1. M1（第 2 周末）

- 完成 P0。
- 可提交“可复现 E2E 报告 + 差异收口报告”。

2. M2（第 5 周末）

- 完成 P1。
- 调度失败恢复能力具备，优先级与策略可观测。

3. M3（第 8 周末）

- 完成 P2。
- 支持容器化一键部署，环境配置分层清晰。

4. M4（第 10 周末）

- 完成 P3 核心项。
- 安全与可观测性达到“课程设计/毕设答辩可交付”标准。

## 风险与应对

1. 风险：跨服务改造范围大，回归成本高。

- 应对：每阶段先建回归清单，执行“改一项，回归一项”。

2. 风险：调度策略改造引入状态机异常。

- 应对：先加状态迁移测试，再改业务逻辑。

3. 风险：容器化后路径和网络与本地不一致。

- 应对：统一路径映射规范，先用 compose 固化网络别名。
