# 4.2 详细开发计划（可执行版）

- 阶段 A（P0，1~2 周）：联调闭环与口径收口
- 阶段 B（P1，2~3 周）：调度能力增强与失败恢复

- 阶段 D（P3，2 周+）：安全与治理增强

## 阶段 A（P0）任务清单

1. 端到端烟测固化

- 目标：按主链完成一次可复现验证。

- 验收：至少 2 个任务（1 成功、1 失败）跑完整状态流转。

- 涉及：`scheduler-service`、`task-service`。
- 实施建议：
  - 任务增加调度尝试计数。
  - 区分临时失败与终态失败。
  - 达到上限后进入失败终态。
- 验收：模拟节点拒绝/离线后，任务可重试并最终有明确终态。

2. 节点离线任务恢复

- 目标：节点掉线后任务可重置为待调度或失败。
- 涉及：`NodeOfflineCheckJob`、任务状态迁移。
- 验收：节点离线后不会存在长期悬挂 `RUNNING` 任务。

3. 调度策略可配置化

- 目标：在策略框架上增加至少 1 个新策略并可切换。
- 建议新增：`PriorityLeastLoadStrategy`。
- 验收：策略名写入调度记录，切换策略后行为可观察。

- 网关虽然预留了 discovery 配置，但没有真正接入服务发现体系。

- 多实例调度安全未完成。
- 更高级的调度策略未完成。
- 调度“解释字段”已实现，但调度控制能力本身仍偏基础版。

#### 4.4 执行环境

- `solver-service` 已定义 `execMode`，但 `node-agent` 尚未真正区分 `LOCAL / CONTAINER` 执行模式。
- 当前执行仍是本机进程执行。
- 没有容器运行时集成、资源隔离、沙箱隔离策略。

#### 4.5 存储与结果管理

- 文件存储仍是本地文件系统 / 共享目录思路。
- 没有对象存储抽象落地。
- 没有真正的分布式共享制品管理。
- 结果文件收集仍是简单目录扫描，没有归档、过滤、索引、生命周期治理。

#### 4.6 可观测性与测试

- 没有成体系的自动化单元测试 / 集成测试目录。
- 当前更多依赖 `docs/e2e-smoke-checklist.md` 和脚本清单。
- 没有统一 traceId、metrics、链路追踪、Prometheus / OpenTelemetry 等体系。

### 5. 与较早分析相比，需要修正的旧判断

- 旧判断“容器化基本未开始”已不准确：

- 已具备“中心调度 + 节点代理执行 + 状态回传”的基本分布式执行架构。
- 节点状态不是纯数据库静态展示，而是由注册、心跳、离线检测动态驱动。
- 调度侧已经具备按节点在线状态、启用状态、节点能力、并发上限进行筛选的基础能力。
- 当前属于“单调度器 + 多执行节点”的分布式原型阶段，满足本科毕设对分布式原型的基本要求。

### 3. 当前已实现但应明确标注为“基础版”的内容

- 网关路由已实现，但仍是静态服务地址路由，尚未接入注册中心或真实服务发现。
- 鉴权已实现，但 `token` 仍为简化方案，密码存储也仍为基础实现，不应在论文中表述为生产级安全体系。
- 调度已实现，但当前仍以基础策略和单实例调度器为主，不应表述为高可用分布式调度平台。
- 容器化已实现，但当前主要完成平台服务容器化，尚未完成“求解器运行环境全面容器化”。
- 节点执行已实现，但当前仍以本地进程执行为主，`execMode` 尚未真正落到容器执行分支。
- 存储已实现，但主要是本地 / 共享目录存储，尚未升级到对象存储方案。

### 4. 对本科毕设而言仍需补齐的关键项

以下内容建议视为“答辩前仍应完成”的重点，但都应坚持简单、可控、不过度扩展的原则。

#### 4.1 P0：必须补齐，直接影响答辩演示

1. 多节点演示落地方案

- 当前代码架构支持多节点，但默认 `compose.yaml` 只编排了一个 `node-agent` 实例。

  - 第二个节点代理实例的启动配置示例；
  - 不同 `nodeCode / nodeName / advertisedHost / port / workRoot` 的配置方法；
  - 一份“两个节点并行注册与执行”的演示步骤说明。

2. 演示数据与初始化收口

- 需要确保默认数据库初始化后，至少具备可直接演示的管理员、普通用户、求解器、模板、规则、节点示例数据。

- 需要补至少一条异常演示路径：失败任务重试，或节点离线后任务失败补偿。

3. 验收脚本与烟测清单收口

- 需要把现有烟测脚本和接口清单整理成“最终验收清单”。
- 最终目标不是覆盖全部边角，而是保证核心闭环可重复验证。

4. 文档统一口径

- 需求分析与系统设计、后端设计、接口设计、数据库设计、开发文档之间，需要保持“系统定位、功能边界、技术指标、扩展点”的一致口径。

### 8. 最终建议结论

- 如果按照“本科毕设原型平台”目标来衡量，当前项目并不是“还没做出来”，而是“核心能力已经做出来，剩下的是演示化、材料化、收口化工作”。
- 后续开发重点不应继续大幅增加系统复杂度，而应优先保证已有主链路稳定、技术指标明确、演示过程流畅。
- 因此，现阶段最合理的路线不是继续追求生产级架构，而是把现有成果收敛为一个“核心功能完整、技术指标明确、扩展边界清楚”的毕业设计最终版本。
