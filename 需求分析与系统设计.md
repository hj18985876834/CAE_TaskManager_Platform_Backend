# 船舶CAE多求解器仿真任务管理与调度平台

## 需求分析与系统设计（完善版）

------

# 1. 项目概述

## 1.1 项目背景

随着船舶工业向数字化设计和智能制造发展，CAE 仿真已成为结构设计、性能分析和安全评估的重要手段。实际工程中，仿真任务通常具有任务数量多、计算时间长、资源消耗高、输入文件复杂、求解环境依赖强等特点。传统单机执行或人工调度方式存在如下问题：

1. 仿真任务提交方式分散，缺乏统一入口；
2. 不同仿真软件的输入文件格式和运行方式差异较大；
3. 多用户并发提交时资源利用率低，任务排队与执行过程难以统一管理；
4. 计算节点数量增加后，手工分配方式难以维护；
5. 软件环境、依赖版本和部署方式不一致，导致运行稳定性差。

因此，有必要设计一个面向 CAE 场景的**多求解器仿真任务管理与分布式执行平台**，对任务从提交、校验、排队、调度、执行、监控到结果反馈进行统一管理。

------

## 1.2 建设目标

本系统建设目标如下：

1. 实现面向船舶 CAE 场景的仿真任务统一管理；
2. 支持多用户同时提交不同类型的仿真任务；
3. 支持接入多个真实开源求解器，并保留模拟执行器用于测试与演示；
4. 支持根据“求解器 + 任务类型”自动确定输入文件要求并进行校验；
5. 支持多计算节点协同执行任务，体现分布式能力；
6. 采用微服务架构实现模块解耦，提高系统可维护性和可扩展性；
7. 采用容器化方式保证部署一致性，并便于后续扩展和迁移；
8. 通过可视化界面展示任务状态、节点状态、调度记录与结果信息。

------

## 1.3 系统定位

本系统定位为：

> **面向多求解器的 CAE 仿真任务管理、调度与执行平台**

系统重点在于：

- 仿真任务生命周期管理；
- 多求解器接入与统一调用；
- 输入文件模板校验；
- 分布式任务调度；
- 节点执行过程监控；
- 结果摘要与文件反馈；
- 微服务化、容器化部署。

系统**不以实现具体求解算法为重点**，而是聚焦于仿真任务的工程化管理与执行框架设计。

------

## 1.4 系统边界说明

为控制实现复杂度并保证课题可落地，系统边界定义如下：

### 系统负责的内容

- 用户管理与权限控制；
- 任务创建、查询、取消、状态管理；
- 求解器注册与任务类型管理；
- 输入文件模板定义与校验；
- 任务队列维护与节点调度；
- 节点心跳、状态上报、日志回传；
- 结果摘要展示与结果文件管理。

### 系统不负责的内容

- 复杂 CAD/CAE 建模与前处理；
- 商业仿真软件许可证管理；
- 高精度三维后处理与专业可视化；
- 复杂 HPC 集群编排与弹性资源调度；
- 全自动网格生成与复杂前处理流程。

本系统默认用户提交的输入文件已完成基础准备，平台负责统一组织、校验、分发和执行。

# 2. 求解器接入思路与可行性

## 2.1 真实求解器与模拟执行器并存

为兼顾真实性与实现难度，本系统采用“双轨制”方案：

1. **真实开源求解器**
   - 用于体现平台的实际工程价值；
   - 首批接入代表性开源软件，如 OpenFOAM、CalculiX；
   - 平台通过统一接口调用求解器完成真实计算。
2. **模拟执行器**
   - 用于测试任务流程、演示异常场景、回归验证；
   - 在真实求解器不可用时可继续验证系统功能；
   - 便于调试调度逻辑和日志回传逻辑。

这种设计既保留真实计算能力，又避免整个平台过度依赖外部软件。

------

## 2.2 首批接入求解器选择

### 1）OpenFOAM

适合作为流体类仿真求解器。官方文档说明其应用主要通过命令行执行，按 case 目录中的数据文件读写，且天然适合自动化和批处理；官方用户指南还专门说明了并行运行方式与 case 文件结构。([OpenFOAM](https://www.openfoam.com/documentation/user-guide/3-running-applications/3-1-running-applications?utm_source=chatgpt.com))

适合接入的任务类型：

- 稳态流体分析
- 瞬态流体分析
- 简单管道或外流场基础案例

### 2）CalculiX

适合作为结构与热分析求解器。CalculiX 官方站点说明其是三维有限元程序，包含建模、计算和后处理能力，且可基于求解输入文件完成结构与热类计算。([CalculiX](https://www.calculix.de/?utm_source=chatgpt.com))

适合接入的任务类型：

- 结构静力分析
- 热分析
- 简单线性动力分析（可选扩展）

### 3）MockSolver

系统自定义模拟执行器，用于：

- 功能演示；
- 异常流程测试；
- 无真实环境时的兜底执行。

# 4. 核心业务对象设计

本系统的核心业务对象不再只有“任务”，而是扩展为以下五类：

## 4.1 任务类型（Task Type）

表示业务层面的仿真类别，例如：

- 结构静力分析
- 热分析
- 流体稳态分析
- 流体瞬态分析

## 4.2 求解器（Solver）

表示实际执行软件，例如：

- OpenFOAM
- CalculiX
- MockSolver

## 4.3 求解器任务模板（Solver Task Profile）

表示“某求解器支持某种任务类型时”的执行规范，包括：

- 模板元信息（模板编码、模板名称、适用任务类型）；
- 文件规则集合（必需/可选文件、路径与命名约束）；
- 参数定义；
- 上传模式；
- 命令模板；
- 结果解析方式；
- 默认超时策略。

## 4.4 计算节点（Compute Node）

表示可参与任务执行的节点，记录其：

- 在线状态；
- 调度启用状态；
- 支持的求解器及安装版本；
- 各求解器能力是否允许参与调度；
- 当前负载；
- 任务数；
- 最近心跳时间。

## 4.5 仿真任务（Simulation Task）

表示用户提交的具体任务，包括：

- 所选求解器；
- 所选任务类型；
- 输入文件；
- 参数配置；
- 当前状态；
- 所属节点；
- 日志与结果信息。

## 5.1 用户业务需求

用户应能够通过账号密码登录系统，并获取访问令牌。

用户应能够在创建任务时查看平台当前支持的求解器列表，并选择其中一种。

在选定求解器后，系统应只展示该求解器支持的任务类型，避免非法组合。

### 5.1.4 输入文件模板提示

当用户选定“求解器 + 任务类型”后，系统应自动展示该任务所需的文件类型、目录结构和上传要求。

### 5.1.5 任务提交

用户可上传输入文件、填写参数并提交任务，系统应执行表单校验、文件校验和模板校验。任务提交入队前应以当前求解器和模板状态为准重新确认其仍启用，且模板归属求解器与任务类型仍匹配。

### 5.1.6 任务查看

用户应能够查看自己创建的任务列表，并根据状态、时间、名称进行查询。

### 5.1.7 任务详情查看

用户应能够查看任务基础信息、状态流转记录、所属节点、日志信息、结果摘要和结果文件列表。
当任务处于 `QUEUED` 状态时，系统还应向用户展示当前排队原因说明，例如“前方仍有更高优先级任务”“暂无满足条件的可用节点”“候选节点当前满载”等，帮助用户理解任务为何尚未被调度。该说明属于面向用户的解释字段，不作为调度审计依据；真正的状态变化、调度过程与失败原因仍以状态流转记录和调度记录为准。

### 5.1.8 日志查看

执行过程中用户应能够动态查看日志输出。

### 5.1.9 结果查看

任务完成后，用户应能够查看结果摘要，并下载结果文件。

### 5.1.10 任务取消

首版固定支持对排队中的任务执行取消；运行中的终止请求不纳入首版基线，若后续需要可作为扩展项单独实现。

# 7. 任务生命周期设计

## 7.1 任务状态定义

为尽量兼容现有前后端框架，本系统仍保留单一 `sim_task` 任务对象，不额外拆分草稿表或草稿接口。但在语义上应明确：

1. `CREATED` 表示任务已创建、正在编辑或等待重新校验，此时尚未进入队列；
2. `VALIDATED` 表示当前文件与参数已校验通过，可以提交入队；
3. `QUEUED` 及其后的状态表示任务已经进入正式执行生命周期。

首版统一定义如下状态：

1. `CREATED`：任务已创建但尚未提交，可反复上传和校验
2. `VALIDATED`：输入校验完成，可以提交
3. `QUEUED`：任务进入待调度队列
4. `SCHEDULED`：调度服务已选定节点
5. `DISPATCHED`：任务已下发到节点
6. `RUNNING`：节点代理已启动求解
7. `SUCCESS`：执行成功
8. `FAILED`：执行失败
9. `CANCELED`：已取消
10. `TIMEOUT`：执行超时

------

## 7.2 状态流转规则

- 新建任务只允许调用一次 `POST /api/tasks`，后续上传和校验必须复用同一个 `taskId`；
- 同一任务在重新上传文件或修改参数后，状态回到 `CREATED`；
- 参数修改应通过任务编辑接口完成，而不是重新创建任务；
- 校验失败时任务仍保留在当前 `taskId` 上，不新建任务记录；
- 校验通过后才允许流转到 `VALIDATED`；

任务执行阶段状态流转关系如下：

- `CREATED → VALIDATED`
- `VALIDATED → CREATED`（文件或参数变更后重新校验）
- `VALIDATED → QUEUED`
- `QUEUED → SCHEDULED`
- `SCHEDULED → DISPATCHED`
- `SCHEDULED → QUEUED`（节点临时不可用、派发失败但可重试时回退队列）
- `SCHEDULED → FAILED`（调度或派发阶段不可恢复失败）
- `DISPATCHED → RUNNING`
- `DISPATCHED → QUEUED`（节点已接收但未真正开始执行且允许重新入队）
- `DISPATCHED → FAILED`（节点拒收或执行前不可恢复失败）
- `RUNNING → SUCCESS`
- `RUNNING → FAILED`
- `RUNNING → TIMEOUT`
- `FAILED → QUEUED`（管理员手动重试）
- `TIMEOUT → QUEUED`（管理员手动重试）
- `QUEUED → CANCELED`

要求：

- 状态切换必须合法；
- 每次状态变化必须有记录；
- 非法状态变更必须阻止；
- 重复回传应按幂等处理；
- 同一任务重复校验应复用同一 `taskId`；
- 管理员手动重试应复用原 `taskId`，不额外创建“重试副本任务”；
- 手动重试回到 `QUEUED` 时，应清理上一轮执行留下的 `nodeId`、失败信息、开始/结束时间等运行态字段，并将本次重新入队时间作为新的排队时间；
- 用户在 `CREATED`、`VALIDATED` 阶段主动取消创建时，应执行“放弃未提交任务（discard）”，直接丢弃当前未提交任务，而不是转为 `CANCELED`；
- 用户中途离开页面或长期未提交时，应由“未提交任务定时清理”策略对这类任务做软删除和目录回收，避免残留数据堆积。
- 为保证状态流转可解释，节点执行结果上报不应跨越 `RUNNING` 直接把 `DISPATCHED` 任务置为终态；若存在“调度或派发失败但未真正运行”的情况，应优先回退到 `QUEUED`，不可恢复时才允许在 `SCHEDULED` / `DISPATCHED` 阶段转为 `FAILED`，并记录明确 `failType` 与原因。
- node-agent 在执行前准备失败且尚未进入 `RUNNING` 时，可通过 `dispatch-failed` 回写派发阶段失败；该节点侧回写必须进行节点 token 校验，避免未授权节点伪造失败状态；状态成功回写后必须释放本轮节点预占，避免 `reserved_count` 泄漏。
- 若 node-agent 在 `status-report(RUNNING)` 重试耗尽后改走 `dispatch-failed(recoverable=true)`，但中心已先进入 `RUNNING`，则 task-service 应拒绝该迟到回队并补一条 same-status 节点状态历史说明，标记“迟到 dispatch-failed 已被拒绝”；node-agent 再改用 `mark-failed(RUNTIME_ERROR)` 收口，避免 ACK 丢失场景下只能依赖服务日志推断。
- 首版不要求实现 `RUNNING → CANCELED`；若第二阶段引入运行中终止，再补充该状态流转与 node-agent 取消链路。

# 10. 分布式执行需求分析

# 11. 调度需求分析

## 11.1 调度目标

调度服务负责从待执行任务中选择合适任务，并分配给满足条件的节点执行。

## 11.2 基线调度策略

采用：

> **优先级优先 + 先来先服务（FCFS） + 求解器匹配 + 最小负载优先**

调度流程为：

1. 扫描 `QUEUED` 状态任务；
2. 读取节点状态；
3. 按任务优先级降序、提交时间升序读取待调度任务；
4. 过滤不在线、不启用、不支持所需求解器或对应节点-求解器能力未启用的节点；
5. 过滤已满载节点；
6. 从候选节点中选择负载最低者；
7. 对目标节点执行容量预占；
8. 原子确认任务仍处于 `QUEUED` 状态并写入目标节点；
9. 若任务确认失败，应根据返回状态区分处理：同节点仍为 `SCHEDULED` 可继续下发，同节点已 `DISPATCHED` 说明已有在途派发应停止本轮且不释放该在途预占，已 `RUNNING` 或终态不得重复下发并可释放预占兜底，其他失败场景释放节点预占并继续下一次调度；
10. 对于上述“确认失败且不再继续本轮下发”的场景，若原因是任务已进入 `RUNNING`、终态，或已被其他节点/其他调度实例抢占，scheduler-service 应尽量补写一条 `schedule_record(FAILED)` 短说明，标记“本轮调度抢占已被拒绝”；同节点已 `DISPATCHED` 的在途幂等场景不额外补写失败调度记录；
11. 下发任务并记录调度结果。

补充说明：

- 节点容量判断应区分“节点真实运行数”和“调度预占数”；
- 调度可用条件统一表达为：`running_count + reserved_count < max_concurrency`；
- `reserved_count` 只作为节点预占汇总值，`task_id + node_id` 级幂等应由 `node_reservation` 记录承担；
- scheduler-service 在预占或释放时，应在同一本地事务中同时维护 `node_reservation` 与 `reserved_count`，task-service 不直接写节点预占表；
- 若任务因节点离线补偿从 `SCHEDULED` 或 `DISPATCHED` 回退到 `QUEUED`，应在状态更新成功后释放对应节点预占；
- 当节点临时拒绝、网络闪断或派发过程出现可恢复错误时，任务应允许回退到 `QUEUED`，而不是一律直接置为 `FAILED`。
- node-agent 对同一 `taskId` 的重复派发应幂等返回成功且不重复启动执行线程，避免网络重试或迟到响应导致中心误回队。

## 11.3 扩展策略

后续可扩展：

- 预估时长调度；
- 类型亲和性调度；
- 失败重试调度。

# 12. 异常场景分析与解决方案

## 12.1 输入文件不合法

### 问题

用户上传的文件缺失、命名错误或目录结构错误。

### 解决方案

- 模板化校验；
- 校验失败不入队；
- 返回明确错误信息。

## 12.2 求解器未安装或不可用

### 问题

节点已注册，但求解器环境缺失或版本不匹配。

### 解决方案

- 节点注册时同步上报本机实际安装或本地配置启用的求解器及版本；
- 平台应先校验节点上报的求解器是否存在于全局求解器定义中，未知求解器不应写入节点能力表；
- 管理员只控制“是否允许该节点 / 该节点-求解器能力参与调度”，不直接虚构节点能力；
- 调度只向“全局求解器启用 + 模板启用 + 模板归属求解器匹配 + 任务类型匹配 + 节点在线 + 节点启用 + 节点-求解器能力启用”的节点派发；
- 节点执行前再做一次本地检查；
- 命令模板只允许轻量主命令和稳定占位符，首版稳定占位符包括任务标识、求解器标识、模板标识、任务类型、`${solverExecPath}`、`${taskDir}`、`${inputDir}`、`${outputDir}`、`${logDir}` 以及任务参数，禁止使用 `${workDir}` 作为对外模板占位符。

## 12.3 节点离线

### 问题

执行过程中节点心跳丢失。

### 解决方案

- 节点每 10 秒发送一次心跳；
- 超过 30 秒未收到心跳则标记离线；
- 该节点不再参与调度；
- scheduler-service 应通过内部接口 `POST /internal/tasks/node-offline/fail` 对该节点上尚未完成的任务做统一补偿收口：
  - `SCHEDULED`、`DISPATCHED` 阶段任务统一回退到 `QUEUED`；
  - 对回退到 `QUEUED` 的任务，`NODE_OFFLINE` 只写入状态历史或调度记录，不写回 `sim_task.fail_type`；
  - 对回退到 `QUEUED` 的任务，task-service 在状态更新成功后应释放对应节点预占；
  - `RUNNING` 阶段任务在首版统一收口为 `FAILED`，并记录 `failType = NODE_OFFLINE`；
  - 若 `RUNNING` 任务在更早的 `DISPATCHED -> RUNNING` 阶段曾出现预占释放失败，后续无论正常终态还是 `NODE_OFFLINE` 终态，在状态收口成功后都应尽力再次释放该残留预占，但不得因此回滚终态；
  - 对每条实际完成状态变更的任务，应尽量补写一条调度失败记录，便于管理员从调度时间线直观看到“节点离线导致本轮调度收口”；
- 离线补偿按节点逐个执行，单个节点补偿失败不得影响本轮其他节点处理；失败节点由后续离线检查周期继续重试补偿。

## 12.4 日志丢失或结果丢失

### 解决方案

- 节点本地先写文件；
- 平台周期同步日志；
- 结果文件统一存储在共享目录或挂载卷中。

## 12.5 重复调度或重复回传

### 解决方案

- 调度前检查当前状态；
- 状态流转加条件限制；
- 节点回传按“`taskId` + 当前绑定 `nodeId` + 合法状态阶段”做幂等处理；
- `mark-dispatched` 作为调度确认接口，正常只允许 `SCHEDULED -> DISPATCHED`；若同节点迟到确认发生在任务已进入 `RUNNING` 或终态后，应幂等返回当前状态，不得回退状态。
- 若任务已进入目标状态或已进入终态，重复回传应直接忽略，不得重复释放预占或重复写终态。
- 结果摘要和结果文件只允许在 `RUNNING / SUCCESS / TIMEOUT` 阶段接收；已 `FAILED / CANCELED` 的迟到回传不得继续写入成功结果，`mark-finished / mark-failed` 客户端应把中心返回的任一终态视为幂等 ACK。
- 若求解器已成功结束，但结果摘要、结果文件或 `mark-finished` 回调持续失败，node-agent 应按 `CALLBACK_ERROR` 将任务收口到 `FAILED`；task-service 成功落库后应尽力清理已写入的结果摘要与结果文件元数据，避免失败任务仍展示成功结果；若清理失败，只记录告警，不回滚终态。
- 若节点在结果回传阶段被判离线，task-service 将 `RUNNING` 任务补偿为 `FAILED(NODE_OFFLINE)` 后，也应尽力清理此前已写入的结果摘要与结果文件元数据，避免失败任务继续展示成功结果；若清理失败，只记录告警，不回滚终态。
- 对外结果摘要查看、结果文件列表与下载接口只在任务当前处于 `RUNNING / SUCCESS / TIMEOUT` 时开放；若任务已为 `FAILED / CANCELED` 或其他未产出结果的前置状态，即使存在历史残留结果元数据，也不得继续向前端暴露。
- task-service 在接收结果摘要、结果文件回传时，应把“阶段校验”和“结果元数据写入”放在同一事务和同一任务行锁内执行，避免并发终态更新后仍写入成功结果。
- 若中心已先进入终态，node-agent 后续迟到的 `mark-finished / mark-failed` 虽仍按幂等接受，但系统应补充一条不改状态的节点审计说明，便于解释 ACK 丢失后的重复终态回调。
- 若中心已先进入 `FAILED / CANCELED` 等不允许结果写入的终态，迟到 `result-summary-report / result-file-report` 应被拒绝，且系统应补充一条不改状态的节点审计说明，便于解释“迟到结果回调已被拒绝”，而不是静默丢弃。

## 12.6 运行中取消困难

### 解决方案

- 核心版本先支持排队中取消；
- 运行中取消作为扩展项；
- 若支持终止，则由节点代理 kill 对应进程。

# 14. 系统总体设计

------

## 14.1 总体架构设计

系统采用前后端分离 + 微服务架构，整体由以下部分组成：

1. 前端展示层
2. API 网关层
3. 用户服务
4. 任务服务
5. 求解器管理服务
6. 调度服务
7. 计算节点代理
8. 数据存储与文件存储层

整体逻辑为：

> 用户创建任务 → 上传文件并填写参数 → 任务服务对当前 taskId 执行模板校验 → 校验通过后提交任务 → 任务入队 → 调度服务选择节点 → 节点代理执行求解器 → 回传日志与状态 → 平台展示结果

职责：

- 权限校验。

### 2）User Service

职责：

- 用户登录；
- 用户管理；
- 角色权限控制。

### 3）Task Service

职责：

- 任务创建；
- 文件上传；
- 任务查询；
- 状态管理；
- 日志与结果管理。

### 4）Solver Service

职责：

- 求解器定义管理；
- 模板管理；
- 文件规范管理；
- 解析器定义管理。

### 5）Scheduler Service

职责：

- 维护待调度队列；
- 读取节点状态；

### 6）Node Agent

## 14.3 为什么这样划分

这种划分既能清晰体现微服务要求，也不会把系统拆得过细，适合本科阶段实现。
特别是新增 `solver-service` 后，系统能够更好支撑“多求解器接入与扩展”，避免将求解器逻辑硬编码在任务服务中。

# 15. 多求解器接入架构设计

## 15.1 核心思想

采用：

> **模板驱动 + 适配器模式**

### 模板驱动

用配置描述：

- 支持的任务类型；
- 输入文件要求；
- 命令模板；
- 结果识别规则；
- 默认超时；
- 解析器名称。

补充约束：

- 模板配置应优先描述“不同求解器、不同任务类型之间的差异”，不应把平台级执行流程全部塞入模板；
- `command_template` 应保持轻量，主要表达主执行命令及少量必要占位符，例如 `${solverExecPath}`、`${taskDir}`、`${inputDir}`、`${outputDir}`、`${logDir}`；
- 任务参数占位符必须来自模板参数 JSON Schema 的 `properties`，模板保存阶段应前置校验未知占位符，防止配置错误延迟到节点执行阶段才暴露；
- 模板参数 JSON Schema 本身也应在模板保存阶段完成基本合法性校验，非法 JSON、错误的 `properties/required/items/additionalProperties` 结构和非法正则不应进入可用模板；
- `taskDir` 统一表示节点侧最终执行目录，由任务校验阶段记录的 `task_file.unpack_dir` 映射得到；首版不再单独引入 `${workDir}` 作为模板占位符；
- 目录准备、通用执行包装、通用结果归集、日志收集等共性流程应由节点代理或求解器适配器承担；
- 首版不鼓励把复杂 shell、长流程控制或大量正则派生逻辑写入数据库模板，避免后续新增模板时配置成本过高。

### 适配器模式

为每个求解器实现一个适配器，例如：

- `OpenFoamAdapter`
- `CalculiXAdapter`
- `MockSolverAdapter`

统一接口可定义为：

- `validateInput()`
- `prepareWorkspace()`
- `buildCommand()`
- `execute()`
- `collectLogs()`
- `parseResult()`

职责边界：

- 模板负责声明“需要什么”和“主要执行什么”；
- 适配器负责处理“如何准备、如何构造最终命令、如何收集结果”；
- 对 OpenFOAM、CalculiX 等命令行求解器，可先使用通用命令模板执行器，但后续遇到明显求解器特有流程时，应优先下沉到适配器，而不是继续扩张模板 JSON。

------

## 15.2 接入新求解器流程

当后续接入新软件时，只需：

1. 注册新的求解器定义；
2. 新增对应任务类型模板；
3. 编写一个新的适配器；
4. 编写一个结果解析器；
5. 在节点 agent 的本地配置或探测逻辑中声明并上报其实际支持该求解器。

这样即可最大程度减少对原有系统的改动。

# 16. 分布式执行设计

## 16.1 节点代理设计

节点代理部署在计算节点上，负责：

- 注册节点；
- 定时发送心跳；
- 上报本机实际支持的求解器及版本；
- 接收调度任务；
- 启动对应求解器进程；
- 周期回传日志和状态；
- 回传结果文件信息。

## 16.2 任务下发方式

调度服务选定节点后，通过 REST 接口将任务下发给节点代理，节点代理读取任务工作目录并启动执行。

## 16.3 节点工作目录

首版统一目录结构：

```text
/task-data/{taskId}/
  input/
  workdir/
  output/
  logs/
  meta/
```

其中：

- `input/`：保存用户原始上传归档文件；
- `workdir/`：校验阶段解压与整理后的工作区；
- `output/`：结果文件；
- `logs/`：日志文件；
- `meta/`：任务配置与模板信息。

补充说明：

- 后端应在任务文件元数据中记录当前有效输入归档及其 `unpack_dir`，该字段表示最终真实执行目录；
- 节点代理收到任务后，首版应复用已校验的 `unpack_dir` 作为实际执行目录；调度请求携带 `unpack_dir` 时，节点必须在创建运行目录前确认映射后的目录真实存在且为目录，不能通过创建空 `workdir` 让校验误通过；
- task-service 下发给调度器的排队任务必须已经完成 ZIP 校验，并在输入归档元数据中具备 `unpack_dir`；未校验归档不得进入正常派发链路；
- 这样可以保证“校验所见目录”和“执行所用目录”保持一致，减少路径漂移和重复 I/O。

# 17. 容器化设计

## 17.1 容器化目标

容器化不仅用于平台服务部署，还用于提升节点环境一致性。

## 17.2 容器化范围

建议容器化以下组件：

- 前端服务；
- 网关服务；
- 用户服务；
- 任务服务；
- 求解器服务；
- 调度服务；
- MySQL；
- Redis（可选）；
- Node Agent。

## 17.3 求解器环境容器化策略

考虑实现难度，采用两阶段方案：

### 阶段一：宿主机安装求解器

- 节点宿主机安装 OpenFOAM 或 CalculiX；
- Node Agent 负责调用本地可执行程序；
- 系统通过节点能力登记区分可执行的软件。

### 阶段二：容器化求解环境（扩展）

- 为 OpenFOAM、CalculiX 制作独立镜像；
- Node Agent 按任务类型拉起对应容器执行。

其中阶段一更容易实现，阶段二可作为扩展亮点。

# 18. 数据设计建议

首版核心数据表与服务归属如下：

- `user-service`：`sys_role`、`sys_user`
- `solver-service`：`solver_definition`、`solver_task_profile`、`solver_profile_file_rule`
- `task-service`：`sim_task`、`task_status_history`、`task_file`、`task_result_summary`、`task_result_file`、`task_log_chunk`
- `scheduler-service`：`compute_node`、`node_solver_capability`、`schedule_record`

其中，前端展示用的 `queueReason` 应作为查询时动态派生字段返回，不单独落库；调度审计仍以 `task_status_history` 与 `schedule_record` 为准。
统一命名约定：接口 JSON 字段使用 `camelCase`，数据库字段使用 `snake_case`，例如 `taskId <-> task_id`、`nodeId <-> node_id`、`submitTime <-> submit_time`、`runningCount <-> running_count`。

## 18.1 角色表 `sys_role`

- id
- role_code
- role_name
- created_at

说明：

- 仅保留 `ADMIN`、`USER` 两类角色；
- 用户接口返回的 `roleCode` 由该表统一定义。

## 18.2 用户表 `sys_user`

- id
- username
- password
- real_name
- role_id
- status
- created_at
- updated_at

说明：

- 通过 `role_id` 关联 `sys_role`，而不是在用户表中直接写字符串角色字段；
- `status` 用于表达账号启用/禁用状态。

## 18.3 求解器定义表 `solver_definition`

- id
- solver_code
- solver_name
- version
- exec_mode
- exec_path
- enabled
- description
- created_at
- updated_at

## 18.4 求解器任务模板表 `solver_task_profile`

- id
- solver_id
- profile_code
- task_type
- profile_name
- upload_mode
- command_template
- params_schema_json
- parser_name
- timeout_seconds
- enabled
- description
- created_at
- updated_at

说明：

- 模板负责定义“求解器 + 任务类型”的执行语义；
- `upload_mode` 首版统一采用 `ZIP_ONLY`；
- `command_template` 用于保存轻量主命令模板，不应长期承载目录创建、结果复制、退出码包装等平台共性 shell 流程；
- `command_template` 属于管理员模板配置项，普通用户创建任务时只选择模板并填写参数，不直接编辑执行命令；
- 参数结构通过 `params_schema_json` 描述，便于前后端与校验逻辑共用。

## 18.5 模板文件规则表 `solver_profile_file_rule`

- id
- profile_id
- file_key
- path_pattern
- file_name_pattern
- file_type
- required_flag
- sort_order
- rule_json
- description

说明：

- 文件规则从模板表中拆出，避免将必需文件、可选文件、目录约束硬编码在单个 JSON 字段里；
- `solver_task_profile` 与 `solver_profile_file_rule` 共同定义上传规范、校验规则与执行目录约束；
- `rule_json` 首版应优先用于后缀、大小、数量等简单约束；
- 在确有必要时，也允许少量作用域选择器，如 `solverCodes`、`taskTypes`，以及受控的 `deriveParam/deriveParams` 用于参数派生；同一条规则内 `deriveParam` 与 `deriveParams` 互斥，派生参数涉及的字段、正则与预处理子项应采用白名单机制，并在模板规则保存阶段前置校验；
- 作用域选择器必须采用白名单机制，未知选择器不得静默忽略，避免模板字段拼写错误导致规则误生效或误跳过；
- `path_pattern` 必须是相对校验后 `unpack_dir/caseDir` 的路径 glob，统一使用 `/` 分隔并支持 `*`、`?`、`**`，不得配置绝对路径或包含 `..`；`file_name_pattern` 只表示文件名 glob，不得包含路径分隔符。历史非法规则在任务校验阶段应按模板规则冲突处理。
- 复杂参数派生或求解器特有解析逻辑应谨慎使用，优先由适配器或后端校验代码承担。
- `allowSuffix` 的语义是文件扩展名本身，允许配置为 `zip` 或 `.zip`，系统内部统一按去除前导点的小写后缀比较。
- `file_type` 必须采用枚举语义，首版只支持 `ZIP`、`FILE`、`DIR`、`DIRECTORY`，非法类型不得作为有效文件规则参与校验。
- 必需规则与数量规则不能自相矛盾，`required_flag=1` 时不允许配置 `maxCount/max=0`；数量约束与必需标记应尽量保持语义一致，避免模板使用者产生误解。
- `maxSizeMb` 必须为正整数，`minCount/min/maxCount/max/group` 必须为非负整数；历史非法 `rule_json` 在任务校验阶段应作为模板规则冲突处理，不得静默忽略。

## 18.10 任务结果文件表 `task_result_file`

- id
- task_id
- file_type
- file_name
- storage_path
- file_size
- created_at

说明：

- 同一任务下相同 `file_type + file_name` 的结果文件重复上报时，应按幂等覆盖处理；
- 数据库层应通过唯一约束 `uk_task_file_type_name(task_id, file_type, file_name)` 防止重复插入；
- `file_name` 是面向前端展示和幂等去重的普通文件名，不得包含路径分隔符或控制字符，且长度不得超过 255；当节点收集到 `output/` 下的嵌套结果文件时，应把相对目录信息编码进展示名，必要时使用稳定截断加短哈希避免超长；
- `storage_path` 是后端内部定位物理文件的路径，必须位于当前任务 `output/` 目录下并真实存在，`file_name` 不要求等于 `storage_path` 的路径末段。

- seq_no

- created_at

说明：

- 若系统实现偏简化，也可采用“日志文件 + 偏移读取”方案；
- 若需要支持更直观的实时日志轮询，则建议保留日志分片表。

## 18.12 节点表 `compute_node`

- id
- node_code
- node_name
- node_token
- host
- port
- status
- enabled
- max_concurrency
- running_count
- reserved_count
- cpu_usage
- memory_usage
- last_heartbeat_time
- created_at
- updated_at

说明：

- `status` 只表示节点运行态在线/离线；
- `enabled` 表示管理员是否允许该节点参与调度；
- `running_count` 表示节点心跳上报的真实运行任务数；
- `reserved_count` 表示调度服务在“已选中但尚未真正运行”期间持有的预占数；
- `node_reservation` 表示 `task_id + node_id` 粒度的预占明细与幂等状态，`RESERVED` 表示当前仍占用预占，`RELEASED` 表示该轮预占已释放；
- `node_reservation` 不承担完整调度历史审计，多轮调度过程仍以 `schedule_record` 与 `task_status_history` 为准；
- `host + port` 共同构成 scheduler-service 调用 node-agent 的访问地址，不能只保留其一。

## 18.13 节点求解器能力表 `node_solver_capability`

- id
- node_id

- solver_version
- enabled
- created_at

说明：

- 节点注册时上报本机事实能力；
- 上报的 `solver_id` 必须能关联到全局求解器定义，否则不应生成能力记录；
- 管理员通过 `enabled` 控制该节点是否允许以此求解器参与调度；
- 节点重新注册时应更新事实能力信息，但不应直接覆盖管理员启用状态。

- schedule_status

- created_at
