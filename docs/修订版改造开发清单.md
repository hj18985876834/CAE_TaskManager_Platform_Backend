# 修订版改造开发清单

## 1. 文档目的

本文档用于在以下文档与当前代码实现之间做差异收敛，并形成可直接执行的开发清单：

- `docs/任务全流程设计与改造方案（修订版）.md`
- `需求分析与系统设计.md`
- `接口设计.md`
- `数据库设计.md`
- `后端设计.md`

使用规则：

1. 本清单只记录“当前仍需继续开发或收口”的事项；
2. 每完成一项，必须同步更新本清单状态；
3. 状态约定：
   - `TODO`：尚未开发
   - `DOING`：正在开发
   - `DONE`：已完成并至少通过一次编译验证
   - `BLOCKED`：存在外部依赖或需额外决策

---

## 2. 当前判断

当前项目已经完成修订版方案的大部分主链路改造，已具备：

- 任务创建、上传、校验、提交、调度、派发、执行、回传的主闭环
- `reservedCount` 容量模型
- `dispatch-failed` 可恢复/不可恢复分流
- 重试复用原 `taskId`
- `queueReason` 与 `canRetry` 查询层贯通

但按修订版方案和相关设计文档逐项对照，当前仍有若干细节未完全对齐，需要继续收口。

## 4. 每项工作说明

### P0-01 贯通任务查询返回 `canCancel`

- 差异来源：
  - `接口设计.md` 已要求任务列表/详情返回 `canCancel`
  - 当前代码只返回 `canRetry`、`queueReason`
- 目标：
  - `TaskListItemResponse`
  - `TaskDetailResponse`
  - `TaskAssembler`
  三处统一补齐 `canCancel`
- 验收：
  - 编译通过
  - 列表/详情返回结构包含 `canCancel`

### P0-02 收紧 `mark-finished` 终态约束

- 差异来源：
  - 设计要求 `mark-finished.finalStatus` 只允许 `SUCCESS/TIMEOUT/CANCELED`
  - 当前实现未显式排除其他值
- 目标：
  - 请求层校验
  - 业务层兜底校验
- 验收：
  - 编译通过
  - 非法 `finalStatus` 会被拒绝

### P0-03 补齐节点选择最后比较条件

- 差异来源：
  - 修订版方案要求节点选择顺序最后包含“最近心跳最新”
  - 当前策略未比较 `lastHeartbeatTime`
- 目标：
  - 在不破坏现有最小负载策略的前提下补足最后一层排序
- 验收：
  - 编译通过
  - 策略比较器包含 `lastHeartbeatTime`

### P0-04 调度前校验求解器/模板启用状态

- 差异来源：
  - 文档要求调度需同时满足“全局求解器启用、模板启用”
  - 当前调度只过滤节点与能力，不校验求解器/模板实时启用状态
- 目标：
  - `scheduler-service` 接入 `solver-service` 的最小查询能力
  - 调度前校验 solver/profile 是否启用
- 验收：
  - 编译通过
  - 求解器或模板被禁用后，排队任务不会继续被调度派发

### P0-05 节点求解器返回补充 `solverName`

- 差异来源：
  - 接口设计要求节点求解器列表含 `solverName`
  - 当前只返回 `solverId/solverVersion/enabled`
- 目标：
  - 补齐返回字段
  - 如有必要引入 `scheduler-service -> solver-service` 查询
- 验收：
  - 编译通过
  - 节点详情/节点求解器列表返回 `solverName`

### P0-06 为本轮关键改造补最小回归测试或验收脚本

- 差异来源：
  - 当前仓库缺少与本轮改造直接绑定的自动化回归
- 目标：
  - 至少补一组最小化验证
- 验收：
  - 有可重复执行的验证入口
