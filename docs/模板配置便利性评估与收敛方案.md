# 模板配置便利性评估与收敛方案

## 1. 文档目的

本文档用于回答当前项目中“模板驱动设计是否便于真实使用”这一问题，并给出一版与现有设计文档保持一致的收敛方案。

目标不是推翻当前“多求解器、模板驱动、可扩展”的总体设计，而是在该设计基线下，把模板配置复杂度降到更适合真实项目维护、扩展和演示的水平。

---

## 2. 当前判断

一句话结论：

当前项目的数据库结构和总体方向基本合理，但模板配置层的复杂度已经开始高于首版项目真实使用所需，主要表现为“模板中承载了过多运行细节”，这会降低后续新增模板和维护模板的便利性。

更具体地说：

- 当前表结构本身没有明显问题，`solver_definition`、`solver_task_profile`、`solver_profile_file_rule` 的三层划分仍然是合理的；
- 问题主要不在“表多”，而在“单个模板配置项承担的职责过重”；
- 当前实现已经出现把部分平台共性逻辑写入 `command_template` 和 `rule_json` 的趋势；
- 这会让模板从“描述差异的配置”逐渐变成“嵌入式脚本”，不利于真实使用。

---

## 3. 与核心设计文档的对照结论

### 3.1 与《需求分析与系统设计》的一致部分

当前实现与设计文档一致的部分：

- 采用“模板驱动 + 适配器模式”的总体思路；
- 支持多求解器、多模板；
- 模板负责描述上传要求、输入规则、命令模板、解析器、超时等；
- 首版上传模式统一为 `ZIP_ONLY`；
- 任务在校验阶段解压到 `workdir`，节点执行优先复用已校验目录。

### 3.2 当前实现偏重的部分

设计文档强调的是：

- 模板负责描述差异；
- 适配器负责 `validateInput / prepareWorkspace / buildCommand / execute / collectLogs / parseResult` 等共性执行逻辑。

而当前实现中，以下职责有一部分被压进了模板配置：

- 通过长 `command_template` 处理目录复制、结果收集、退出码包装等执行细节；
- 通过 `solver_profile_file_rule.rule_json` 中的 `deriveParam` 承担较复杂的参数派生；
- 通过单一模板兼容多种 OpenFOAM application，导致单模板规则持续膨胀。

因此当前问题不是“违背设计方向”，而是“实现时把模板层做得比设计文档要求更重”。

---

## 4. 当前复杂度的主要来源

### 4.1 `command_template` 过长

当前典型模板示例：

- OpenFOAM 模板不仅描述求解命令；
- 还包含 `mkdir -p`、`cp -r`、`exit $code` 等流程控制和结果收集动作。

这会带来三个问题：

- 新增模板时需要写较长 shell；
- 模板维护者必须理解节点执行细节；
- 平台共性行为分散到多个模板后，后续修改成本很高。

### 4.2 `rule_json` 承担了过多“智能逻辑”

当前 `solver_profile_file_rule.rule_json` 不仅用于简单约束：

- 后缀限制；
- 大小限制；
- 数量限制；

还承载了：

- `deriveParam`
- `deriveParams`
- 正则预处理
- 动态变量提取

这会带来两个直接问题：

- 规则虽然灵活，但录入和排错门槛高；
- 模板维护容易退化成“写 JSON 逻辑程序”。

### 4.3 单模板兼容过多场景

以 OpenFOAM 为例，当前模板已经开始尝试兼容多个 application。

这会让模板配置同时承担：

- 求解器选择；
- 输入差异判断；
- 参数派生；
- 命令生成；
- 结果收集。

对于首版项目来说，这种做法抽象能力很强，但真实使用并不友好。

---

## 5. 收敛原则

后续模板配置应遵循以下原则：

### 5.1 模板只描述差异，不描述整套流程

模板层负责：

- 这个模板属于哪个求解器；
- 支持什么任务类型；
- 需要什么输入文件；
- 需要什么用户参数；
- 最终执行哪条主命令；
- 使用哪个结果解析器；
- 默认超时时间。

模板层不宜继续承担：

- 平台级目录准备；
- 通用结果复制；
- 通用 shell 包装；
- 复杂流程控制。

### 5.2 共性逻辑回到代码

适合回到代码或适配器层的共性逻辑包括：

- `workdir / input / output / log` 目录准备；
- 通用执行前检查；
- 通用执行后结果收集；
- 不同求解器的默认命令构建策略；
- 从固定输入文件中提取运行命令或关键变量。

### 5.3 首版优先“易配置”，不优先“极致抽象”

首版项目应优先保证：

- 新增一个模板不需要写很长的 shell；
- 调整文件规则不需要理解复杂 `rule_json`；
- 出问题时能快速定位是模板错、文件错还是节点执行错。

因此，应避免为了兼容极端情况，把日常模板配置设计得过重。

---

## 6. 推荐的模板配置边界

## 6.1 `solver_task_profile` 建议保留的职责

建议继续保留以下字段和用途：

- `solver_id`
- `profile_code`
- `task_type`
- `profile_name`
- `upload_mode`
- `command_template`
- `params_schema_json`
- `parser_name`
- `timeout_seconds`
- `enabled`
- `description`

但应收紧 `command_template` 的使用边界：

- 它应主要描述“主执行命令”；
- 可以使用少量占位符；
- 不建议继续承载大段 shell 流程。

推荐的占位符边界：

- `${solverExecPath}`
- `${taskDir}`
- `${inputDir}`
- `${outputDir}`
- `${logDir}`
- 用户参数占位符

不推荐在模板中长期保留的行为：

- `mkdir -p ...`
- `cp -r ...`
- `exit $code`
- 多段串联 shell 流程

## 6.2 `solver_profile_file_rule` 建议保留的职责

规则表建议继续用于：

- 必需文件或目录声明；
- 路径模式；
- 文件类型；
- 是否必需；
- 排序；
- 简单约束 JSON。

推荐保留的 `rule_json` 能力：

- `allowSuffix`
- `minCount`
- `maxCount`
- `maxSizeMb`

推荐谨慎使用的能力：

- `solverCodes`
- `openfoamApplications`

不建议在首版继续扩张的能力：

- 更复杂的 `deriveParam` 链式逻辑；
- 大量预处理和正则拼装；
- 让规则层承担过多“运行命令推导”职责。

---

## 7. 对当前项目最合适的收敛方案

## 7.1 短期方案：不改表，先改使用约定

这是当前最稳妥的方案。

短期不建议立即修改数据库结构，而是先统一模板使用规范：

- 继续保留现有三张模板相关表；
- 继续保留 `ZIP_ONLY`；
- 继续保留文件规则表和参数 schema；
- 但在新增模板时，强制采用“轻模板”写法。

轻模板写法要求：

- `command_template` 只写主命令；
- 文件规则只写必要的输入检查；
- 能用代码适配器处理的共性逻辑，不再塞进模板。

## 7.2 中期方案：把求解器共性逻辑回收到适配器

结合设计文档中的“适配器模式”，建议中期把以下能力回收到代码：

- OpenFOAM 的 application 提取逻辑；
- OpenFOAM/CalculiX 的默认命令拼装逻辑；
- 求解结束后的默认结果收集逻辑；
- 求解器特有的输入准备逻辑。

这样模板层只需要保留：

- 这是哪个求解器的哪个任务模板；
- 需要哪些文件；
- 需要哪些参数；
- 调用哪个解析器。

## 7.3 长期方案：模板管理体验优化

如果后续继续扩展模板管理，推荐增加：

- 模板复制功能；
- 模板预设；
- 文件规则表单化编辑；
- 规则 JSON 的可视化表单；
- 模板测试/预览功能。

这部分属于“体验增强”，不是当前主流程的阻断项。

---

## 8. 对当前 OpenFOAM 模板的具体建议

当前 OpenFOAM 模板之所以显得复杂，核心原因是：

- 同时做了 application 提取；
- 同时做了执行命令拼装；
- 同时做了结果复制。

建议的收敛方式是：

### 8.1 保留

- `system/controlDict`
- `system/fvSchemes`
- `system/fvSolution`
- 不同任务类型下的关键输入文件规则

### 8.2 下沉到适配器或节点代码

- 从 `controlDict` 提取 application；
- 根据 application 生成最终执行命令；
- 求解后将标准结果目录归集到 `output/`；
- 平台统一的执行包装逻辑。

### 8.3 模板层最终希望保留的形式

理想状态下，OpenFOAM 模板应更接近：

- 模板定义任务类型和文件规则；
- 模板声明“使用 OpenFOAM 通用执行策略”；
- 适配器根据输入文件自动生成最终命令。

而不是把一大段 shell 直接放在 `command_template` 中。

---

## 9. 对当前 CalculiX 模板的判断

CalculiX 当前模板相对更接近合理状态：

- 主命令较短；
- 文件规则简单；
- 配置负担较轻。

这说明当前系统并不是“数据库模板设计必然复杂”，而是：

- 某些求解器模板已经开始承担了过多平台共性逻辑；
- 应该把这些共性逻辑重新收回代码层。

---

## 10. 结论与后续建议

最终结论：

- 当前数据库结构可以保留；
- 当前模板驱动方向可以保留；
- 当前“模板配置越来越复杂”的担心是合理的；
- 问题主要在于模板职责过重，而不是表设计本身错误；
- 后续应坚持“模板描述差异，适配器承担共性”的收敛方向。

建议的实际推进顺序：

1. 先把当前模板使用规范收紧为“轻模板”；
2. 再把 OpenFOAM 这类求解器中的共性流程逐步回收到适配器或节点代码；
3. 最后再视项目时间决定是否补模板管理体验优化。

---

## 11. 对当前开发的约束性结论

从本文件开始，后续与模板相关的开发应遵循以下约束：

- 不新增会明显提高模板手工维护难度的数据库配置能力；
- 不把平台共性 shell 流程继续塞进 `command_template`；
- 不让 `rule_json` 继续膨胀成复杂逻辑脚本；
- 所有模板相关改动都应同时对照《需求分析与系统设计》《接口设计》《数据库设计》；
- 若实现便利性与当前实现方式冲突，应优先回到设计基线重新收敛，而不是继续堆配置复杂度。
