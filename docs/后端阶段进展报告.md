# 后端阶段进展报告

## 1. 目前毕业论文（设计）工作进展及取得的成果

结合《需求分析与系统设计》《后端设计》《接口设计》《开发文档》以及当前后端代码实现情况来看，本课题后端部分已经完成了从“方案设计”向“核心原型实现”的实质性推进，整体进展处于“主链路基本打通、工程化细节持续完善”的阶段。课题以船舶 CAE 仿真任务管理与分布式执行为应用背景，目标并不在于求解算法本身，而在于围绕仿真任务全生命周期建立统一的管理、调度与执行平台。围绕这一目标，当前后端已经较好地完成了系统总体架构设计、微服务模块拆分、数据库模型设计、核心接口定义以及关键业务流程实现等工作。

首先，在系统架构层面，后端已经按照设计文档完成了较清晰的微服务划分，形成了 `gateway-service`、`user-service`、`solver-service`、`task-service`、`scheduler-service`、`node-agent` 以及公共模块 `common-lib` 的整体结构。各服务职责边界与需求分析中的功能划分基本一致：网关服务负责统一入口与基础鉴权，用户服务负责登录认证与用户管理，求解器服务负责求解器定义、模板及文件规则管理，任务服务负责仿真任务全生命周期管理，调度服务负责节点管理与任务调度，节点代理负责在计算节点上接收任务并执行。这样的拆分方式较好体现了微服务架构“高内聚、低耦合、便于扩展”的设计思想，也符合课题背景中“通过现代软件架构改进传统单机和人工调度模式”的研究目标。

其次，在数据模型与接口设计方面，后端已经完成了较完整的数据库初始化脚本和主要业务表结构设计，能够支撑多用户、多任务、多节点运行场景。当前已形成求解器定义、任务模板、文件规则、仿真任务、任务状态历史、任务日志、任务结果、计算节点、节点求解器能力、调度记录等核心实体模型。其中，任务、节点、调度三者之间的关联关系已经能够表达“任务入队、节点选择、任务派发、节点回传”的完整业务闭环。接口方面，设计文档中所要求的外部 REST 接口和内部服务调用接口也已基本落地，特别是 `task-service` 与 `scheduler-service`、`scheduler-service` 与 `node-agent`、`node-agent` 与 `task-service` 之间的内部接口链路已经形成，说明系统不仅完成了静态设计，而且具备了真实联调和验证的基础。

再次，在核心业务实现层面，目前完成度最高的是任务管理、任务调度与节点执行三条核心链路。`task-service` 已经实现了任务创建、参数修改、输入压缩包上传、任务校验、任务提交、取消、管理员重试、任务分页查询、详情查询、状态历史查询、日志查询、结果查询等能力，较完整地覆盖了需求分析文档中对“任务全过程管理”的要求。特别是针对 CAE 仿真任务输入复杂、文件结构要求严格的特点，系统已经实现了 ZIP 压缩包上传、解压、目录穿越防护、模板规则匹配、文件后缀与数量校验等逻辑，这一成果能够较好体现课题与普通任务管理系统的差异化特点。

调度模块方面，`scheduler-service` 已实现节点注册、节点心跳、节点状态维护、节点求解器能力管理、可用节点筛选、定时轮询调度、调度记录写入以及节点离线检查等能力。当前调度策略以“先进先出 + 最小负载”为主，虽然还属于基础版策略，但已经能够满足原型平台阶段对“多计算节点协同执行”和“集中式任务派发”的验证需求。与此同时，节点代理 `node-agent` 已实现启动注册、周期性心跳、接收调度任务、准备工作目录、复制输入文件、选择执行器、启动本地进程、读取日志、回传状态与结果、处理取消请求等功能，初步形成了“中心服务统一调度 + 节点本地执行 + 执行结果回传”的分布式执行闭环。这一部分是本课题后端最具有代表性的成果之一，因为它直接体现了计算节点从“静态资源”转变为“可注册、可调度、可监控的执行主体”。

另外，在课题要求强调的“多求解器接入与容器化思路”方面，后端也取得了阶段性成果。目前 `solver-service` 已经完成求解器定义、模板管理、执行元数据管理、上传规范生成等功能，并支持以 OpenFOAM、CalculiX、Mock 执行器为代表的多求解器场景。虽然当前执行侧仍以本地进程调用为主，容器运行时能力尚未完全做实，但系统已经在模型层和服务边界上为后续扩展预留了明确位置。与此同时，项目中已经提供了 `compose.yaml`、服务构建脚本和节点部署脚本，说明系统具备了基础容器化部署条件，为答辩演示和进一步完善提供了良好基础。

综合来看，后端部分已经不再停留在概念设计阶段，而是完成了“面向船舶 CAE 仿真场景的微服务后端原型平台”的核心实现。尤其是在任务管理、调度联动、节点执行、日志结果回传等方面，已较好支撑论文中关于“微服务架构、分布式调度、节点协同执行”的技术目标表达，也为后续测试验证、材料整理和论文撰写提供了充分基础。

## 2. 存在的问题及拟解决措施

虽然当前后端已经形成了较完整的原型闭环，但距离“稳定可演示、口径完全统一、可直接用于答辩展示”的最终状态仍存在一些问题。第一，当前系统虽已采用微服务拆分，但服务治理能力仍偏基础，部分服务调用仍依赖显式地址配置，尚未形成生产级的服务发现、重试与熔断机制。第二，调度策略目前仍以先进先出和最小负载为主，任务优先级、失败重试、重调度、多调度实例去重等高级能力尚未完善。第三，节点执行端当前以本地进程方式为主，容器执行模式、更加严格的执行隔离和更强的结果收集规则仍需增强。第四，文件存储和结果管理当前以本地文件系统为主，在共享存储、一致性和部署灵活性方面仍有提升空间。第五，设计文档、SQL 脚本和部分实现细节之间还存在少量口径不完全一致的问题，需要进一步收口。

针对上述问题，拟采取以下措施：一是继续以“保证主链路稳定”为优先目标，围绕任务创建、上传、校验、提交、调度、执行、回传开展端到端联调和缺陷修复；二是对关键设计点进行统一核对，及时修正数据库字段约束、接口返回口径和状态流转规则；三是在不显著增加系统复杂度的前提下，补充优先级调度、调度失败补偿和多节点演示能力；四是进一步完善容器化部署脚本和环境配置，确保答辩时系统能够稳定启动、稳定演示；五是同步完善开发文档和模块分析文档，使论文中的描述与实际后端实现保持一致。

## 3. 下一步工作计划

下一步工作将围绕“系统收口、功能验证、材料成型”三个方向展开。首先，在功能完善方面，将继续对现有后端主链路进行联调优化，重点检查任务上传与校验、任务状态流转、调度记录、节点执行、日志回传、结果回传等关键流程，确保至少能够稳定完成一次完整的“登录 -> 创建任务 -> 上传输入 -> 校验 -> 提交 -> 调度 -> 节点执行 -> 查看日志与结果”的演示过程。同时，结合当前开发文档中已经识别出的短板，优先补齐任务优先级真正参与调度、节点异常场景处理、调度失败后的状态说明与记录一致性等问题，以增强系统完整性与可解释性。

其次，在部署与验证方面，将继续完善 `Docker Compose` 环境与节点代理部署方案，至少形成一套适用于演示环境的稳定部署方式。对于 Ubuntu 虚拟机上的 `node-agent` 节点，将重点验证节点注册、心跳上报、任务派发、求解器执行和结果回传的端到端过程；对于本地中心服务环境，将验证多服务协同运行的稳定性。若时间允许，还将进一步补充一到两个典型烟雾测试场景，形成更清晰的验证记录，为论文中的“系统测试与结果分析”章节提供素材。

再次，在论文材料与文档整理方面，将根据当前已实现的后端结构和功能，持续完善《需求分析与系统设计》《后端设计》《接口设计》《开发文档》以及各微服务模块分析文档，使论文中的系统设计、实现说明和测试分析能够与现有代码完全对应。在论文写作上，将重点突出三条主线：一是面向船舶 CAE 场景的任务管理需求；二是采用微服务架构对系统进行解耦设计；三是通过节点注册、心跳监控、任务调度与结果回传实现分布式执行原型。最终目标不是继续无限扩展功能边界，而是在现有基础上收敛出一个“核心功能完整、技术路线清晰、可运行可演示、可支撑答辩讲解”的毕业设计最终版本。
