# 任务全流程设计与改造方案

## 1. 文档目的

本文档面向当前后端项目中“任务提交到计算完成”的完整链路，结合以下内容进行统一梳理：

- 现有设计文档：
  - `需求分析与系统设计.md`
  - `后端设计.md`
  - `数据库设计.md`
- 当前代码实现：
  - `task-service`
  - `scheduler-service`
  - `solver-service`
  - `node-agent`

文档目标不是重复接口说明，而是明确：

1. 当前实现的真实口径是什么。
2. 哪些地方已经符合设计。
3. 哪些地方与“多求解器、可扩展、分布式调度、严格状态管理”的设计思想还不一致。
4. 后续应按什么顺序改造，才能把任务主链路收敛到稳定、可演示、可继续扩展的状态。

阅读约定：

- 第 5、6 章用于说明当前实现现状及遗留问题，属于 **As-Is 分析**；
- 第 7、8 章用于给出建议收敛后的目标流程，属于 **To-Be 基线**；
- 若与其他设计文档产生理解差异，开发与联调应优先以第 7、8 章的目标口径为准。

---

## 2. 总体结论

当前项目的任务主链路已经具备基本闭环：

1. 用户创建任务。
2. 上传 ZIP 压缩包。
3. 按模板规则解压并校验。
4. 校验通过后提交入队。
5. 调度器轮询队列并选择节点。
6. 节点代理接收任务并执行命令模板。
7. 节点回传运行状态、日志、结果摘要和结果文件。
8. 任务进入成功、失败、取消或超时等终态。

但从设计要求看，当前还存在几个关键问题：

1. 调度与状态流转的一致性还不够严格。
2. 节点容量统计混用了“调度预占”和“节点实时报数”。
3. 多实例调度场景下缺乏真正原子化的抢占与占位机制。
4. 状态机允许过多“跳转捷径”，容易导致竞态。
5. 求解器和模板的启用开关还没有在任务提交、调度、执行全链路严格贯通。

因此，当前最合理的目标不是继续堆新功能，而是把任务主链路按设计文档收口为一套严格、可解释、可扩展的流程。

## 3. 任务链路的职责分工

### 3.1 `solver-service`

职责：提供“任务如何校验、如何执行”的平台定义。

它管理的是：

- 求解器定义
- 求解器任务模板
- 文件规则
- 上传规则
- 执行模板
- 结果解析方式

它不负责：

- 保存用户任务实例
- 调度排队
- 节点实际执行

### 3.2 `task-service`

职责：作为任务生命周期中心。

它管理的是：

- 任务创建、修改、上传
- 任务文件与工作目录
- 模板化校验
- 入队、取消、重试
- 状态机与状态历史
- 日志、结果摘要、结果文件

它应当是任务状态的唯一真实来源。

### 3.3 `scheduler-service`

职责：作为调度中心。

它管理的是：

- 节点注册与心跳
- 节点能力
- 队列任务选点
- 节点占位
- 调度记录
- 节点离线处理

它不应直接决定任务业务语义，只负责“是否有合适节点、把任务发给谁”。

### 3.4 `node-agent`

职责：作为节点执行器。

它负责：

- 节点注册
- 节点心跳
- 接收调度下发任务
- 准备目录
- 复用已校验工作目录
- 执行命令模板
- 回传状态、日志和结果

它不应负责平台级调度决策，也不应成为任务状态的最终裁判。

## 5. 当前任务全流程现状

## 5.1 创建任务

用户创建任务时，`TaskLifecycleManager.createTask()` 会写入 `sim_task`，初始状态为 `CREATED`。

当前口径正确：

- 状态初始值明确
- 记录状态历史
- 任务实例与求解器/模板是解耦的

这部分基本符合设计。

## 5.2 上传任务压缩包

当前上传由 `TaskLifecycleManager.uploadTaskFile()` 负责。

现状：

- 当前只支持 `ZIP_ONLY`
- 上传限制来自 `solver-service` 提供的上传规则元数据
- 文件会写入任务输入目录
- 同一个归档文件再次上传时会替换旧文件，并清理旧校验产物

这部分已经开始符合“模板驱动而不是硬编码某一求解器”的设计思想。

## 5.3 校验

当前校验由 `TaskValidationManager.validateTask()` 完成。

现状：

1. 读取模板文件规则。
2. 找到归档文件。
3. 解压 ZIP 到任务 `workdir`。
4. 根据压缩包结构自动推导真实 `caseDir`。
5. 依据文件规则、路径规则、后缀规则、数量规则进行校验。
6. 从文件内容或文件名推导派生参数。
7. 校验通过后将任务转为 `VALIDATED`。
8. 将归档对应的 `unpackDir` 落库，供节点执行时复用。

这部分是当前任务链路中最符合设计思想的一段，因为它已经做到：

- 用模板规则定义校验，而不是把 OpenFOAM、CalculiX 等逻辑直接写死在程序里。
- 在任务校验阶段就准备好工作目录，为节点复用提供基础。

结论：校验阶段应继续坚持“模板 + 规则 + 派生参数”的方向，不应回退到按某一种求解器硬编码文件名。

## 5.4 提交入队

当前提交由 `TaskLifecycleManager.submitTask()` 完成。

现状：

- 只有 `VALIDATED` 任务允许提交
- 提交后转为 `QUEUED`
- 设置 `submitTime`
- 调用 `SchedulerClient.notifyTaskSubmitted(taskId)`

问题：

- `notifyTaskSubmitted()` 目前是空实现，调度实际上完全依赖轮询

影响：

- 功能上仍然可用
- 但不具备“提交后即时唤起调度”的能力

## 5.5 队列读取与排序

当前调度器从 `task-service` 拉取 `/internal/tasks/queued`。

`TaskMapper.selectByStatus()` 的当前排序为：

1. `priority DESC`
2. `COALESCE(submit_time, created_at) ASC`
3. `id ASC`

这部分是合理的，已经基本符合“优先级优先，其次先来先服务”的设计口径。

## 5.6 调度选点

当前调度流程位于：

- `TaskScheduleJob`
- `ScheduleAppService`
- `FcfsLeastLoadStrategy`

说明：

- `FcfsLeastLoadStrategy` 是当前实现中的策略类名；
- 对外接口、调度记录和跨文档统一表述时，建议使用稳定策略标识 `FCFS_MIN_LOAD`。

当前流程为：

1. 调度器轮询取出 `QUEUED` 任务列表
2. 根据求解器能力过滤在线节点
3. 按 `runningCount -> cpuUsage -> memoryUsage` 选择最小负载节点
4. 直接把该节点的 `runningCount + 1`
5. 调用 `task-service` 标记任务为 `SCHEDULED`
6. 调用 `node-agent` 下发任务
7. 成功后再标记为 `DISPATCHED`

这部分的主要问题有三个：

1. 节点占位不是原子的。
2. `runningCount` 被同时当成“调度预占”和“节点真实运行数”。
3. 如果任务在被取出后状态已经变化，调度器仍可能继续尝试下发。

## 5.7 节点执行

节点执行流程位于：

- `node-agent/.../TaskDispatchManager.java`
- `node-agent/.../TaskExecuteManager.java`
- `node-agent/.../InputFilePrepareService.java`

当前流程为：

1. 节点判断本地并发是否已满。
2. 注册运行态。
3. 回传“接受任务”。
4. 准备目录。
5. 准备输入文件。
6. 回传 `RUNNING`。
7. 选择执行器。
8. 执行模板命令。
9. 回传成功或失败。

当前一个重要改进已经形成：

- 如果 `task-service` 在校验阶段已经解压并保存了 `unpackDir`，节点会优先复用这个目录作为 `taskDir`

这与“统一目录合同、校验阶段准备工作目录、节点执行阶段复用”的思路是一致的。

## 5.8 结果回传

当前由 `TaskReportManager` 和 `TaskResultManager` 配合完成。

现状：

- 运行中可按块上报日志
- 可上报结果摘要
- 可上报结果文件
- 最终上报 `SUCCESS` / `FAILED`

问题：

- 终态处理前仍保留了从 `SCHEDULED` 自动补跳到 `DISPATCHED` 的逻辑
- 这使状态机看起来“能兜底”，但也削弱了流程的严格性

## 6. 当前实现中最主要的问题

## 6.1 状态机不够严格

当前 `TaskStatusRule` 允许：

- `DISPATCHED -> SUCCESS`
- `DISPATCHED -> FAILED`
- `DISPATCHED -> TIMEOUT`
- `DISPATCHED -> CANCELED`

这意味着节点即使没有显式回传 `RUNNING`，任务也能直接进入终态。

另外：

- `TaskLifecycleManager.reportStatus()`
- `TaskResultManager.finishTask()`

还保留了“如果当前还是 `SCHEDULED`，先自动补到 `DISPATCHED` 再继续”的逻辑。

问题本质：

- 这会让状态流转表面上显得顺滑，但实际上隐藏了分发确认丢失、运行状态丢失、接口调用顺序错乱等问题。

## 6.2 调度占位与节点实时报数混用

当前 `ScheduleAppService.scheduleTask()` 会先把 `ComputeNode.runningCount + 1`。

但同时：

- 节点心跳 `NodeAppService.heartbeat()` 又会用节点上报的 `runningCount` 覆盖数据库。
- 节点执行完成后，`TaskReportAppService.updateRunningCount(-1)` 又会再走一条减计数路径。

这会导致：

1. 预占被心跳覆盖掉。
2. 完成回报和心跳重复修改同一字段。
3. 节点负载可能偏大也可能偏小。

## 6.3 调度流程缺少真正的抢占确认

当前流程是：

1. 调度器先查 `QUEUED`
2. 在本地选任务
3. 再调用 `markScheduled()`

虽然 `markScheduled()` 内部会锁任务，但如果任务已经不是 `QUEUED`，当前实现很多时候只是返回，不会让整个调度循环感知“这个任务抢占失败”。

风险：

- 多个调度实例可能同时尝试调度同一个任务
- 任务取消、重试、管理员操作可能与调度产生竞态

## 6.4 调度失败后的处理过于简单

当前 `TaskScheduleJob` 中：

- 只要任务已标记 `SCHEDULED` 且派发未成功，就可能直接调用 `markTaskFailed()`

这会带来一个问题：

- 节点临时拒绝、网络闪断、瞬时繁忙，这类“可恢复错误”会直接把任务打成 `FAILED`

更合理的口径应是区分：

1. 可恢复派发失败：回退 `QUEUED`
2. 不可恢复派发失败：转 `FAILED`

## 6.5 求解器/模板启用开关尚未全链路生效

设计上应同时满足：

1. 全局求解器启用
2. 模板启用
3. 节点启用
4. 节点求解器能力启用

当前代码中：

- 节点与节点求解器能力启用参与了调度
- 但任务校验、提交、调度全过程对求解器/模板全局启用状态还未做足够强校验

## 7. 推荐的目标流程

## 7.1 标准状态机

推荐收敛为以下主流程：

1. `CREATED`
2. `VALIDATED`
3. `QUEUED`
4. `SCHEDULED`
5. `DISPATCHED`
6. `RUNNING`
7. `SUCCESS | FAILED | TIMEOUT | CANCELED`

推荐规则：

- `CREATED -> VALIDATED`
- `VALIDATED -> CREATED`
- `VALIDATED -> QUEUED`
- `QUEUED -> SCHEDULED`
- `QUEUED -> CANCELED`
- `SCHEDULED -> DISPATCHED`
- `SCHEDULED -> QUEUED`
- `SCHEDULED -> FAILED`
- `DISPATCHED -> RUNNING`
- `DISPATCHED -> QUEUED`
- `DISPATCHED -> FAILED`
- `RUNNING -> SUCCESS`
- `RUNNING -> FAILED`
- `RUNNING -> TIMEOUT`
- `FAILED -> QUEUED`
- `TIMEOUT -> QUEUED`

其中要点是：

1. 首版执行结果类终态固定只允许从 `RUNNING` 进入。
2. `SCHEDULED -> FAILED`、`DISPATCHED -> FAILED` 仅用于调度/派发阶段的不可恢复失败，必须记录明确 `failType` 与原因，不能当作节点执行结果上报使用。
3. `SCHEDULED` 和 `DISPATCHED` 应增加“回退队列”能力，用于可恢复失败。
4. 首版不要求实现 `RUNNING -> CANCELED`；若第二阶段引入运行中终止，再补充该状态流转与 node-agent 取消链路。
5. 去掉节点结果上报时的“自动补跳多级状态”。

## 7.2 文件与目录合同

推荐统一目录合同如下：

1. 用户上传归档文件保存在任务输入目录。
2. 任务校验阶段解压到任务 `workdir`。
3. `task_file.unpack_dir` 记录真实执行目录。
4. 节点收到任务后优先复用 `unpack_dir`。
5. 节点不再对已校验归档重复做平台级解压。

这样做的好处：

1. 校验与执行使用同一份目录结构。
2. 节点执行不会因为重复解压造成路径漂移。
3. 多求解器模板只需要关心“真实工作目录”，不需要关心 ZIP 原始层级。

## 7.3 调度队列策略

推荐队列排序保持为：

1. `priority DESC`
2. `submit_time ASC`
3. `id ASC`

解释：

- 优先级高的任务先调度
- 同优先级下先提交先调度
- `id` 作为最终稳定排序键

## 7.4 节点选点策略

推荐节点选择依据：

1. 可调度节点集合
2. 预估总占用最小
3. 实际运行数最小
4. CPU 使用率最小
5. 内存使用率最小
6. 最近心跳最新

对外统一建议：

- 调度策略类名可保留 `FcfsLeastLoadStrategy`；
- 但接口参数、调度记录表 `strategy_name`、测试记录和运维日志中，统一使用 `FCFS_MIN_LOAD` 作为策略标识。

其中“可调度节点集合”应同时满足：

1. 节点 `ONLINE`
2. 节点 `enabled = 1`
3. 存在对应 `node_solver_capability`
4. 该能力 `enabled = 1`
5. 全局求解器启用
6. 模板启用
7. `running_count + reserved_count < max_concurrency`

## 7.5 节点容量模型

推荐新增 `reserved_count` 字段，职责拆分为：

- `running_count`：节点通过心跳上报的真实运行数
- `reserved_count`：调度器在“已选中但节点未真正开始运行”期间持有的预占数

这样：

- 调度器不再直接修改 `running_count`
- 心跳也不会覆盖掉调度预占
- 可调度判断使用 `running_count + reserved_count`

## 7.6 调度原子性

推荐新增“调度抢占确认”接口或方法。该动作发生在调度器已经选出目标节点并完成节点容量预占之后，其语义是：

1. 只有当前状态仍为 `QUEUED` 的任务，才允许被一个调度器实例成功抢占
2. 抢占成功后一次性写入：
   - `node_id`
   - `status = SCHEDULED`
   - 调度历史
3. 如果抢占失败，说明任务已被取消、重试或被其他调度实例处理，调度器必须释放刚才的节点预占

理想上应使用数据库条件更新或行锁实现。

## 7.7 节点执行流程

推荐节点执行按以下顺序固定：

1. 节点校验本地并发是否允许接受
2. 节点登记本地接收/准备态
3. 回传“任务已接收”，中心任务状态进入 `DISPATCHED`
4. 准备目录
5. 复用已校验 `workdir`
6. 真正启动求解进程后回传 `RUNNING`
7. 根据模板构造命令
8. 执行命令
9. 实时回传日志
10. 上报结果摘要与结果文件
11. 回传终态

这里要强调：

- 节点只执行模板，不负责理解某个求解器具体业务语义
- 不允许因为支持 OpenFOAM 就把 OpenFOAM 文件要求写死在节点代理中

## 8. 推荐的端到端任务流程

## 8.1 创建与上传阶段

1. 用户选择求解器与模板，创建任务，状态为 `CREATED`
2. 用户上传 ZIP 压缩包
3. 平台记录归档文件信息

## 8.2 校验阶段

1. 读取模板文件规则
2. 校验求解器与模板是否启用
3. 解压 ZIP 至任务 `workdir`
4. 推导真实工作目录 `caseDir`
5. 执行模板规则校验
6. 派生参数并写回任务参数
7. 归档文件记录 `unpackDir`
8. 状态转为 `VALIDATED`

## 8.3 提交入队阶段

1. 仅允许 `VALIDATED` 任务提交
2. 提交后状态转为 `QUEUED`
3. 刷新 `submitTime`
4. 可选地发送调度唤醒事件

## 8.4 调度阶段

1. 调度器轮询或接收唤醒信号
2. 按优先级降序、提交时间升序读取 `QUEUED` 任务
3. 查询满足求解器能力、启用状态与容量约束的候选节点
4. 按 `FCFS_MIN_LOAD` 策略选出目标节点
5. 对目标节点原子增加 `reserved_count`
6. 调用任务服务原子确认任务仍为 `QUEUED`，并绑定 `node_id`、转为 `SCHEDULED`
7. 如果任务确认失败，释放目标节点预占并继续处理下一条任务
8. 写入调度记录

## 8.5 下发阶段

1. 调度器调用目标节点代理接口
2. 节点若接受，则任务进入 `DISPATCHED`
3. 节点若临时拒绝，则释放节点预占并回到 `QUEUED`
4. 节点若明确不可执行，则按规则置为 `FAILED`

## 8.6 执行阶段

1. 节点通过 task-service 回传 `RUNNING`
2. task-service 成功完成 `DISPATCHED -> RUNNING` 后，调用 scheduler-service 释放对应 `reserved_count`
3. 节点真实运行数体现在后续心跳 `running_count`
4. 节点持续回传日志
5. 节点完成后回传结果与终态

## 8.7 结束阶段

1. 若成功，任务进入 `SUCCESS`
2. 若运行报错，进入 `FAILED`
3. 若超过超时阈值，进入 `TIMEOUT`
4. 若用户在排队阶段取消，进入 `CANCELED`

## 9. 与当前设计思想的一致性判断

## 9.1 已经符合设计思想的部分

1. 任务上传已收敛到模板驱动的 `ZIP_ONLY`
2. 校验阶段已采用模板文件规则而非按单一求解器写死
3. 解压目录与真实执行目录的识别已初步统一
4. 节点能力注册采用“节点上报事实能力、平台落表、管理员控制启用”的两层责任模型
5. 调度队列已经具备“优先级 + FCFS”基本顺序

## 9.2 仍不完全符合设计思想的部分

1. 状态机仍然偏宽松，不能完全体现严格生命周期
2. 调度容量模型仍不适合多实例和高并发
3. 调度失败策略过于粗糙，恢复性失败没有回队机制
4. 求解器/模板启用状态没有全链路硬约束
5. 节点运行数量同步机制不够清晰

结论：

当前几次改动方向总体是对的，没有把系统改成“只支持 OpenFOAM 的硬编码实现”；但要真正达到设计文档中的“多求解器、可扩展、分布式调度”的目标，还需要进一步把调度一致性和状态机严格化。

## 10. 推荐改造顺序

## P0：先把主链路收口

1. 收紧 `TaskStatusRule`
2. 移除节点结果上报中的多级补跳逻辑
3. 明确 `SCHEDULED` / `DISPATCHED` 的回退规则
4. 统一目录合同：校验解压，节点复用

## P1：收敛调度一致性

1. 为任务增加原子调度抢占
2. 为节点增加 `reserved_count`
3. 把调度容量判断改为 `running_count + reserved_count < max_concurrency`
4. 取消调度器直接修改 `running_count`

## P2：增强恢复能力

1. 区分“可恢复派发失败”和“不可恢复派发失败”
2. 增加重新入队策略
3. 补充分发失败、节点拒绝、节点离线的状态与审计说明

## P3：补全扩展能力

1. 打通提交后的即时调度唤醒
2. 完整校验求解器启用、模板启用、节点能力启用
3. 为后续非 `ZIP_ONLY` 上传方式预留扩展点

## 12. 最终建议

从项目阶段看，当前最重要的不是新增更多业务功能，而是把“任务主链路”真正打磨成一条稳定、严格、可解释的链路。

如果后续按本文档推进，建议以以下目标作为验收标准：

1. 至少 1 条 OpenFOAM 任务能够稳定走完整条链路。
2. 至少 1 条非 OpenFOAM 模板任务能够复用相同链路。
3. 任意任务状态变化都能解释“为什么这样转”。
4. 节点容量统计在心跳、调度、完成三个场景下保持一致。
5. 代码中不出现把某一种求解器规则硬编码到平台通用流程里的实现。

达到这一步后，系统才算真正接近设计文档所定义的目标架构。
