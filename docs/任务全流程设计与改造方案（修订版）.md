# 任务全流程设计与改造方案（修订版）

## 1. 文档目的

本文档基于以下材料重新收敛任务主链路改造方案：

- `需求分析与系统设计.md`
- `后端设计.md`
- `数据库设计.md`
- `接口设计.md`
- 当前后端真实实现
- 前一版《任务全流程设计与改造方案》的评估与复盘结论

本修订版的目标不是追求“更理想的架构”，而是给出一套：

1. 与当前项目架构边界一致；
2. 能解决现有核心问题；
3. 不会明显破坏现有成功链路；
4. 可分阶段落地并便于联调验证；
5. 仍然符合“多求解器、模板驱动、分布式调度”设计思想

的改造方案。

一句话概括：

> 本方案采用“保守收口、最小闭环、逐步增强”的原则，对当前任务链路进行一致性重构，而不是进行推倒重来的架构升级。

---

## 2. 总体判断

## 2.1 方案是否合理

合理。

因为当前项目已经具备以下基础：

1. `solver-service` 已承担求解器、模板、规则定义职责；
2. `task-service` 已承担任务生命周期中心职责；
3. `scheduler-service` 已承担节点注册、能力管理、调度职责；
4. `node-agent` 已承担节点执行职责；
5. 校验阶段已支持 ZIP 解压、规则校验和 `workdir` 准备；
6. 调度阶段已具备优先级 + 提交时间排序基础；
7. 节点执行阶段已基本支持复用校验产物。

因此，本次改造不是改变系统主架构，而是补齐“状态一致性、调度一致性、容量模型一致性”。

## 2.2 当前方案要避免的误区

虽然总体方向正确，但不能采用过于理想化的改法，必须避免：

1. 把状态机收得过死，导致真实错误场景无处表达；
2. 为了调度一致性引入超出当前项目复杂度的中间件；
3. 把 `task-service`、`scheduler-service`、`node-agent` 的职责再次搅乱；
4. 一次性扩大改造范围，导致原本能跑通的主链路失稳。

---

## 3. 修订原则

本修订版按以下原则设计。

## 3.1 保持现有服务边界不变

### `solver-service`

继续负责：

- 求解器定义
- 模板定义
- 文件规则
- 上传规则
- 执行模板
- 结果解析配置

不负责：

- 任务实例状态
- 调度排队
- 节点运行态

### `task-service`

继续负责：

- 任务创建、上传、校验、提交、取消、重试
- 状态机与状态历史
- 任务文件、结果、日志

它仍然是任务状态的唯一真实来源。

### `scheduler-service`

继续负责：

- 节点注册、心跳、能力管理
- 调度选点
- 节点容量占位
- 调度记录
- 节点离线补偿

它负责“谁来执行”，不负责“任务业务状态如何解释”。

### `node-agent`

继续负责：

- 接受下发任务
- 准备本地目录
- 执行模板命令
- 回传状态、日志和结果

它是执行器，不是状态裁判者。

## 3.2 保持模板驱动，不为单一求解器硬编码

无论是 OpenFOAM、CalculiX 还是后续其他求解器，平台级通用流程都应基于：

- 上传规则
- 文件规则
- 命令模板
- 结果模板

工作，而不是在任务主链路中写死某个求解器文件要求。

## 3.3 保持 ZIP_ONLY 现阶段边界

当前上传方式只做 `ZIP_ONLY`，这是可接受的工程边界。

本次改造重点不是扩展上传方式，而是保证：

1. ZIP 校验逻辑一致；
2. ZIP 解压目录合同一致；
3. 节点执行与校验产物一致。

## 3.4 优先消除不一致，不优先追求更复杂能力

本次改造优先解决：

1. 状态竞态
2. 节点容量统计混用
3. 调度抢占不明确
4. 派发失败处理过粗

暂不优先解决：

1. MQ 事件化
2. 自动重试策略体系
3. 多维资源调度
4. 多调度实例完整高可用编排

## 4. 当前项目的真实问题

## 4.1 状态机过宽，且带有自动补跳

当前问题主要有两类：

1. `DISPATCHED` 可以直接进入多个终态；
2. 节点上报或结果回写时，服务端会在某些情况下自动补跳到 `DISPATCHED`。

风险：

- 隐藏状态缺口；
- 隐藏接口调用顺序错误；
- 增加调度和回传的竞态复杂度。

## 4.2 `runningCount` 同时承担两种职责

当前 `runningCount` 同时被用于：

1. 调度时预占节点容量；
2. 节点心跳上报真实运行数；
3. 执行结束后再手工减计数。

这是当前最核心的不一致来源。

## 4.3 调度对任务的抢占确认不够明确

当前调度流程本质上是：

1. 先从 `QUEUED` 列表取任务；
2. 本地选节点；
3. 再调用 `markScheduled`。

虽然已经有行锁保护，但对“是否抢占成功”的反馈语义还不够强，导致：

- 多实例调度场景下语义不够清晰；
- 状态变化后调度器不一定能及时感知“本次抢占已失败”。

## 4.4 派发失败全部打成失败过于粗糙

现在很多“节点临时拒收、网络闪断、瞬时繁忙”的场景，都可能最终表现成：

- `SCHEDULED -> FAILED`

这会降低系统可恢复性，也与调度本身的瞬时性不完全匹配。

## 4.5 节点执行前准备失败属于真实场景

这里必须特别强调。

当前 `node-agent` 的执行顺序大致为：

1. 接收任务
2. 准备目录
3. 准备输入文件
4. 回传 `RUNNING`
5. 真正启动执行器

这意味着：

- 节点已接受任务，但在真正进入 `RUNNING` 之前，本地准备阶段就可能失败；
- 因而 `DISPATCHED -> FAILED` 这种状态路径在当前项目中不是错误设计，而是必须保留的现实场景表达。

因此，状态机不能被收得过死。

## 5. 修订后的目标模型

## 5.1 主状态集合

主状态仍保持为：

1. `CREATED`
2. `VALIDATED`
3. `QUEUED`
4. `SCHEDULED`
5. `DISPATCHED`
6. `RUNNING`
7. `SUCCESS`
8. `FAILED`
9. `CANCELED`
10. `TIMEOUT`

不新增新的主状态，保证与当前数据库、前端和文档总体兼容。

## 5.2 修订后的状态流转

### 保留的基础流转

- `CREATED -> VALIDATED`
- `VALIDATED -> CREATED`
- `VALIDATED -> QUEUED`
- `QUEUED -> SCHEDULED`
- `SCHEDULED -> DISPATCHED`
- `DISPATCHED -> RUNNING`
- `RUNNING -> SUCCESS`
- `RUNNING -> FAILED`
- `RUNNING -> TIMEOUT`
- `QUEUED -> CANCELED`
- `RUNNING -> CANCELED`
- `FAILED -> QUEUED`
- `TIMEOUT -> QUEUED`

### 修订后增加的恢复性流转

- `SCHEDULED -> QUEUED`
- `DISPATCHED -> QUEUED`

它们用于表达：

1. 节点临时不可用；
2. 派发网络失败；
3. 节点拒收但允许重新调度；
4. 节点已接收但尚未真正开始运行，且允许重新入队。

### 必须保留的失败流转

- `SCHEDULED -> FAILED`
- `DISPATCHED -> FAILED`

原因：

1. 存在不可恢复的调度失败；
2. 存在节点已接受任务但本地准备阶段失败的真实场景；
3. 强行去掉这些路径，会让错误场景失去准确表达。

## 5.3 状态机收紧原则

本次改造不追求“只有 `RUNNING` 才能进入终态”的绝对规则，而采用更适配当前项目的规则：

1. 成功类终态原则上应从 `RUNNING` 进入；
2. 运行期失败、超时、取消应从 `RUNNING` 进入；
3. 调度期和派发期允许少量必要失败路径；
4. 节点回传不应长期依赖跨级自动补跳；
5. 若任务尚未真正开始运行，应优先回队而不是直接失败。

## 6. 修订后的目录与文件合同

## 6.1 基本合同

统一目录合同如下：

1. 用户上传 ZIP 文件，保存到任务输入目录；
2. `task-service` 在校验阶段解压到任务 `workdir`；
3. 根据 ZIP 结构推导真实 `caseDir`；
4. 在 `task_file.unpack_dir` 中记录真实执行目录；
5. `node-agent` 执行阶段优先复用 `unpack_dir`；
6. 节点不再重复做平台级解压。

术语约定补充如下：

- `workdir` 表示任务工作目录层级，是校验阶段解压与整理后的工作区；
- `caseDir` 表示最终实际执行目录，可能等于 `workdir`，也可能是 `workdir` 下某个符合模板规则的子目录；
- `task_file.unpack_dir` 存储的应是最终可直接执行的真实目录，通常对应 `caseDir`，供 `node-agent` 直接复用。

## 6.2 这样设计的原因

1. 校验与执行使用同一套目录结构；
2. 能避免“节点解压后路径和校验时路径不一致”的问题；
3. 模板只需要关心真实工作目录，不需要关心压缩包层级细节；
4. 更符合当前项目已经跑通的实现方向。

## 6.3 非目标

本次不引入：

- 多上传模式并行支持
- 在线解压策略切换
- 节点二次重组输入结构

这些都属于后续扩展，不是当前主链路收口的重点。

## 7. 修订后的调度模型

## 7.1 调度排序

队列排序保持为：

1. `priority DESC`
2. `submit_time ASC`
3. `id ASC`

这与当前实现和设计文档一致，不需要大改。

## 7.2 节点选择原则

可调度节点必须同时满足：

1. 节点 `ONLINE`
2. 节点 `enabled = 1`
3. 全局求解器启用
4. 模板启用
5. 存在对应 `node_solver_capability`
6. 该能力 `enabled = 1`
7. 节点剩余容量充足

选择顺序建议为：

1. 最小总占用
2. 最小真实运行数
3. 最低 CPU 使用率
4. 最低内存使用率
5. 最近心跳最新

## 7.3 节点容量模型

本次改造最关键的结构性调整是引入：

- `running_count`
- `reserved_count`

其中：

### `running_count`

表示节点通过心跳上报的真实运行任务数。

### `reserved_count`

表示调度服务在以下阶段持有的预占数：

- 节点已被选中
- 任务已准备调度或已完成派发
- 但节点尚未由心跳反映真实运行数

### 可调度判断

统一表达为：

`running_count + reserved_count < max_concurrency`

## 7.4 为什么只做任务数预占

当前项目更适合只引入“任务数级别”的预占，而不扩展到：

- CPU 核数预占
- 内存容量预占
- 求解器独占锁
- 复杂资源池

原因是这些能力超出了当前系统复杂度边界。

## 8. 修订后的调度与派发流程

## 8.1 任务提交阶段

1. 用户创建任务，状态为 `CREATED`
2. 上传 ZIP 文件
3. 校验通过后进入 `VALIDATED`
4. 用户提交任务
5. 任务进入 `QUEUED`
6. 写入新的 `submitTime`

## 8.2 调度阶段

1. 调度器轮询待调度任务
2. 按优先级和提交时间排序读取任务
3. 读取可调度节点集合
4. 选择目标节点
5. 为目标节点增加 `reserved_count`
6. 原子确认任务当前仍处于 `QUEUED`
7. 若确认成功，任务转为 `SCHEDULED`
8. 写入调度记录

## 8.3 派发阶段

1. 调度器调用 `node-agent` 下发任务
2. 若节点接受：
   - 任务转为 `DISPATCHED`
3. 若节点临时拒绝或网络失败，且判断为可恢复：
   - 释放 `reserved_count`
   - 任务回退到 `QUEUED`
   - 写入调度失败记录
4. 若节点明确不可执行或错误不可恢复：
   - 释放 `reserved_count`
   - 任务转为 `FAILED`
   - 写入失败原因

## 8.4 节点执行阶段

节点执行采用以下顺序：

1. 接收任务
2. 准备工作目录
3. 复用校验阶段 `unpack_dir`
4. 准备输入文件
5. 回传 `RUNNING`
6. 真正启动执行器
7. 持续回传日志
8. 回传结果摘要与结果文件
9. 回传最终状态

## 8.5 `reserved_count` 的释放时机

建议统一在以下情况释放：

1. 派发失败且回队；
2. 派发失败且失败终止；
3. 节点已成功进入 `RUNNING`；
4. 调度确认失败、需要回滚；
5. 节点离线补偿时统一清理。

这里必须强调：

`reserved_count` 的引入必须伴随“明确释放时机”，否则会造成节点长期假满载。

## 9. 修订后的接口语义

## 9.1 `markScheduled`

对应正式内部接口：`POST /internal/tasks/{taskId}/mark-scheduled`

该接口不应只是“把任务状态从 `QUEUED` 改成 `SCHEDULED`”，而应具备：

- 原子抢占调度权

的语义。

也就是说：

1. 只有当前状态仍为 `QUEUED` 时才允许成功；
2. 若任务已被其他调度实例抢占或状态已变化，应明确返回失败或未抢占；
3. 不应继续静默派发。

## 9.2 `dispatch-failed`

该接口应支持两类结果：

1. 可恢复失败：
   - 回退到 `QUEUED`
2. 不可恢复失败：
   - 进入 `FAILED`

不应再把所有派发失败都直接归为不可恢复失败。

## 9.3 节点运行数量接口

不建议继续保留：

- `/running-count +1/-1`

这种接口语义。

更适合当前项目的方式是：

1. 节点通过心跳上报真实 `runningCount`
2. 调度服务通过内部逻辑或接口管理 `reservedCount`

两者职责分开。

## 9.4 节点终态回传约束

节点终态回传前，原则上应已完成：

- `RUNNING` 状态回传

但考虑当前项目真实执行顺序，仍允许：

- `DISPATCHED -> FAILED`

用于表达执行前准备失败。

不建议保留：

- 大量依赖服务端自动补跳来掩盖链路不一致。

## 10. 与当前项目的适配性判断

## 10.1 适配的部分

本方案与当前项目适配良好的原因在于：

1. 不改变服务划分；
2. 不新增主状态；
3. 不改变模板驱动和 ZIP_ONLY 主体方向；
4. 只增强调度与状态一致性；
5. 与当前前端主状态展示兼容；
6. 与当前节点注册与能力管理方式兼容。

## 10.2 需要保守推进的部分

以下内容必须保守处理：

1. 状态机不能绝对化；
2. 不引入 MQ 或复杂分布式锁；
3. 不把 `node-agent` 变成状态管理中心；
4. 不在这次改造中同时展开非 ZIP_ONLY 上传扩展；
5. 不扩大为多维资源调度系统。

## 10.3 如果机械照搬旧方案会出现的问题

如果直接采用旧版偏理想化方案，可能导致：

1. `DISPATCHED -> FAILED` 场景被错误删除；
2. 节点本地准备失败无法准确表达；
3. 改造范围过大，影响当前已跑通链路；
4. 文档、代码、前端再次失配。

因此，本修订版才是更适合当前项目的开发基线。

## 11. 实施边界

## 11.1 本次必须做

1. 确认实际数据库与文档一致，尤其是 `compute_node.reserved_count`、`task_file.unpack_dir`、相关索引和初始化数据；
2. 收紧状态机，但保留 `SCHEDULED -> QUEUED`、`DISPATCHED -> QUEUED`、`SCHEDULED -> FAILED`、`DISPATCHED -> FAILED` 等必要路径；
3. 调度采用“容量预占 + 原子抢占确认 + 派发 + 派发结果回写”的流程；
4. 可恢复派发失败支持回退到 `QUEUED`，不可恢复派发失败进入 `FAILED`；
5. 去掉 `runningCount` 的双重职责，节点真实运行数只由心跳上报，调度预占只由 `reservedCount` 表达；
6. 统一释放 `reserved_count` 的时机，包括抢占失败、派发失败、进入 `RUNNING`、节点离线等场景；
7. 清理或隔离旧任务状态入口，避免绕过 `TaskDispatchManager` 的原子状态确认逻辑；
8. 统一文档、数据库、接口、后端实现和测试用例的语义。

## 11.2 本次建议做

1. 前端节点详情展示 `reservedCount`；
2. 调度页更清楚展示调度失败原因；
3. 任务详情页展示“回队”类调度记录；
4. 增加任务全流程后端单元测试和必要的端到端烟测脚本；
5. 准备一套 OpenFOAM 演示模板与 ZIP_ONLY 示例压缩包，用于验证通用模板执行链路。

## 11.3 本次暂缓

1. 提交即触发调度的事件化改造；
2. 自动重试策略；
3. 多维资源调度；
4. 非 ZIP_ONLY 上传；
5. 多调度实例高可用治理；
6. 求解器结果解析插件体系深度扩展。

## 12. 分阶段改造计划

## Phase 0：开发前置检查与环境收口

涉及：

- 数据库初始化脚本
- 本地 Maven 构建环境
- Ubuntu 节点代理运行环境
- 文档基线

9. [已完成] 确认派发失败、抢占失败、节点离线时都会释放或清零对应预占；
10. [已完成] 确认旧的 `/running-count +1/-1` 路径不再承担调度预占职责，仅作为兼容空实现或逐步移除。

## 13. 最终结论

本修订版方案的核心立场是：

1. 当前项目不需要重做架构；
2. 当前项目确实需要重构任务主链路的一致性；
3. 最合理的方向是“保守收口”，而不是“理想升级”；
4. 改造重点应放在状态机、调度抢占、容量预占、派发失败分流；
5. 只要严格控制边界，这次改造不会与现有项目打架，反而能显著提升稳定性和解释性。

因此，后续开发应以本文档作为任务主链路改造的正式基线。
