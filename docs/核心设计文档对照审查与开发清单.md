## 3. 本轮审查范围

本轮重点复核以下四类一致性问题：

1. 数据库表结构、字段命名、字段语义是否与设计文档一致
2. 任务状态流转是否与“状态流转实现表”和“状态流转校验规则表”一致
3. 接口是否同步、是否缺少、是否存在超范围行为
4. 多求解器、模板驱动、`ZIP_ONLY` 上传与节点执行合同是否保持一致

## 4. 总体结论

当前项目已经完成了任务主链路的大部分改造，以下方向与修订后的设计文档已经基本一致：

- 首版上传模式统一为 `ZIP_ONLY`
- 校验阶段负责解压到 `workdir`，并写入 `task_file.unpack_dir`
- 节点代理优先复用后端校验后的 `unpackDir`
- 查询层已支持 `queueReason`
- 任务详情已聚合 `statusHistory` 和 `scheduleRecords`
- 调度容量判断已采用 `runningCount + reservedCount < maxConcurrency`
- 调度策略主标识统一为 `FCFS_MIN_LOAD`
- 模板驱动执行链路已支持 `${taskDir}`、`${workDir}`、`${outputDir}` 等变量
- OpenFOAM 已按模板和文件规则接入，主流程未对某一种求解器硬编码

但当前仍有一批关键偏差尚未收口，主要集中在：

- 首版状态机仍偏宽
- 节点预占释放尚未做到 `taskId + nodeId` 级幂等
- 节点回传接口仍有超出首版合同的行为
- 节点离线补偿逻辑与设计文档不完全一致
- 部分数据库约束尚不足以支撑设计文档要求的幂等语义
- 个别设计文档示例仍残留旧口径

## 6. 二次审查结论

### 6.1 数据库与字段一致性结论

`[需收口]` 当前数据库主表和字段命名总体与设计文档一致，但以下几点仍需处理：

1. 若严格落实“`POST /internal/nodes/{nodeId}/reserve` / `release-reservation` 按 `taskId + nodeId` 幂等”的设计要求，仅依赖 `compute_node.reserved_count` 不足以可靠支撑幂等语义。
2. 当前库表中没有“节点预占记录”类结构，无法可靠区分：
   - 是否已经为某个 `taskId + nodeId` 预占过
   - 是否已经为某个 `taskId + nodeId` 释放过
3. `task_result_file` 当前只有普通索引，没有天然支撑 `taskId + fileType + fileName` 幂等的唯一约束。
4. `数据库设计.md` 中 OpenFOAM 初始化模板示例仍残留旧命令模板，和当前项目实际动态模板口径不一致。

结论：

- 数据库主干表设计没有方向性问题
- 但若要真正完成调度幂等和结果文件幂等，仍需补数据库设计或补仓储级幂等逻辑

### 6.2 状态流转一致性结论

`[存在偏差]` 当前实现与新增状态流转表相比，主要有以下不一致：

1. 首版本应只允许 `QUEUED -> CANCELED`，但当前代码仍允许：
   - `DISPATCHED -> CANCELED`
   - `RUNNING -> CANCELED`
2. 首版 `status-report` 应固定只负责 `DISPATCHED -> RUNNING`，但当前实现仍保留“通用状态回写”能力。
3. 首版 `mark-finished` 应只允许 `SUCCESS / TIMEOUT`，当前实现仍允许 `CANCELED`。
4. 首版不实现运行中取消，但 node-agent 当前仍会回写 `CANCELED`。
5. 节点离线补偿按新设计应分流：
   - `SCHEDULED / DISPATCHED` 优先走 `dispatch-failed`
   - `RUNNING` 才统一补偿为 `FAILED(NODE_OFFLINE)`
   当前实现仍是统一按失败收口。

结论：

- 当前状态机主链路可运行
- 但还没有完全收口到“首版严格状态机”

### 6.3 接口同步性结论

`[基本完整，局部不一致]` 当前接口数量基本完整，但仍有以下关键问题：

1. `dispatch-failed` 当前请求体没有显式携带 `nodeId`，不利于严格校验“回写节点必须等于任务当前绑定节点”。
2. 节点预占接口虽然已有 `taskId` 请求体，但当前服务端逻辑并未实际使用。
3. `mark-dispatched` 当前实现允许重新绑定 `nodeId`，与文档“必须校验和 `mark-scheduled` 绑定节点一致”的要求不完全一致。
4. node-agent 当前仍主动回写 `DISPATCHED`，而文档约定该状态应由 scheduler-service 经 `mark-dispatched` 统一写入。
5. 单任务调度记录查询当前按倒序返回，和接口设计中“便于渲染时间线的升序返回”不一致。

结论：

- 接口没有出现“主流程缺失”的大洞
- 但仍有一批接口行为超出或偏离新设计合同

### 6.4 文档同步性结论

`[仍需同步]` 四份核心设计文档总体已经向修订版方案靠拢，但还存在少量示例级残留旧口径。

当前最明确的问题：

- [数据库设计.md](../数据库设计.md) 中 OpenFOAM 初始化命令示例仍是旧的 `simpleFoam -case ${taskDir}`

结论：

- 当前更大的问题已经不是“文档没有写”
- 而是“少量细节口径尚未完全同步到所有文档示例”

## 9. 下一步建议顺序

推荐按以下顺序推进：

1. 先做 `A-01`、`A-02`、`A-03`、`A-04`、`A-05`、`A-13`
2. 再做 `A-06`、`A-07`
3. 然后做 `A-08`、`A-09`
4. 最后收口 `A-10`、`A-11`、`A-12`

原因：

- 第一批是“状态机和接口合同收口”，优先级最高
- `A-13` 虽然是字段合同问题，但它直接影响节点注册与调度派发地址闭环，应与第一批一起处理
- 第二批是“调度幂等和数据库支撑”，改动较大但价值最高
- 第三批是“异常场景与幂等补全”
- 第四批是“展示一致性和文档最终收口”
