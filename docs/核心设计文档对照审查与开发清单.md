## 4. 总体结论

当前项目已经完成了任务主链路的大部分收口，以下关键方向已与最新设计基本一致：

- 首版上传模式统一为 `ZIP_ONLY`
- 校验阶段负责解压到 `workdir`，节点执行优先复用已校验 `workdir`
- `compute_node.port` 已打通数据库、实体、注册、查询与调度派发链路
- 调度容量模型已统一为 `runningCount + reservedCount < maxConcurrency`
- `node_reservation` 已正式落库，说明项目已转入“预占明细 + 汇总计数”模式
- 任务查询层已支持 `queueReason`
- 任务详情已聚合 `task_status_history` 与 `schedule_record`
- 首版状态机已完成第一轮收紧：`status-report` 仅负责 `DISPATCHED -> RUNNING`，`mark-finished` 仅允许 `SUCCESS / TIMEOUT`，node-agent 不再主动回写 `DISPATCHED / CANCELED`

但和最新设计相比，当前仍有 5 个关键偏差尚未收口：

1. `reserve / release-reservation` 内部接口返回契约和冲突语义仍是旧口径
2. 节点离线补偿仍是“一刀切失败”，未按 `SCHEDULED/DISPATCHED` 与 `RUNNING` 分流
3. 日志与结果文件上报的幂等性仍未真正落地
4. 单任务调度记录查询顺序仍为倒序，和时间线设计不一致
5. 节点管理查询接口仍混用 `id` 与 `nodeId`
6. `dispatch-failed` 仍在 `task-service` 内部主动释放预占，和最新职责边界不一致
7. 调度链路当前“先标记 `DISPATCHED`、再真正下发节点”，时序语义偏前置
8. 节点日志/结果上报接口缺少状态阶段校验
9. 存在设计外重复暴露的管理员任务列表接口 `/api/tasks/admin`
10. 旧版 `running-count` 内部接口仍对外暴露
11. 个别核心设计文档示例与汇总段仍残留旧口径，需要同步修订
12. 管理员调整优先级接口当前放宽到了 `SCHEDULED / DISPATCHED / RUNNING`，超出最新状态约束
13. `user-service`、`solver-service`、`profile-service` 仍保留设计文档外的 `PUT .../status` 兼容接口
14. `task_file` “同一任务仅保留一份有效输入压缩包”的约束当前仅靠应用层最佳努力，缺少事务与数据库级兜底
15. 创建任务接口尚未在写入前校验“求解器 / 模板 / 任务类型”三者合法性
16. `scheduler-service` 仍保留设计文档外的旧版节点注册/心跳内部入口 `/internal/scheduler/nodes/register|heartbeat`
17. `task-service` 仍依赖设计文档外的内部辅助接口 `/internal/tasks/{taskId}/status` 与 `/internal/tasks/node-offline/fail`
18. 模板参数结构 `paramsSchema/paramsSchemaJson` 虽已建模，但任务创建/校验阶段尚未真正执行参数 schema 校验
19. 任务文件校验 `issues[].errorCode` 体系与接口设计文档未统一，当前实现仍会回传文档外错误码
20. `mark-scheduled` 内部接口返回契约仍为布尔值，未按文档返回 `claimed/taskId/nodeId/status`
21. `status-report` 在携带 `fromStatus` 时的幂等处理仍不完整，重复 `DISPATCHED -> RUNNING` 回报可能被误判为冲突
22. 任务文件、结果文件、调度记录等响应字段仍混用 `id` 与文档命名的 `fileId/scheduleId`

结论：当前后端已具备继续开发的清晰基线，但还不能认定为“与最新设计完全一致”。

## 6. 关键偏差审查记录

说明：

- 本节保留本轮审查中识别出的关键偏差记录，便于追踪改造背景；
- 是否仍未收口，以第 8 节“最新正式开发清单”的状态为准。

### 6.1 偏差一：`reserve / release-reservation` 接口契约未按新设计收口

当前实现：

- `InternalNodeReservationController` 仍返回 `Result<Void>`
- `NodeAppService.releaseReservation(...)` 在“记录不存在”或“已释放”时直接静默返回
- `ScheduleAppService.releaseNodeReservation(...)` 也是静默返回

最新设计要求：

- `POST /internal/nodes/{nodeId}/reserve` 返回：
  - `taskId`
  - `nodeId`
  - `reservationStatus`
  - `reservedCount`
- `POST /internal/nodes/{nodeId}/release-reservation` 返回：
  - `taskId`
  - `nodeId`
  - `reservationStatus`
  - `reservedCount`
- 若释放时不存在对应 `node_reservation` 记录，应返回明确冲突或失败，不能静默成功
- 若已为 `RELEASED`，应按幂等成功返回当前状态与计数

影响：

- 当前代码虽然已经有 `node_reservation` 表，但内部接口契约仍停留在旧版“只调用、不反馈”的模式
- 这会导致前后服务无法严格按文档约定观察预占状态，也不利于后续联调和问题定位

涉及文件：

- `scheduler-service/src/main/java/com/example/cae/scheduler/interfaces/internal/InternalNodeReservationController.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/application/service/NodeAppService.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/application/service/ScheduleAppService.java`
- `接口设计.md`

结论：

- 这是当前新增 `node_reservation` 后最明显的接口-实现偏差，优先级应上调到 `P0`。

当前状态：

- `2026-04-22` 已完成：`reserve` / `release-reservation` 已返回 `taskId/nodeId/reservationStatus/reservedCount`，`release` 对“无记录”返回明确冲突，对“已释放”按幂等成功返回当前状态。

### 6.2 偏差二：节点离线补偿仍未按新版状态规则分流

当前实现：

- `NodeHeartbeatChecker.markOfflineNodes()` 直接调用 `taskClient.markNodeOfflineTasksFailed(...)`
- `TaskDispatchManager.markNodeOfflineTasksFailed(...)` 会把 `SCHEDULED / DISPATCHED / RUNNING` 全部统一改成 `FAILED`
- 同时直接写入 `failType = NODE_OFFLINE`
- `NodeAppService.markNodeOffline(...)` 会把节点 `reservedCount` 直接清零

最新设计要求：

- `SCHEDULED / DISPATCHED` 阶段优先按派发失败处理
  - 可恢复则回退 `QUEUED`
  - 不可恢复才转 `FAILED`
- 若最终回到 `QUEUED`，`NODE_OFFLINE` 只写状态历史或调度记录，不写回 `sim_task.fail_type`
- `RUNNING` 阶段任务才统一收口为 `FAILED(NODE_OFFLINE)`
- 离线补偿过程应与预占释放和 `node_reservation` 状态保持一致

影响：

- 当前补偿逻辑会把本应可回队的任务直接打成失败
- 任务主记录中的失败字段会被过早写死
- 节点侧 `reservedCount` 与 `node_reservation` 的关系在离线场景下也未完全按设计收口

涉及文件：

- `scheduler-service/src/main/java/com/example/cae/scheduler/infrastructure/support/NodeHeartbeatChecker.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/infrastructure/client/impl/TaskClientStub.java`
- `task-service/src/main/java/com/example/cae/task/application/manager/TaskDispatchManager.java`
- `需求分析与系统设计.md`
- `任务状态流转实现表.md`
- `状态流转校验规则表.md`

结论：

- 这是当前功能正确性风险最高的一项，优先级应保持 `P0/P1` 之间的最高级别，建议紧接着预占接口契约之后处理。

当前状态：

- `2026-04-22` 已完成第一轮收口：节点离线时，`SCHEDULED/DISPATCHED` 已改为回退 `QUEUED`，`RUNNING` 才收口为 `FAILED(NODE_OFFLINE)`，并去除了节点离线时直接清零 `reservedCount` 的旧实现。

### 6.3 偏差三：日志与结果文件幂等性未真正落地

当前实现：

- `TaskLogChunkMapper.insert(...)` 为普通插入
- SQL 中 `task_log_chunk` 只有 `uk_task_seq(task_id, seq_no)`，重复上报会直接异常，而不是“幂等忽略或覆盖”
- `TaskResultSummaryRepositoryImpl.saveOrUpdate(...)` 已基本满足 `taskId` 维度幂等
- `TaskResultFileRepositoryImpl.save(...)` 为普通插入
- `task_result_file` 当前没有 `task_id + file_type + file_name` 唯一约束

最新设计要求：

- 日志按 `taskId + seqNo` 幂等
- 结果摘要按 `taskId` 幂等覆盖或忽略
- 结果文件按 `taskId + fileType + fileName` 幂等

影响：

- 节点重复重试上报时，日志和结果文件无法稳定按设计处理
- 一旦网络抖动、节点补传或重复回传，会出现插入异常或重复文件记录

涉及文件：

- `task-service/src/main/java/com/example/cae/task/infrastructure/persistence/mapper/TaskLogChunkMapper.java`
- `task-service/src/main/java/com/example/cae/task/infrastructure/persistence/repository/TaskLogRepositoryImpl.java`
- `task-service/src/main/java/com/example/cae/task/infrastructure/persistence/mapper/TaskResultFileMapper.java`
- `task-service/src/main/java/com/example/cae/task/infrastructure/persistence/repository/TaskResultFileRepositoryImpl.java`
- `docs/sql/init_databases.sql`
- `数据库设计.md`

结论：

- 结果摘要这条线已基本正确，剩下的问题主要集中在日志和结果文件。

当前状态：

- `2026-04-22` 已完成：`task_log_chunk` 已按 `taskId + seqNo` 使用数据库级 upsert，`task_result_summary` 已按 `taskId` 使用数据库级 upsert，`task_result_file` 已补齐 `uk_task_file_type_name(task_id, file_type, file_name)` 并改为数据库级 upsert，同时补充了现网可执行补丁脚本。

### 6.4 偏差四：单任务调度记录查询顺序与设计不一致

当前实现：

- `ScheduleRecordMapper.selectByTaskId(...)` 使用 `ORDER BY id DESC`

最新设计要求：

- 单任务调度记录应按时间线升序返回，便于前端直接渲染

影响：

- 前端任务详情页若直接按后端结果展示，调度时间线会倒序

涉及文件：

- `scheduler-service/src/main/java/com/example/cae/scheduler/infrastructure/persistence/mapper/ScheduleRecordMapper.java`
- `接口设计.md`

结论：

- 该问题不影响主流程正确性，但会影响展示合同，优先级为 `P1`。

当前状态：

- `2026-04-22` 已完成：`schedule_record` 按 `created_at ASC, id ASC` 返回单任务调度记录，已与接口设计中“时间线升序展示”的口径保持一致。

### 6.5 偏差五：节点管理查询接口仍混用 `id` 与 `nodeId`

当前实现：

- `NodeListItemResponse` 同时包含 `id` 与 `nodeId`
- `NodeDetailResponse` 同时包含 `id` 与 `nodeId`
- `NodeAssembler` 与 `NodeAppService` 都在同时设置这两个字段

最新设计要求：

- 节点管理对前端的 JSON 响应统一只使用 `nodeId`
- 不应在接口层对前端同时暴露 `id` 和 `nodeId`

影响：

- 前后端容易在节点主键字段上继续形成双口径
- 后续接口文档、前端页面和调试日志会出现“有时看 `id`，有时看 `nodeId`”的漂移

涉及文件：

- `scheduler-service/src/main/java/com/example/cae/scheduler/interfaces/response/NodeListItemResponse.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/interfaces/response/NodeDetailResponse.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/application/assembler/NodeAssembler.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/application/service/NodeAppService.java`
- `接口设计.md`

结论：

- 这是一个典型的“功能可用但响应口径不统一”问题，应纳入正式清单。

当前状态：

- `2026-04-22` 已完成：节点列表与节点详情响应已移除兼容 `id` 字段，对外统一仅返回 `nodeId`，与接口设计和路径语义保持一致。

### 6.6 偏差六：`dispatch-failed` 释放预占职责边界未完全收口

当前实现：

- `TaskDispatchManager.markFailed(...)` 在状态回写后会调用 `releaseReservationQuietly(...)`
- 同时 `TaskScheduleJob` 在派发失败路径上也会调用 `taskScheduleManager.releaseNodeReservation(...)`

最新设计要求：

- `dispatch-failed` 只负责任务状态回写
- 调度失败场景下的预占释放应由 `scheduler-service` 单独调用 `/internal/nodes/{nodeId}/release-reservation`
- `task-service` 不应在 `dispatch-failed` 内部再次承担预占释放职责

影响：

- 当前虽然借助 `node_reservation` 幂等可以避免重复减计数，但职责边界已经偏离最新设计
- 后续联调时容易误判“释放动作到底应该由谁发起”

涉及文件：

- `task-service/src/main/java/com/example/cae/task/application/manager/TaskDispatchManager.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/application/scheduler/TaskScheduleJob.java`
- `接口设计.md`
- `任务状态流转实现表.md`

结论：

- 当前虽然能工作，但调度冲突语义被压缩成单一布尔值
- 不利于后续日志审计、问题定位与内部接口统一

涉及文件：

- `task-service/src/main/java/com/example/cae/task/interfaces/internal/InternalTaskDispatchController.java`
- `task-service/src/main/java/com/example/cae/task/application/manager/TaskDispatchManager.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/infrastructure/client/impl/TaskClientStub.java`
- `接口设计.md`

结论：

- 这是本轮按“内部返回字段合同”复核后新增发现的问题，应作为 `P1/P2` 级接口收口项处理。

当前状态：

- `2026-04-22` 已完成：`mark-scheduled` 已改为返回结构化 `claimed/taskId/nodeId/status`，`scheduler-service` 已同步消费该合同，并通过 `mvn -pl scheduler-service,task-service -am -DskipTests compile`。

### 6.21 偏差二十一：`status-report` 在携带 `fromStatus` 时的幂等处理仍不完整

当前实现：

- `TaskLifecycleManager.reportStatus(...)` 先校验：
  - `request.fromStatus == task.currentStatus`
- 然后才判断“若当前已是 `RUNNING` 则幂等返回”
- 这意味着当节点或中间链路按文档重试：
  - `fromStatus = DISPATCHED`
  - `toStatus = RUNNING`
  但任务已在上一次请求中成功变为 `RUNNING` 时，请求会先命中 `TASK_STATUS_MISMATCH`

最新设计要求：

- 对同一 `taskId + nodeId` 的重复 `RUNNING` 回传应按幂等成功处理
- 文档允许 `fromStatus` 作为状态校验辅助信息存在，不应因此破坏已进入目标状态后的重试幂等

影响：

- 当前实现对“严格按文档回传 `fromStatus` 的节点重试”不够稳健
- 这类问题平时不一定暴露，但一旦出现网络重试或代理补发，就会造成不必要冲突

涉及文件：

- `task-service/src/main/java/com/example/cae/task/application/manager/TaskLifecycleManager.java`
- `task-service/src/main/java/com/example/cae/task/interfaces/request/StatusReportRequest.java`
- `接口设计.md`
- `状态流转校验规则表.md`

结论：

- 这是本轮按“状态回传幂等”维度复核后新增确认的问题，应进入正式清单。

当前状态：

- `2026-04-22` 已完成：`status-report` 已先处理“同节点重复回传 `RUNNING`”的幂等成功，再进行 `fromStatus` 严格比对，避免合法重试被误判为 `TASK_STATUS_MISMATCH`。

### 6.22 偏差二十二：多个响应 DTO 仍混用 `id` 与文档命名字段

当前实现：

- `TaskFileResponse` 只返回 `id`，未按接口示例使用 `fileId`
- `TaskResultFileResponse` 只返回 `id`，未按接口示例使用 `fileId`
- `TaskScheduleRecordResponse` 只返回 `id`，未按调度记录接口口径统一为 `scheduleId`
- `ScheduleRecordResponse` 虽同时返回 `id` 与 `scheduleId`，但任务详情聚合对象仍只有 `id`

最新设计要求：

- 接口文档当前已经在多个地方明确使用：
  - `fileId`
  - `scheduleId`
  - `nodeId`
  - `taskId`
- 前后端应收口到一套稳定命名，而不是在不同接口中混用“通用 id”和“业务 id”

影响：

- 会让前端类型定义、接口封装与文档说明继续分裂
- 这类问题不一定阻断流程，但会持续制造联调摩擦

涉及文件：

- `task-service/src/main/java/com/example/cae/task/interfaces/response/TaskFileResponse.java`
- `task-service/src/main/java/com/example/cae/task/interfaces/response/TaskResultFileResponse.java`
- `task-service/src/main/java/com/example/cae/task/interfaces/response/TaskScheduleRecordResponse.java`
- `scheduler-service/src/main/java/com/example/cae/scheduler/interfaces/response/ScheduleRecordResponse.java`

结论：

## 7. 非阻断性优化建议

以下问题当前不一定阻断功能，但建议后续整理：

### 7.1 预占逻辑存在重复实现

当前 `NodeAppService` 与 `ScheduleAppService` 都各自持有一套 `reserve/release` 逻辑，虽然功能上能工作，但和设计文档中建议的 `NodeCapacityManager` 收口方向不一致。

建议：

- 后续可抽取统一的节点容量管理组件
- 把预占、释放、幂等返回、冲突语义统一到一个入口

价值：

- 减少逻辑分叉
- 降低后续维护和联调时的语义漂移风险

### 7.2 网关仍保留未使用的旧路由匹配项

当前 `GatewayRouteConfig` 仍包含：

- `/api/scheduler/nodes/**`
- `/api/scheduler/records/**`

但当前后端并没有对应控制器，也不在最新接口文档中。

建议：

- 后续在清理遗留兼容路径时一并收口这些旧路由匹配

价值：

- 降低网关配置和接口设计文档之间的漂移
- 避免未来误以为系统仍保留这组路径
