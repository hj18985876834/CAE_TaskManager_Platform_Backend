# 当前项目下一阶段推进路线图

## 1. 文档目的

本文档用于在当前后端代码、已有设计文档和最近联调问题的基础上，对项目下一阶段的推进方向进行统一规划，作为后续开发、联调、部署演示和论文材料整理的共同参考。

本文档重点回答以下问题：

- 当前项目整体处于什么阶段
- 目前哪些能力已经基本成型
- 接下来最应该优先做什么
- 每一项工作的涉及服务、关键文件和验收标准是什么

## 2. 当前阶段判断

当前项目已经不再是“功能骨架搭建期”，而是进入了“核心原型闭环已形成，下一步重点是稳定主链路、收口设计口径、补足工程化能力”的阶段。

从现有代码实现看，系统已经具备以下基本闭环：

- 用户登录与身份识别
- 求解器、模板、文件规则管理
- 任务创建、上传、校验、提交
- 调度器轮询待执行任务并选择节点
- 节点代理接收任务并执行命令
- 节点向中心服务回传状态、日志和结果

从工程成熟度看，项目当前更适合定义为：

- 一个可以联调、可以演示、可以支撑毕业设计答辩的后端原型平台
- 而不是已经达到生产级别的完整分布式微服务平台

因此，下一阶段的核心目标不应再是盲目扩功能边界，而应优先保证“真实链路稳定可演示、文档与实现一致、关键设计点可解释”。

## 3. 当前完成度概览

## 4. 需要优先关注的事实

### 4.1 当前最关键的问题不是“缺模块”，而是“链路还需跑稳”

近期已经针对任务执行链路做过实质性修正，包括：

- 调度下发的任务对象已补充执行元数据
- 节点执行上下文已补充 `taskDir`
- ZIP 输入包解压后已支持将实际运行目录切换到解压根目录
- 命令模板已支持 `${taskDir}` 变量

这说明当前最应该做的是再次验证真实求解器任务，特别是 OpenFOAM 压缩包任务，而不是立刻再扩展新模块。

### 4.2 节点能力注册当前是“配置式上报 + 平台落表 + 管理员控制开关”

当前实现中：

- 节点注册时会上报 `nodeCode`、`nodeName`、`host`、`maxConcurrency`
- 节点注册时会按本地配置上报 `solverIds` 和 `solverVersions`
- 心跳时会上报 `cpuUsage`、`memoryUsage`、`runningCount`
- 平台端会将节点上报能力落到 `node_solver_capability`
- 管理员可以控制节点整体开关和节点-求解器能力开关

因此，这部分的真实口径是：

- 不是管理员手工凭空绑定求解器
- 也不是节点自动扫描系统后完整自发现全部求解器
- 而是“节点按配置声明能力，平台生成关联，管理员控制是否参与调度”

后续文档与论文材料应按照这一真实实现口径描述，除非后续继续补做自动探测能力。

### 4.3 部署骨架已经存在，但演示环境还需要标准化

仓库中已经存在：

- `compose.yaml`
- `Dockerfile.service`
- `node-agent/Dockerfile`

因此当前不是没有容器化基础，而是还缺少一套稳定、可复现、适合演示的部署和验证方案。

## 5. 推荐推进顺序

建议按 `P0 -> P1 -> P2 -> P3` 的顺序推进。

其中：

- `P0` 是必须优先完成的工作，直接影响系统能否稳定联调和答辩演示
- `P1` 是显著提升系统完整度和说服力的工作
- `P2` 是工程化增强项，建议在主链路稳定后推进
- `P3` 是偏长期演进方向，可作为论文展望内容

## 10. 推荐的近期执行节奏

建议按以下节奏推进：

### 第一阶段

- 跑通真实端到端烟雾流程
- 修复 OpenFOAM 等真实样例中的链路问题
- 固化演示样例、演示环境和排障方法

### 第二阶段

- 收口设计文档、数据库脚本和模块说明
- 完善节点能力注册、任务状态流转、调度失败处理的口径
- 输出论文可直接引用的后端描述材料

### 第三阶段

- 继续补调度增强
- 完善 Docker Compose 演示方案
- 视时间决定是否补自动探测或容器执行模式

## 11. 当前最不建议优先做的事情

在主链路未完全稳定前，不建议把主要精力放在以下方向：

- 一次性引入消息队列、服务注册中心、配置中心等大改造
- 将所有模块全部重构为生产级治理体系
- 过早做复杂的多实例调度高可用方案
- 为了追求“看起来先进”而继续扩大功能边界

这些内容可以写入论文展望，但不应压过当前“稳定演示闭环”的首要目标。

## 12. 总结

当前项目的最佳推进策略不是继续横向扩展，而是围绕已有闭环做纵向收口。

下一阶段最合理的路径是：

1. 先把真实任务链路跑稳，特别是压缩包上传、调度下发、节点执行这条主线
2. 再把设计文档、数据库脚本和代码实现收口统一
3. 然后补调度增强和部署标准化
4. 最后把安全、服务治理和多实例调度作为后续增强或论文展望

按照这一顺序推进，最有利于项目按期完成，也最有利于形成“代码可运行、文档可解释、论文可落地、答辩可展示”的最终成果。
