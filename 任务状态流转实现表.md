# 任务状态流转实现表

## 1. 文档说明

本文档用于把首版任务生命周期中的状态流转、触发接口、触发方、前置条件、主要副作用与幂等要求统一固化，作为后端状态机实现、前后端联调和测试用例设计的直接依据。

本文档以修订版《任务全流程设计与改造方案》第 7、8 章为基线，同时与以下文档保持一致：

- `需求分析与系统设计.md`
- `接口设计.md`
- `数据库设计.md`
- `后端设计.md`
- `前端设计.md`

## 2. 首版范围

### 2.1 纳入首版

- 创建任务
- 上传 ZIP 输入包
- 任务校验
- 提交入队
- 调度选点
- 节点派发
- 节点执行
- 日志回传
- 结果回传
- 派发失败回退
- 排队取消
- 管理员手动重试

### 2.2 不纳入首版

- `RUNNING -> CANCELED`
- node-agent 运行中取消
- 自动重试
- 复杂调度策略切换

## 3. 状态集合

首版统一任务状态如下：

- `CREATED`
- `VALIDATED`
- `QUEUED`
- `SCHEDULED`
- `DISPATCHED`
- `RUNNING`
- `SUCCESS`
- `FAILED`
- `CANCELED`
- `TIMEOUT`

## 4. 任务状态流转实现表

| 序号 | fromStatus | toStatus | 触发接口/动作 | 触发方 | 首版实现 | 前置条件 | 主要副作用 | 幂等/冲突要求 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | 无 | `CREATED` | `POST /api/tasks` | 用户 | 是 | 求解器、模板、任务类型合法 | 创建 `sim_task`，生成 `taskId/taskNo`，写状态历史 | 前端应避免重复创建；后端正常新增 |
| 2 | `CREATED` | `CREATED` | `POST /api/tasks/{taskId}/files` | 用户 | 是 | 任务存在且归属当前用户 | 覆盖 ZIP，更新 `task_file`，保持或重置为 `CREATED` | 重复上传允许，覆盖旧文件 |
| 3 | `VALIDATED` | `CREATED` | `PUT /api/tasks/{taskId}` 或 `POST /api/tasks/{taskId}/files` | 用户/管理员 | 是 | 仅未提交任务允许修改 | 更新任务名称、优先级、参数或输入包，清除已校验语义 | 合法重复调用可覆盖 |
| 4 | `CREATED` | `VALIDATED` | `POST /api/tasks/{taskId}/validate` | 用户 | 是 | ZIP 已上传，模板规则可校验 | 生成或更新 `workdir/unpackDir`，派生参数，写状态历史 | 同一任务允许重复发起校验；若仅为同一轮重复请求，可按幂等成功处理，不新增额外流转 |
| 5 | `VALIDATED` | `QUEUED` | `POST /api/tasks/{taskId}/submit` | 用户 | 是 | 当前必须为 `VALIDATED` | 刷新 `submitTime`，写状态历史 | 非 `VALIDATED` 调用必须拒绝 |
| 6 | `QUEUED` | `SCHEDULED` | `POST /internal/tasks/{taskId}/mark-scheduled` | scheduler-service | 是 | 已选出目标节点并预占容量；任务仍为 `QUEUED` | 绑定 `nodeId`，写状态历史 | 条件更新；失败返回冲突或 `claimed=false` |
| 7 | `SCHEDULED` | `DISPATCHED` | `POST /internal/tasks/{taskId}/mark-dispatched` | scheduler-service | 是 | `nodeId` 匹配，任务仍为 `SCHEDULED` | 写状态历史 | 同一 `taskId + nodeId` 重试幂等；若迟到确认时任务已进入 `RUNNING` 或终态，只返回当前状态，不回退、不补写状态历史 |
| 8 | `SCHEDULED` | `QUEUED` | `POST /internal/tasks/{taskId}/dispatch-failed`，`recoverable=true` 或 `POST /internal/tasks/node-offline/fail` | scheduler-service | 是 | 派发失败且未真正运行，或节点离线补偿触发回队 | 清理 `nodeId/fail/start/end` 等运行态字段，回队，写状态历史 | task-service 只回写状态；预占由 scheduler-service 在状态回写成功后释放 |
| 9 | `SCHEDULED` | `FAILED` | `POST /internal/tasks/{taskId}/dispatch-failed`，`recoverable=false` | scheduler-service | 是 | 派发失败且不可恢复 | 写 `failType/failMessage`，写状态历史 | task-service 只回写状态；预占由 scheduler-service 在状态回写成功后释放 |
| 10 | `DISPATCHED` | `RUNNING` | `POST /internal/tasks/{taskId}/status-report` | node-agent | 是 | 当前必须仍为 `DISPATCHED`，`nodeId` 匹配 | 写状态历史；随后调用 `release-reservation` | 重复 `RUNNING` 回传不得重复释放预占 |
| 11 | `DISPATCHED` | `QUEUED` | `POST /internal/tasks/{taskId}/dispatch-failed`，`recoverable=true` 或 `POST /internal/tasks/node-offline/fail` | scheduler-service | 是 | 节点已接收但未真正运行，或节点离线补偿触发回队 | 清理本轮节点绑定与运行态字段，回队 | task-service 只回写状态；预占由 scheduler-service 或离线补偿路径释放 |
| 12 | `DISPATCHED` | `FAILED` | `POST /internal/tasks/{taskId}/dispatch-failed`，`recoverable=false` | scheduler-service | 是 | 节点拒收或执行前即不可恢复失败 | 写 `failType/failMessage`，写状态历史 | task-service 只回写状态；预占由 scheduler-service 在状态回写成功后释放 |
| 13 | `RUNNING` | `SUCCESS` | `POST /internal/tasks/{taskId}/mark-finished`，`finalStatus=SUCCESS` | node-agent | 是 | 当前必须为 `RUNNING`，`nodeId` 匹配 | 写结束时间、写状态历史；随后尽力释放残留预占 | 终态重复回传应忽略；残留预占释放失败不得回滚终态 |
| 14 | `RUNNING` | `TIMEOUT` | `POST /internal/tasks/{taskId}/mark-finished`，`finalStatus=TIMEOUT` | node-agent/系统 | 是 | 当前必须为 `RUNNING` | 写结束时间、写状态历史；随后尽力释放残留预占 | 终态重复回传应忽略；残留预占释放失败不得回滚终态 |
| 15 | `RUNNING` | `FAILED` | `POST /internal/tasks/{taskId}/mark-failed` | node-agent | 是 | 当前必须为 `RUNNING`，`nodeId` 匹配 | 写 `failType/failMessage/endTime`，写状态历史；随后尽力释放残留预占 | 终态重复回传应忽略；残留预占释放失败不得回滚终态 |
| 16 | `RUNNING` | `FAILED` | `POST /internal/tasks/node-offline/fail` | scheduler-service | 是 | 节点超时离线且任务未完成 | 写 `failType=NODE_OFFLINE`，写状态历史；随后尽力释放残留预占 | 只补偿一次；残留预占释放失败不得回滚终态 |
| 17 | `QUEUED` | `CANCELED` | `POST /api/tasks/{taskId}/cancel` | 用户 | 是 | 当前必须为 `QUEUED` | 写状态历史 | 同一任务已是 `CANCELED` 可幂等返回；其他状态不得静默取消 |
| 18 | `FAILED` | `QUEUED` | `POST /api/admin/tasks/{taskId}/retry` | 管理员 | 是 | 当前必须为 `FAILED` | 清理 `nodeId/fail/start/end`，刷新 `submitTime`，写状态历史 | 重复重试应受当前状态控制 |
| 19 | `TIMEOUT` | `QUEUED` | `POST /api/admin/tasks/{taskId}/retry` | 管理员 | 是 | 当前必须为 `TIMEOUT` | 清理 `nodeId/fail/start/end`，刷新 `submitTime`，写状态历史 | 重复重试应受当前状态控制 |

## 5. 非状态流转动作

以下接口重要，但不直接改变主状态机终态：

| 动作 | 接口 | 说明 |
| --- | --- | --- |
| 放弃未提交任务 | `POST /api/tasks/{taskId}/discard` | 仅 `CREATED/VALIDATED` 允许；逻辑删除并清理目录；不转 `CANCELED` |
| 上传日志块 | `POST /internal/tasks/{taskId}/log-report` | 不改变任务状态；按 `taskId + seqNo` 幂等 |
| 上传结果摘要 | `POST /internal/tasks/{taskId}/result-summary-report` | 不改变任务状态；按 `taskId` 幂等覆盖或忽略 |
| 上传结果文件 | `POST /internal/tasks/{taskId}/result-file-report` | 不改变任务状态；按 `taskId + fileType + fileName` 幂等 |
| 节点预占容量 | `POST /internal/nodes/{nodeId}/reserve` | 调度前执行，不是任务状态流转；由 scheduler-service 写入或更新 `node_reservation=RESERVED`，成功后再增加 `reserved_count` |
| 释放节点预占 | `POST /internal/nodes/{nodeId}/release-reservation` | `DISPATCHED -> RUNNING` 成功后或派发失败时释放；由 scheduler-service 仅在 `node_reservation` 从 `RESERVED` 改为 `RELEASED` 时减少 `reserved_count` |

## 6. 首版禁止流转

以下流转在首版必须禁止：

| 禁止流转 | 原因 |
| --- | --- |
| `CREATED -> QUEUED` | 必须先校验通过 |
| `CREATED/VALIDATED -> CANCELED` | 应走 `discard`，不是正式取消 |
| `SCHEDULED/DISPATCHED -> SUCCESS/TIMEOUT` | 终态不能跨过 `RUNNING` |
| `RUNNING -> CANCELED` | 不属于首版 |
| 终态后再次流转到其他终态 | 终态后不得继续修改 |
| 任意状态自动补跳多级状态 | 禁止隐式补状态 |

## 7. 每次流转必须统一执行的动作

后端实现时建议统一封装以下动作，不要散落在不同接口处理器中：

1. 校验当前状态是否合法
2. 校验 `nodeId` 是否与任务当前绑定节点一致
3. 执行条件更新，确保状态变更原子性
4. 写入 `task_status_history`
5. 必要时更新 `nodeId`、失败信息、开始/结束时间等运行态字段
6. 必要时释放 `reserved_count`
7. 返回明确冲突信息，不做静默成功

## 8. 首版实现安全规则

### 8.1 节点容量

- `running_count` 只由节点心跳维护
- `reserved_count` 只表达调度预占
- 调度容量统一判断为：`running_count + reserved_count < max_concurrency`
- 调度器不直接修改 `running_count`

### 8.2 预占释放

- 只有成功完成 `DISPATCHED -> RUNNING` 后，才释放一次对应预占
- 派发失败统一通过 `dispatch-failed` 链路处理，不得和运行态释放重复
- `reserve` 与 `release-reservation` 都必须以 `node_reservation` 为幂等真值，`reserved_count` 只作为汇总值
- 只有当 `node_reservation(taskId, nodeId)` 从 `RESERVED` 成功更新为 `RELEASED` 时，才允许 `reserved_count - 1`
- 若 `node_reservation(taskId, nodeId)` 已为 `RELEASED`，重复释放必须直接按幂等成功返回，不得再次减少 `reserved_count`
- `node_reservation` 与 `reserved_count` 必须由 scheduler-service 在同一本地事务中维护

### 8.3 重复回传

- 状态回传按“`taskId + 当前绑定 nodeId + 合法状态阶段`”处理
- 若任务已进入目标状态或终态，重复回传直接忽略
- 日志块按 `taskId + seqNo` 幂等
- 结果文件按 `taskId + fileType + fileName` 幂等
- 结果文件落库以唯一约束 `uk_task_file_type_name(task_id, file_type, file_name)` 作为最终幂等兜底

### 8.4 节点离线补偿

- 超过 30 秒未收到心跳则标记 `OFFLINE`
- 该节点不再参与调度
- scheduler-service 通过 `POST /internal/tasks/node-offline/fail` 触发受影响任务的批量补偿回写
- `SCHEDULED`、`DISPATCHED` 任务首版统一回退到 `QUEUED`
- 若最终回退到 `QUEUED`，`NODE_OFFLINE` 只进入状态历史或调度记录，不写回任务主记录的 `failType`
- 对这类回队任务，task-service 在状态更新成功后必须逐条调用 `release-reservation` 释放对应节点预占；释放失败不得回滚已经完成的状态回队，应记录告警并等待后续补偿或审计
- `RUNNING` 任务首版统一收口为 `FAILED`，并记录 `failType = NODE_OFFLINE`

## 9. 开发使用建议

- 后端实现时，应再基于本文档生成“状态流转校验规则表”，把每个接口允许的 `fromStatus` 明确到代码级
- 测试阶段应覆盖正常流程、派发失败、重复回传、节点离线、管理员重试、排队取消等关键场景
- 前端按钮显隐建议统一依赖后端返回的 `canCancel`、`canRetry`，不要在多个页面重复硬编码状态判断
